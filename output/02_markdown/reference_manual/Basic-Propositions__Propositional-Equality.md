#  19.4. Propositional Equality[ūüĒó](find/?domain=Verso.Genre.Manual.section&name=propositional-equality "Permalink")
_Propositional equality_ is the operator that allows the equality of two terms to be stated as a proposition. [Definitional equality](The-Type-System/#--tech-term-definitional-equality) is checked automatically where necessary. As a result, its expressive power is limited in order to keep the algorithm that checks it fast and understandable. Propositional equality, on the other hand, must be explicitly proved and explicitly used‚ÄĒLean checks the validity of the proofs, rather than determining whether the statement is true. In exchange, it is much more expressive: many terms are propositionally equal without being definitionally equal.
Propositional equality is defined as an inductive type. Its sole constructor `[Eq.refl](Basic-Propositions/Propositional-Equality/#Eq___refl "Documentation for Eq.refl")` requires that both of the equated values be the same; this is implicitly an appeal to [definitional equality](The-Type-System/#--tech-term-definitional-equality). Propositional equality can also be thought of as the least reflexive relation modulo definitional equality. In addition to `[Eq.refl](Basic-Propositions/Propositional-Equality/#Eq___refl "Documentation for Eq.refl")`, equality proofs are generated by the `[propext](The-Type-System/Propositions/#propext "Documentation for propext")` and `[Quot.sound](The-Type-System/Quotients/#Quot___sound "Documentation for Quot.sound")` axioms.

##  19.4.2. Heterogeneous Equality[ūüĒó](find/?domain=Verso.Genre.Manual.section&name=HEq "Permalink")
_Heterogeneous equality_ is a version of [propositional equality](Basic-Propositions/Propositional-Equality/#--tech-term-Propositional-equality) that does not require that the two equated terms have the same type. However, _proving_ that the terms are equal using its version of `[rfl](Basic-Propositions/Propositional-Equality/#rfl-next "Documentation for rfl")` requires that both the types and the terms are definitionally equal. In other words, it allows more statements to be formulated.
Heterogeneous equality is typically less convenient in practice than ordinary propositional equality. The greater flexibility afforded by not requiring both sides of the equality to have the same type means that it has fewer useful properties. It is often encountered as a result of dependent pattern matching: the `[split](Tactic-Proofs/Tactic-Reference/#split "Documentation for tactic")` tactic and functional induction add heterogeneous equality assumptions to the context when the ordinary equality assumptions that are needed to accurate reflect the corresponding control flow would not be type correct. In these cases, the built-in automation has no choice but to use heterogeneous equality.

Heterogeneous equality. `a ‚Čć b` asserts that `a` and `b` have the same type, and casting `a` across the equality yields `b`, and vice versa.
You should avoid using this type if you can. Heterogeneous equality does not have all the same properties as `[Eq](Basic-Propositions/Propositional-Equality/#Eq___refl "Documentation for Eq")`, because the assumption that the types of `a` and `b` are equal is often too weak to prove theorems of interest. One public important non-theorem is the analogue of `[congr](Basic-Propositions/Propositional-Equality/#congr-next "Documentation for congr")`: If `f ‚Čć g` and `x ‚Čć y` and `f x` and `g y` are well typed it does not follow that `f x ‚Čć g y`. (This does follow if you have `f = g` instead.) However if `a` and `b` have the same type then `a = b` and `a ‚Čć b` are equivalent.
