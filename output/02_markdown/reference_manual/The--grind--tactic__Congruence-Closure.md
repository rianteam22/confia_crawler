#  16.3. Congruence Closure[ūüĒó](find/?domain=Verso.Genre.Manual.section&name=congruence-closure "Permalink")
_Congruence closure_ maintains equivalence classes of terms under the reflexive, symmetric, and transitive closure of ‚Äúis equal to‚ÄĚ _and_ the rule that equal arguments yield equal function results. Formally, if `a = a'` and `b = b'`, then `f a b = f a' b'` is added. The algorithm merges equivalence classes until a fixed point is reached. If a contradiction is discovered, then the goal can be closed immediately.
Using the analogy of the shared whiteboard:
  1. Every hypothesis `h : t‚āĀ = t‚āā` writes a line connecting `t‚āĀ` and `t‚āā`.
  2. Whenever two terms are connected by one or more lines, they're considered to be equal. Soon, whole constellations (`f a`, `g (f a)`, ‚Ä¶) are connected.
  3. If two different constructors of the same inductive type are connected by one or more lines, then a contradiction is discovered and the goal is closed. For example, equating `[True](Basic-Propositions/Truth/#True___intro "Documentation for True")` and `[False](Basic-Propositions/Truth/#False "Documentation for False")` or `[none](Basic-Types/Optional-Values/#Option___none "Documentation for Option.none")` and `[some](Basic-Types/Optional-Values/#Option___none "Documentation for Option.some") 1` would be a contradiction.

Initially, `f y`, `g y`, `x`, and `y` are in separate equivalence classes. The congruence closure engine uses `h‚āĀ` to merge `x` and `y`, after which the equivalence classes are `{x, y}`, `f y`, and `g y`. Next, `h‚āā` is used to merge `f y` and `g y`, after which the classes are `{x, y}` and `{f y, g y}`. This is sufficient to prove that `f x = g x`, because `y` and `x` are in the same class.

##  16.3.1. Congruence Closure vs. Simplification[ūüĒó](find/?domain=Verso.Genre.Manual.section&name=The-Lean-Language-Reference--The--grind--tactic--Congruence-Closure--Congruence-Closure-vs___-Simplification "Permalink")
Congruence closure is a fundamentally different operation from simplification:
  * `[simp](Tactic-Proofs/Tactic-Reference/#simp "Documentation for tactic")` _rewrites_ a goal, replacing occurrences of `t‚āĀ` with `t‚āā` as soon as it sees `h : t‚āĀ = t‚āā`. The rewrite is directional and destructive.
  * `[grind](Tactic-Proofs/Tactic-Reference/#grind "Documentation for tactic")` _accumulates_ equalities bidirectionally. No term is rewritten; instead, both representatives live in the same class. All other engines ([E‚ÄĎmatching](The--grind--tactic/E___matching/#--tech-term-E-matching), theory solvers, [propagation](The--grind--tactic/Constraint-Propagation/#--tech-term-Constraint-propagation)) can query these classes and add new facts, then the closure updates incrementally.


This makes congruence closure especially robust in the presence of symmetrical reasoning, mutual recursion, and large nestings of constructors where rewriting would duplicate work.
