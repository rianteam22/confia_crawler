#  17.3. Verification Conditions[🔗](find/?domain=Verso.Genre.Manual.section&name=The-Lean-Language-Reference--The--mvcgen--tactic--Verification-Conditions "Permalink")
The `[mvcgen](Tactic-Proofs/Tactic-Reference/#mvcgen "Documentation for tactic")` tactic converts a goal that's expressed in terms of `[SPred](The--mvcgen--tactic/Predicate-Transformers/#Std___Do___SPred "Documentation for Std.Do.SPred")` and weakest preconditions to a set of invariants and verification conditions that, together, suffice to prove the original goal. In particular, [Hoare triples](The--mvcgen--tactic/Predicate-Transformers/#--tech-term-Hoare-triple) are defined in terms of weakest preconditions, so `[mvcgen](Tactic-Proofs/Tactic-Reference/#mvcgen "Documentation for tactic")` can be used to prove them.
The verification conditions for a goal are generated as follows:
  1. A number of simplifications and rewrites are applied.
  2. The goal should now be of the form `P ⊢ₛ [wp⟦](The--mvcgen--tactic/Predicate-Transformers/#Std___Do____FLQQ_termWp____________FLQQ_ "Documentation for syntax")e⟧ Q` (that is, an entailment from some set of stateful assumptions to the weakest precondition that implies a desired postcondition).
  3. [Reducible](Definitions/Recursive-Definitions/#--tech-term-Reducible) constants and definitions marked `[@[](Attributes/#Lean___Parser___Term___attributes-next "Documentation for syntax")spec[]](Attributes/#Lean___Parser___Term___attributes-next "Documentation for syntax")` in the expression `e` are unfolded.
  4. If the expression is an application of an [auxiliary matching function](Elaboration-and-Compilation/#--tech-term-auxiliary-matching-functions) or a conditional (`ite` or `dite`), then it is first simplified. The [discriminant](Terms/Pattern-Matching/#--tech-term-match-discriminants) of each matcher is simplified, and the entire term is reduced in an attempt to eliminate the matcher or conditional. If this fails, then a new goal is generated for each branch.
  5. If the expression is an application of a constant, then the applicable lemmas marked `[@[](Attributes/#Lean___Parser___Term___attributes-next "Documentation for syntax")spec[]](Attributes/#Lean___Parser___Term___attributes-next "Documentation for syntax")` are attempted in priority order. Lean includes specification lemmas for constants such as `[bind](Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind")`, `[pure](Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure")`, and `[ForIn.forIn](Functors___-Monads-and--do--Notation/Syntax/#ForIn___mk "Documentation for ForIn.forIn")` that result from desugaring ``Lean.Parser.Term.do : term``[`do`](Functors___-Monads-and--do--Notation/Syntax/#Lean___Parser___Term___do)-notation. Instantiating the lemma will sometimes discharge its premises, in particular schematic variables due to definitional equalities with the goal. Assumptions of type `[Invariant](The--mvcgen--tactic/Predicate-Transformers/#Std___Do___Invariant "Documentation for Std.Do.Invariant")` are never instantiated this way, however. If the spec lemma's precondition or postcondition do not exactly match those of the goal, then new metavariables are created that prove the necessary entailments. If these cannot be immediately discharged using simple automation that attempts to use local assumptions and decomposes conjunctions in postconditions, then they remain as verification conditions.
  6. Each remaining goal created by this process is recursively processed for verification conditions if it has the form `P ⊢ₛ [wp⟦](The--mvcgen--tactic/Predicate-Transformers/#Std___Do____FLQQ_termWp____________FLQQ_ "Documentation for syntax")e⟧ Q`. If not, it is added to the set of invariants or verification conditions.
  7. The resulting subgoals for invariants and verification conditions are assigned suitable names in the proof state.
  8. Depending on the tactic's configuration parameters, `mvcgen_trivial` and `[mleave](Tactic-Proofs/Tactic-Reference/#mleave "Documentation for tactic")` are attempted in each verification condition.


Verification condition generation can be improved by defining appropriate [specification lemmas](The--mvcgen--tactic/Predicate-Transformers/#--tech-term-Specification-lemmas) for a library. The presence of good specification lemmas results in fewer generated verification conditions. Additionally, ensuring that the [simp normal form](The-Simplifier/Simp-Normal-Forms/#--tech-term-simp-normal-form) of terms is suitable for pattern matching, and that there are sufficient lemmas in the default simp set to reduce every possible term to that normal form, can lead to more conditionals and pattern matches being eliminated.
