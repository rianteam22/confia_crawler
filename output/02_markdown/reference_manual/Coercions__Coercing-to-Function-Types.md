#  11.4. Coercing to Function Types[🔗](find/?domain=Verso.Genre.Manual.section&name=fun-coercion "Permalink")
Another situation where an expected type is not generally available is the function position in a function application term. Dependent function types are common; together with [implicit](Terms/Functions/#--tech-term-implicit) parameters, they cause information to flow from the elaboration of one argument to the elaboration of the others. Attempting to deduce the type required for the function from the expected type of the entire application term and individually-inferred types of arguments will often fail. In these situations, Lean uses the `[CoeFun](Coercions/Coercing-to-Function-Types/#CoeFun___mk "Documentation for CoeFun")` type class to coerce a non-function in an application position into a function. Like `[CoeSort](Coercions/Coercing-to-Sorts/#CoeSort___mk "Documentation for CoeSort")`, `[CoeFun](Coercions/Coercing-to-Function-Types/#CoeFun___mk "Documentation for CoeFun")` instances do not chain with other coercions while inserting a function coercion, but they can be used as `[CoeOut](Coercions/Coercing-Between-Types/#CoeOut___mk "Documentation for CoeOut")` instances during ordinary coercion insertion.
The second parameter to `[CoeFun](Coercions/Coercing-to-Function-Types/#CoeFun___mk "Documentation for CoeFun")` is an output parameter that determines the resulting function type. This output parameter is function that computes the function type from the term that's being coerced, rather than the function type itself. Unlike `[CoeDep](Coercions/Coercing-Between-Types/#CoeDep___mk "Documentation for CoeDep")`, the term itself is not taken into account during instance synthesis; it can, however, be used to create dependently typed coercions where the function type is determined by the term.

`[CoeFun](Coercions/Coercing-to-Function-Types/#CoeFun___mk "Documentation for CoeFun") α (γ : α → Sort v)` is a coercion to a function. `γ a` should be a (coercion-to-)function type, and this is triggered whenever an element `f : α` appears in an application like `f x`, which would not make sense since `f` does not have a function type. `[CoeFun](Coercions/Coercing-to-Function-Types/#CoeFun___mk "Documentation for CoeFun")` instances apply to `[CoeOut](Coercions/Coercing-Between-Types/#CoeOut___mk "Documentation for CoeOut")` as well.

Coercing to Function Types
A well-typed interpreter is an interpreter for a programming language that uses indexed families to rule out run-time type errors. Functions written in the interpreted language can be interpreted as Lean functions, but their underlying source code can also be inspected.
The first step in the well-typed interpreter is to select the subset of Lean types that can be used. These types are represented by an [inductive type](The-Type-System/Inductive-Types/#--tech-term-Inductive-types) of codes `[Ty](Coercions/Coercing-to-Function-Types/#Ty-_LPAR_in-Coercing-to-Function-Types_RPAR_ "Definition of example")` and a function that maps these codes to actual types.
