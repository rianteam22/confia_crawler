##  5.4.1. Recovery from Failure[🔗](find/?domain=Verso.Genre.Manual.section&name=alternative-recovery "Permalink")
`Validate` can also be used in situations where there is more than one way for input to be acceptable. For the input form `RawInput`, an alternative set of business rules that implement conventions from a legacy system might be the following:
  1. All human users must provide a birth year that is four digits.
  2. Users born prior to 1970 do not need to provide names, due to incomplete older records.
  3. Users born after 1970 must provide names.
  4. Companies should enter `"FIRM"` as their year of birth and provide a company name.


No particular provision is made for users born in 1970. It is expected that they will either give up, lie about their year of birth, or call. The company considers this an acceptable cost of doing business.
The following inductive type captures the values that can be produced from these stated rules:
`abbrev NonEmptyString := {s : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") // s ≠ ""}  inductive LegacyCheckedInput where   | humanBefore1970 :     (birthYear : {y : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") // y > 999 ∧ y < 1970}) →     [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") →     LegacyCheckedInput   | humanAfter1970 :     (birthYear : {y : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") // y > 1970}) →     [NonEmptyString](Functors___-Applicative-Functors___-and-Monads/Alternatives/#NonEmptyString "Definition of example") →     LegacyCheckedInput   | company :     [NonEmptyString](Functors___-Applicative-Functors___-and-Monads/Alternatives/#NonEmptyString "Definition of example") →     LegacyCheckedInput deriving [Repr](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Repr___mk "Documentation for Repr in Lean Language Reference")`
A validator for these rules is more complicated, however, as it must address all three cases. While it can be written as a series of nested `if` expressions, it's easier to design the three cases independently and then combine them. This requires a means of recovering from failure while preserving error messages:
`def Validate.orElse     (a : Validate ε α)     (b : [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") → Validate ε α) :     Validate ε α :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") a [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [.ok](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Validate___ok "Definition of example") x => [.ok](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Validate___ok "Definition of example") x   | .errors errs1 =>     [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") b () [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")     | [.ok](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Validate___ok "Definition of example") x => [.ok](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Validate___ok "Definition of example") x     | .errors errs2 => .errors (errs1 ++ errs2)`
This pattern of recovery from failures is common enough that Lean has built-in syntax for it, attached to a type class named `OrElse`:
`class OrElse (α : Type) where   orElse : α → ([Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") → α) → α`
The expression `E1 <|> E2` is short for `OrElse.orElse E1 (fun () => E2)`. An instance of `OrElse` for `Validate` allows this syntax to be used for error recovery:
`instance : OrElse (Validate ε α) where   orElse := [Validate.orElse](Functors___-Applicative-Functors___-and-Monads/Alternatives/#Validate___orElse "Definition of example")`
The validator for `LegacyCheckedInput` can be built from a validator for each constructor. The rules for a company state that the birth year should be the string `"FIRM"` and that the name should be non-empty. The constructor `[LegacyCheckedInput.company](Functors___-Applicative-Functors___-and-Monads/Alternatives/#LegacyCheckedInput___company "Definition of example")`, however, has no representation of the birth year at all, so there's no easy way to carry it out using `<*>`. The key is to use a function with `<*>` that ignores its argument.
Checking that a Boolean condition holds without recording any evidence of this fact in a type can be accomplished with `checkThat`:
`def checkThat (condition : [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference"))     (field : [Field](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Field "Definition of example")) (msg : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) :     Validate ([Field](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Field "Definition of example") × [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") :=   [if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") condition [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") () [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") reportError field msg`
This definition of `checkCompany` uses `checkThat`, and then throws away the resulting `[Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")` value:
`def checkCompany (input : RawInput) :     Validate ([Field](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Field "Definition of example") × [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) LegacyCheckedInput :=   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (fun () name => [.company](Functors___-Applicative-Functors___-and-Monads/Alternatives/#LegacyCheckedInput___company "Definition of example") name) <*>     checkThat (input.[birthYear](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#RawInput___birthYear "Definition of example") == "FIRM")       "birth year" "FIRM if a company" <*>     [checkName](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#checkName "Definition of example") input.[name](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#RawInput___name "Definition of example")`
However, this definition is quite noisy. It can be simplified in two ways. The first is to replace the first use of `<*>` with a specialized version that automatically ignores the value returned by the first argument, called `*>`. This operator is also controlled by a type class, called `SeqRight`, and `E1 *> E2` is syntactic sugar for `[SeqRight.seqRight](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#SeqRight___mk "Documentation for SeqRight.seqRight in Lean Language Reference") E1 (fun () => E2)`:
`class SeqRight (f : Type → Type) where   seqRight : f α → ([Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") → f β) → f β`
There is a default implementation of `seqRight` in terms of `seq`: `seqRight (a : f α) (b : Unit → f β) : f β := pure (fun _ x => x) <*> a <*> b ()`.
Using `seqRight`, `checkCompany` becomes simpler:
`def checkCompany (input : RawInput) :     Validate ([Field](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Field "Definition of example") × [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) LegacyCheckedInput :=   checkThat (input.[birthYear](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#RawInput___birthYear "Definition of example") == "FIRM")     "birth year" "FIRM if a company" *>   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") [.company](Functors___-Applicative-Functors___-and-Monads/Alternatives/#LegacyCheckedInput___company "Definition of example") <*> [checkName](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#checkName "Definition of example") input.[name](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#RawInput___name "Definition of example")`
One more simplification is possible. For every `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")`, `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") f <*> E` is equivalent to `f <$> E`. In other words, using `seq` to apply a function that was placed into the `Applicative` type using `pure` is overkill, and the function could have just been applied using `[Functor.map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")`. This simplification yields:
`def checkCompany (input : RawInput) :     Validate ([Field](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Field "Definition of example") × [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) LegacyCheckedInput :=   checkThat (input.[birthYear](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#RawInput___birthYear "Definition of example") == "FIRM")     "birth year" "FIRM if a company" *>   [.company](Functors___-Applicative-Functors___-and-Monads/Alternatives/#LegacyCheckedInput___company "Definition of example") <$> [checkName](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#checkName "Definition of example") input.[name](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#RawInput___name "Definition of example")`
The remaining two constructors of `LegacyCheckedInput` use subtypes for their fields. A general-purpose tool for checking subtypes will make these easier to read:
`def checkSubtype {α : Type} (v : α) (p : α → Prop) [[Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference") (p v)]     (err : ε) : Validate ε {x : α // p x} :=   [if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference") h : p v [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference")     [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") ⟨v, h⟩   [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference")     .errors { [head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") := err, [tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") := [] }`
In the function's argument list, it's important that the type class `[[Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference") (p v)]` occur after the specification of the arguments `v` and `p`. Otherwise, it would refer to an additional set of automatic implicit arguments, rather than to the manually-provided values. The `[Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference")` instance is what allows the proposition `p v` to be checked using `if`.

##  5.4.2. The `Alternative` Class[🔗](find/?domain=Verso.Genre.Manual.section&name=Alternative "Permalink")
Many types support a notion of failure and recovery. The `Many` monad from the section on [evaluating arithmetic expressions in a variety of monads](Monads/Example___-Arithmetic-in-Monads/#nondeterministic-search) is one such type, as is `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")`. Both support failure without providing a reason (unlike, say, `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")` and `Validate`, which require some indication of what went wrong).
The `Alternative` class describes applicative functors that have additional operators for failure and recovery:
`class Alternative (f : Type → Type) extends [Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference") f where   failure : f α   orElse : f α → ([Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") → f α) → f α`
Just as implementors of `[Add](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Add___mk "Documentation for Add in Lean Language Reference") α` get `[HAdd](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAdd___mk "Documentation for HAdd in Lean Language Reference") α α α` instances for free, implementors of `Alternative` get `OrElse` instances for free:
`instance [Alternative f] : OrElse (f α) where   orElse := [Alternative.orElse](Functors___-Applicative-Functors___-and-Monads/Alternatives/#FakeAlternative___Alternative___orElse "Definition of example")`
The implementation of `Alternative` for `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")` keeps the first non-`[none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")` argument:
`instance : [Alternative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Alternative___mk "Documentation for Alternative in Lean Language Reference") [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") where   failure := [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")   orElse     | [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x, _ => [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x     | [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference"), y => y ()`
Similarly, the implementation for `Many` follows the general structure of `Many.union`, with minor differences due to the laziness-inducing `[Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")` parameters being placed differently:
`def Many.orElse : Many α → ([Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") → Many α) → Many α   | [.none](Monads/Example___-Arithmetic-in-Monads/#Many___none "Definition of example"), ys => ys ()   | .more x xs, ys => .more x (fun () => [orElse](Functors___-Applicative-Functors___-and-Monads/Alternatives/#Many___orElse "Definition of example") (xs ()) ys)  instance : [Alternative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Alternative___mk "Documentation for Alternative in Lean Language Reference") Many where   failure := [.none](Monads/Example___-Arithmetic-in-Monads/#Many___none "Definition of example")   orElse := [Many.orElse](Functors___-Applicative-Functors___-and-Monads/Alternatives/#Many___orElse "Definition of example")`
Like other type classes, `Alternative` enables the definition of a variety of operations that work for _any_ applicative functor that implements `Alternative`. One of the most important is `guard`, which causes `failure` when a decidable proposition is false:
`def guard [[Alternative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Alternative___mk "Documentation for Alternative in Lean Language Reference") f] (p : Prop) [[Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference") p] : f [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") :=   [if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") p [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference")     [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") ()   [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") failure`
It is very useful in monadic programs to terminate execution early. In `Many`, it can be used to filter out a whole branch of a search, as in the following program that computes all even divisors of a natural number:

###  5.4.3.1. Improve Validation Friendliness
The errors returned from `Validate` programs that use `<|>` can be difficult to read, because inclusion in the list of errors simply means that the error can be reached through _some_ code path. A more structured error report can be used to guide the user through the process more accurately:
  * Replace the `NonEmptyList` in `Validate.errors` with a bare type variable, and then update the definitions of the `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference") (Validate ε)` and `OrElse (Validate ε α)` instances to require only that there be an `[Append](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Append___mk "Documentation for Append in Lean Language Reference") ε` instance available.
  * Define a function `Validate.mapErrors : Validate ε α → (ε → ε') → Validate ε' α` that transforms all the errors in a validation run.
  * Using the datatype `TreeError` to represent errors, rewrite the legacy validation system to track its path through the three alternatives.
  * Write a function `report : TreeError → [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")` that outputs a user-friendly view of the `TreeError`'s accumulated warnings and errors.
