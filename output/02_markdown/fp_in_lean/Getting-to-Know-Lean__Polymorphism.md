#  1.6. Polymorphism[🔗](find/?domain=Verso.Genre.Manual.section&name=polymorphism "Permalink")
Just as in most languages, types in Lean can take arguments. For instance, the type `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` describes lists of natural numbers, `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")` describes lists of strings, and `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") ([List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") Point)` describes lists of lists of points. This is very similar to `List<Nat>`, `List<String>`, or `List<List<Point>>` in a language like C# or Java. Just as Lean uses a space to pass an argument to a function, it uses a space to pass an argument to a type.
In functional programming, the term _polymorphism_ typically refers to datatypes and definitions that take types as arguments. This is different from the object-oriented programming community, where the term typically refers to subclasses that may override some behavior of their superclass. In this book, “polymorphism” always refers to the first sense of the word. These type arguments can be used in the datatype or definition, which allows the same datatype or definition to be used with any type that results from replacing the arguments' names with some other types.
The `Point` structure requires that both the `x` and `y` fields are `[Float](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Floating-Point-Numbers/#Float-next "Documentation for Float in Lean Language Reference")`s. There is, however, nothing about points that require a specific representation for each coordinate. A polymorphic version of `Point`, called `PPoint`, can take a type as an argument, and then use that type for both fields:
`structure PPoint (α : Type) where   x : α   y : α`
Just as a function definition's arguments are written immediately after the name being defined, a structure's arguments are written immediately after the structure's name. It is customary to use Greek letters to name type arguments in Lean when no more specific name suggests itself. `Type` is a type that describes other types, so `Nat`, `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")`, and `PPoint [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` all have type `Type`.
Just like `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference")`, `PPoint` can be used by providing a specific type as its argument:
`def natOrigin : PPoint [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") :=   { x := [Nat.zero](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.zero in Lean Language Reference"), y := [Nat.zero](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.zero in Lean Language Reference") }`
In this example, both fields are expected to be `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`s. Just as a function is called by replacing its argument variables with its argument values, providing `PPoint` with the type `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` as an argument yields a structure in which the fields `x` and `y` have the type `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`, because the argument name `α` has been replaced by the argument type `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`. Types are ordinary expressions in Lean, so passing arguments to polymorphic types (like `PPoint`) doesn't require any special syntax.
Definitions may also take types as arguments, which makes them polymorphic. The function `replaceX` replaces the `x` field of a `PPoint` with a new value. In order to allow `replaceX` to work with _any_ polymorphic point, it must be polymorphic itself. This is achieved by having its first argument be the type of the point's fields, with later arguments referring back to the first argument's name.
`def replaceX (α : Type) (point : PPoint α) (newX : α) : PPoint α :=   { point with x := newX }`
In other words, when the types of the arguments `point` and `newX` mention `α`, they are referring to _whichever type was provided as the first argument_. This is similar to the way that function argument names refer to the values that were provided when they occur in the function's body.
This can be seen by asking Lean to check the type of `replaceX`, and then asking it to check the type of `replaceX [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`.
`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") (replaceX)`
```
replaceX : (α : Type) → PPoint α → α → PPoint α
```

This function type includes the _name_ of the first argument, and later arguments in the type refer back to this name. Just as the value of a function application is found by replacing the argument name with the provided argument value in the function's body, the type of a function application is found by replacing the argument's name with the provided value in the function's return type. Providing the first argument, `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`, causes all occurrences of `α` in the remainder of the type to be replaced with `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`:
`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") replaceX [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`
```
replaceX [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") : PPoint [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → PPoint [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")
```

Because the remaining arguments are not explicitly named, no further substitution occurs as more arguments are provided:
`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") replaceX [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") [natOrigin](Getting-to-Know-Lean/Polymorphism/#natOrigin "Definition of example")`

The fact that the type of the whole function application expression was determined by passing a type as an argument has no bearing on the ability to evaluate it.
`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") replaceX [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") [natOrigin](Getting-to-Know-Lean/Polymorphism/#natOrigin "Definition of example") 5`
```
{ x := 5, y := 0 }
```

Polymorphic functions work by taking a named type argument and having later types refer to the argument's name. However, there's nothing special about type arguments that allows them to be named. Given a datatype that represents positive or negative signs:
`inductive Sign where   | pos   | neg`
it is possible to write a function whose argument is a sign. If the argument is positive, the function returns a `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`, while if it's negative, it returns an `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")`:
`def posOrNegThree (s : [Sign](Getting-to-Know-Lean/Polymorphism/#Sign "Definition of example")) :     [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") s [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") | Sign.pos => [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") | Sign.neg => [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") s [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | Sign.pos => (3 : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference"))   | Sign.neg => (-3 : [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference"))`

When Lean is checking this function, it uses the fact that the `match`-expression in the function's body corresponds to the `match`-expression in the type to make `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` be the expected type for the `pos` case and to make `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` be the expected type for the `neg` case.

##  1.6.1. Linked Lists[🔗](find/?domain=Verso.Genre.Manual.section&name=linked-lists "Permalink")
Lean's standard library includes a canonical linked list datatype, called `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference")`, and special syntax that makes it more convenient to use. Lists are written in square brackets. For instance, a list that contains the prime numbers less than 10 can be written:
`def primesUnder10 : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") := [2, 3, 5, 7]`
Behind the scenes, `List` is an inductive datatype, defined like this:
`inductive List (α : Type) where   | nil : List α   | cons : α → List α → List α`
The actual definition in the standard library is slightly different, because it uses features that have not yet been presented, but it is substantially similar. This definition says that `List` takes a single type as its argument, just as `PPoint` did. This type is the type of the entries stored in the list. According to the constructors, a `List α` can be built with either `nil` or `cons`. The constructor `nil` represents empty lists and the constructor `cons` is used for non-empty lists. The first argument to `cons` is the head of the list, and the second argument is its tail. A list that contains `nnn` entries contains `nnn` `cons` constructors, the last of which has `nil` as its tail.
The `primesUnder10` example can be written more explicitly by using `List`'s constructors directly:
`def explicitPrimesUnder10 : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") :=   [List.cons](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference") 2 ([List.cons](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference") 3 ([List.cons](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference") 5 ([List.cons](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference") 7 [List.nil](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.nil in Lean Language Reference"))))`
These two definitions are completely equivalent, but `primesUnder10` is much easier to read than `explicitPrimesUnder10`.
Functions that consume `List`s can be defined in much the same way as functions that consume `Nat`s. Indeed, one way to think of a linked list is as a `Nat` that has an extra data field dangling off each `succ` constructor. From this point of view, computing the length of a list is the process of replacing each `cons` with a `succ` and the final `nil` with a `zero`. Just as `replaceX` took the type of the fields of the point as an argument, `length` takes the type of the list's entries. For example, if the list contains strings, then the first argument is `[String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")`: `length [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") ["Sourdough", "bread"]`. It should compute like this:

The definition of `length` is both polymorphic (because it takes the list entry type as an argument) and recursive (because it refers to itself). Generally, functions follow the shape of the data: recursive datatypes lead to recursive functions, and polymorphic datatypes lead to polymorphic functions.
`def length (α : Type) (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") xs [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [List.nil](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.nil in Lean Language Reference") => [Nat.zero](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.zero in Lean Language Reference")   | [List.cons](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference") y ys => [Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") (length α ys)`
Names such as `xs` and `ys` are conventionally used to stand for lists of unknown values. The `s` in the name indicates that they are plural, so they are pronounced “exes” and “whys” rather than “x s” and “y s”.
To make it easier to read functions on lists, the bracket notation `[]` can be used to pattern-match against `nil`, and an infix `::` can be used in place of `cons`:
`def length (α : Type) (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") xs [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [] => 0   | y :: ys => [Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") ([length](Getting-to-Know-Lean/Polymorphism/#Oooops___length "Definition of example") α ys)`
##  1.6.2. Implicit Arguments[🔗](find/?domain=Verso.Genre.Manual.section&name=implicit-parameters "Permalink")
Both `replaceX` and `length` are somewhat bureaucratic to use, because the type argument is typically uniquely determined by the later values. Indeed, in most languages, the compiler is perfectly capable of determining type arguments on its own, and only occasionally needs help from users. This is also the case in Lean. Arguments can be declared _implicit_ by wrapping them in curly braces instead of parentheses when defining a function. For example, a version of `replaceX` with an implicit type argument looks like this:
`def replaceX {α : Type} (point : PPoint α) (newX : α) : PPoint α :=   { point with x := newX }`
It can be used with `[natOrigin](Getting-to-Know-Lean/Polymorphism/#natOrigin "Definition of example")` without providing `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` explicitly, because Lean can _infer_ the value of `α` from the later arguments:
``{ x := 5, y := 0 }`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") replaceX [natOrigin](Getting-to-Know-Lean/Polymorphism/#natOrigin "Definition of example") 5`
```
{ x := 5, y := 0 }
```

Similarly, `length` can be redefined to take the entry type implicitly:

##  1.6.3. More Built-In Datatypes[🔗](find/?domain=Verso.Genre.Manual.section&name=more-built-in-types "Permalink")
In addition to lists, Lean's standard library contains a number of other structures and inductive datatypes that can be used in a variety of contexts.
###  1.6.3.1. `Option`[🔗](find/?domain=Verso.Genre.Manual.section&name=Option "Permalink")
Not every list has a first entry—some lists are empty. Many operations on collections may fail to find what they are looking for. For instance, a function that finds the first entry in a list may not find any such entry. It must therefore have a way to signal that there was no first entry.
Many languages have a `null` value that represents the absence of a value. Instead of equipping existing types with a special `null` value, Lean provides a datatype called `Option` that equips some other type with an indicator for missing values. For instance, a nullable `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` is represented by `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")`, and a nullable list of strings is represented by the type `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ([List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference"))`. Introducing a new type to represent nullability means that the type system ensures that checks for `null` cannot be forgotten, because an `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` can't be used in a context where an `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` is expected.
`Option` has two constructors, called `some` and `none`, that respectively represent the non-null and null versions of the underlying type. The non-null constructor, `some`, contains the underlying value, while `none` takes no arguments:
`inductive Option (α : Type) : Type where   | none : Option α   | some (val : α) : Option α`
The `Option` type is very similar to nullable types in languages like C# and Kotlin, but it is not identical. In these languages, if a type (say, `Boolean`) always refers to actual values of the type (`true` and `false`), the type `Boolean?` or `Nullable<Boolean>` additionally admits the `null` value. Tracking this in the type system is very useful: the type checker and other tooling can help programmers remember to check for `null`, and APIs that explicitly describe nullability through type signatures are more informative than ones that don't. However, these nullable types differ from Lean's `Option` in one very important way, which is that they don't allow multiple layers of optionality. `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ([Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference"))` can be constructed with `[none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`, `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`, or `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") ([some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") 360)`. Kotlin, on the other hand, treats `T??` as being equivalent to `T?`. This subtle difference is rarely relevant in practice, but it can matter from time to time.
To find the first entry in a list, if it exists, use `List.head?`. The question mark is part of the name, and is not related to the use of question marks to indicate nullable types in C# or Kotlin. In the definition of `List.head?`, an underscore is used to represent the tail of the list. In patterns, underscores match anything at all, but do not introduce variables to refer to the matched data. Using underscores instead of names is a way to clearly communicate to readers that part of the input is ignored.
`def List.head? {α : Type} (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) : [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") α :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") xs [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [] => [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")   | y :: _ => [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") y`
A Lean naming convention is to define operations that might fail in groups using the suffixes `?` for a version that returns an `Option`, `!` for a version that crashes when provided with invalid input, and `D` for a version that returns a default value when the operation would otherwise fail. Following this pattern, `[List.head](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___head "Documentation for List.head in Lean Language Reference")` requires the caller to provide mathematical evidence that the list is not empty, `[List.head?](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___head___ "Documentation for List.head? in Lean Language Reference")` returns an `Option`, `[List.head!](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___head___-next "Documentation for List.head! in Lean Language Reference")` crashes the program when passed an empty list, and `[List.headD](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___headD "Documentation for List.headD in Lean Language Reference")` takes a default value to return in case the list is empty. The question mark and exclamation mark are part of the name, not special syntax, as Lean's naming rules are more liberal than many languages.
Because `[head?](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___head___ "Documentation for List.head? in Lean Language Reference")` is defined in the `List` namespace, it can be used with accessor notation:
``[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") 2`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") primesUnder10.[head?](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___head___ "Documentation for List.head? in Lean Language Reference")`

###  1.6.3.2. `Prod`[🔗](find/?domain=Verso.Genre.Manual.section&name=prod "Permalink")
The `Prod` structure, short for “Product”, is a generic way of joining two values together. For instance, a `[Prod](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod in Lean Language Reference") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")` contains a `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` and a `[String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")`. In other words, `PPoint [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` could be replaced by `[Prod](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod in Lean Language Reference") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`. `[Prod](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod in Lean Language Reference")` is very much like C#'s tuples, the `Pair` and `Triple` types in Kotlin, and `tuple` in C++. Many applications are best served by defining their own structures, even for simple cases like `Point`, because using domain terminology can make it easier to read the code. Additionally, defining structure types helps catch more errors by assigning different types to different domain concepts, preventing them from being mixed up.
On the other hand, there are some cases where it is not worth the overhead of defining a new type. Additionally, some libraries are sufficiently generic that there is no more specific concept than “pair”. Finally, the standard library contains a variety of convenience functions that make it easier to work with the built-in pair type.
The structure `Prod` is defined with two type arguments:
`structure Prod (α : Type) (β : Type) : Type where   fst : α   snd : β`
Lists are used so frequently that there is special syntax to make them more readable. For the same reason, both the product type and its constructor have special syntax. The type `[Prod](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod in Lean Language Reference") α β` is typically written `α × β`, mirroring the usual notation for a Cartesian product of sets. Similarly, the usual mathematical notation for pairs is available for `[Prod](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod in Lean Language Reference")`. In other words, instead of writing:
`def fives : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") × [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") := { [fst](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.fst in Lean Language Reference") := "five", [snd](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.snd in Lean Language Reference") := 5 }`
it suffices to write:
`def fives : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") × [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") := ("five", 5)`
Both notations are right-associative. This means that the following definitions are equivalent:
`def sevens : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") × [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") × [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") := ("VII", 7, 4 + 3)``def sevens : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") × ([Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") × [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) := ("VII", (7, 4 + 3))`
In other words, all products of more than two types, and their corresponding constructors, are actually nested products and nested pairs behind the scenes.
###  1.6.3.3. `Sum`[🔗](find/?domain=Verso.Genre.Manual.section&name=Sum "Permalink")
The `Sum` datatype is a generic way of allowing a choice between values of two different types. For instance, a `[Sum](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` is either a `[String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")` or an `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")`. Like `Prod`, `Sum` should be used either when writing very generic code, for a very small section of code where there is no sensible domain-specific type, or when the standard library contains useful functions. In most situations, it is more readable and maintainable to use a custom inductive type.
Values of type `Sum α β` are either the constructor `inl` applied to a value of type `α` or the constructor `inr` applied to a value of type `β`:
`inductive Sum (α : Type) (β : Type) : Type where   | inl : α → Sum α β   | inr : β → Sum α β`
These names are abbreviations for “left injection” and “right injection”, respectively. Just as the Cartesian product notation is used for `Prod`, a “circled plus” notation is used for `[Sum](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum in Lean Language Reference")`, so `α ⊕ β` is another way to write `[Sum](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum in Lean Language Reference") α β`. There is no special syntax for `Sum.inl` and `Sum.inr`.
As an example, if pet names can either be dog names or cat names, then a type for them can be introduced as a sum of strings:
`def PetName : Type := [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") ⊕ [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")`
In a real program, it would usually be better to define a custom inductive datatype for this purpose with informative constructor names. Here, `[Sum.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference")` is to be used for dog names, and `[Sum.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference")` is to be used for cat names. These constructors can be used to write a list of animal names:
`def animals : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") PetName :=   [[Sum.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") "Spot", [Sum.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference") "Tiger", [Sum.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") "Fifi",    [Sum.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") "Rex", [Sum.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference") "Floof"]`
Pattern matching can be used to distinguish between the two constructors. For instance, a function that counts the number of dogs in a list of animal names (that is, the number of `[Sum.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference")` constructors) looks like this:
`def howManyDogs (pets : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") PetName) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") pets [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [] => 0   | [Sum.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") _ :: morePets => [howManyDogs](Getting-to-Know-Lean/Polymorphism/#howManyDogs "Definition of example") morePets + 1   | [Sum.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference") _ :: morePets => [howManyDogs](Getting-to-Know-Lean/Polymorphism/#howManyDogs "Definition of example") morePets`
Function calls are evaluated before infix operators, so `[howManyDogs](Getting-to-Know-Lean/Polymorphism/#howManyDogs "Definition of example") morePets + 1` is the same as `([howManyDogs](Getting-to-Know-Lean/Polymorphism/#howManyDogs "Definition of example") morePets) + 1`. As expected, ``3`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") [howManyDogs](Getting-to-Know-Lean/Polymorphism/#howManyDogs "Definition of example") [animals](Getting-to-Know-Lean/Polymorphism/#animals "Definition of example")` yields `3`.
###  1.6.3.4. `Unit`[🔗](find/?domain=Verso.Genre.Manual.section&name=Unit "Permalink")
`Unit` is a type with just one argumentless constructor, called `unit`. In other words, it describes only a single value, which consists of said constructor applied to no arguments whatsoever. `Unit` is defined as follows:
`inductive Unit : Type where   | unit : Unit`
On its own, `Unit` is not particularly useful. However, in polymorphic code, it can be used as a placeholder for data that is missing. For instance, the following inductive datatype represents arithmetic expressions:
`inductive ArithExpr (ann : Type) : Type where   | int : ann → [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") → [ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") ann   | plus : ann → [ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") ann → [ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") ann → [ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") ann   | minus : ann → [ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") ann → [ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") ann → [ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") ann   | times : ann → [ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") ann → [ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") ann → [ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") ann`
The type argument `ann` stands for annotations, and each constructor is annotated. Expressions coming from a parser might be annotated with source locations, so a return type of `[ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") [SourcePos](Getting-to-Know-Lean/Polymorphism/#SourcePos "Definition of example")` ensures that the parser put a `SourcePos` at each subexpression. Expressions that don't come from the parser, however, will not have source locations, so their type can be `[ArithExpr](Getting-to-Know-Lean/Polymorphism/#ArithExpr "Definition of example") [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")`.
Additionally, because all Lean functions have arguments, zero-argument functions in other languages can be represented as functions that take a `[Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")` argument. In a return position, the `[Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")` type is similar to `void` in languages derived from C. In the C family, a function that returns `void` will return control to its caller, but it will not return any interesting value. By being an intentionally uninteresting value, `[Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")` allows this to be expressed without requiring a special-purpose `void` feature in the type system. Unit's constructor can be written as empty parentheses: `() : [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")`.
###  1.6.3.5. `Empty`[🔗](find/?domain=Verso.Genre.Manual.section&name=Empty "Permalink")
The `[Empty](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Empty-Type/#Empty "Documentation for Empty in Lean Language Reference")` datatype has no constructors whatsoever. Thus, it indicates unreachable code, because no series of calls can ever terminate with a value at type `[Empty](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Empty-Type/#Empty "Documentation for Empty in Lean Language Reference")`.
`[Empty](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Empty-Type/#Empty "Documentation for Empty in Lean Language Reference")` is not used nearly as often as `[Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")`. However, it is useful in some specialized contexts. Many polymorphic datatypes do not use all of their type arguments in all of their constructors. For instance, `[Sum.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference")` and `[Sum.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference")` each use only one of `[Sum](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum in Lean Language Reference")`'s type arguments. Using `[Empty](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Empty-Type/#Empty "Documentation for Empty in Lean Language Reference")` as one of the type arguments to `[Sum](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum in Lean Language Reference")` can rule out one of the constructors at a particular point in a program. This can allow generic code to be used in contexts that have additional restrictions.
###  1.6.3.6. Naming: Sums, Products, and Units[🔗](find/?domain=Verso.Genre.Manual.section&name=sum-products-units "Permalink")
Generally speaking, types that offer multiple constructors are called _sum types_ , while types whose single constructor takes multiple arguments are called _product types_. These terms are related to sums and products used in ordinary arithmetic. The relationship is easiest to see when the types involved contain a finite number of values. If `α` and `β` are types that contain `nnn` and `kkk` distinct values, respectively, then `α ⊕ β` contains `n+kn + kn+k` distinct values and `α × β` contains `n×kn \times kn×k` distinct values. For instance, `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")` has two values: `true` and `false`, and `Unit` has one value: `[Unit.unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit___unit "Documentation for Unit.unit in Lean Language Reference")`. The product `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") × [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")` has the two values `([true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference"), [Unit.unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit___unit "Documentation for Unit.unit in Lean Language Reference"))` and `([false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference"), [Unit.unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit___unit "Documentation for Unit.unit in Lean Language Reference"))`, and the sum `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") ⊕ [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")` has the three values `[Sum.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")`, `[Sum.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") [false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")`, and `[Sum.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference") [Unit.unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit___unit "Documentation for Unit.unit in Lean Language Reference")`. Similarly, `2×1=22 \times 1 = 22×1=2`, and `2+1=32 + 1 = 32+1=3`.
##  1.6.4. Messages You May Meet[🔗](find/?domain=Verso.Genre.Manual.section&name=polymorphism-messages "Permalink")
Not all definable structures or inductive types can have the type `Type`. In particular, if a constructor takes an arbitrary type as an argument, then the inductive type must have a different type. These errors usually state something about “universe levels”. For example, for this inductive type:
`inductive [MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example") : Type where   `Invalid universe level in constructor `MyType.ctor`: Parameter `α` has type   Type at universe level   2 which is not less than or equal to the inductive type's resulting universe level   1`| ctor : (α : Type) → α → [MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example")`
Lean gives the following error:

```
Invalid universe level in constructor `MyType.ctor`: Parameter `α` has type
  Type
at universe level
  2
which is not less than or equal to the inductive type's resulting universe level
  1
```

A later chapter describes why this is the case, and how to modify definitions to make them work. For now, try making the type an argument to the inductive type as a whole, rather than to the constructor.
Similarly, if a constructor's argument is a function that takes the datatype being defined as an argument, then the definition is rejected. For example:
``(kernel) arg #1 of 'MyType.ctor' has a non positive occurrence of the datatypes being declared`inductive [MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example") : Type where | ctor : ([MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example") → [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")) → [MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example")`
yields the message:

Forgetting an argument to an inductive type can also yield a confusing message. For example, when the argument `α` is not passed to `[MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example")` in `ctor`'s type:
`inductive [MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example") (α : Type) : Type where   | ctor : α → `type expected, got   ([MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example") : Type → Type)`[MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example")`
Lean replies with the following error:

```
type expected, got
  ([MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example") : Type → Type)
```

The error message is saying that `[MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example")`'s type, which is `Type → Type`, does not itself describe types. `[MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example")` requires an argument to become an actual honest-to-goodness type.
The same message can appear when type arguments are omitted in other contexts, such as in a type signature for a definition:
`inductive MyType (α : Type) : Type where   | ctor : α → [MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example") α``def ofFive : `type expected, got   ([MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example") : Type → Type)`[MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example") := ctor 5`
```
type expected, got
  ([MyType](Getting-to-Know-Lean/Polymorphism/#MyType "Definition of example") : Type → Type)
```

Evaluating expressions that use polymorphic types may trigger a situation in which Lean is incapable of displaying a value. The `[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference")` command evaluates the provided expression, using the expression's type to determine how to display the result. For some types, such as functions, this process fails, but Lean is perfectly capable of automatically generating display code for most other types. There is no need, for example, to provide Lean with any specific display code for `WoodSplittingTool`:
`inductive WoodSplittingTool where   | axe   | maul   | froe```WoodSplittingTool.axe`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") [WoodSplittingTool.axe](Getting-to-Know-Lean/Polymorphism/#WoodSplittingTool___axe "Definition of example")`
```
WoodSplittingTool.axe
```

There are limits to the automation that Lean uses here, however. `allTools` is a list of all three tools:
`def allTools : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") WoodSplittingTool := [   [WoodSplittingTool.axe](Getting-to-Know-Lean/Polymorphism/#WoodSplittingTool___axe "Definition of example"),   [WoodSplittingTool.maul](Getting-to-Know-Lean/Polymorphism/#WoodSplittingTool___maul "Definition of example"),   [WoodSplittingTool.froe](Getting-to-Know-Lean/Polymorphism/#WoodSplittingTool___froe "Definition of example") ]`
Evaluating it leads to an error:
``could not synthesize a `ToExpr`, `Repr`, or `ToString` instance for type   [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") WoodSplittingTool`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") allTools`
```
could not synthesize a `ToExpr`, `Repr`, or `ToString` instance for type
  [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") WoodSplittingTool

This is because Lean attempts to use code from a built-in table to display a list, but this code demands that display code for `WoodSplittingTool` already exists. This error can be worked around by instructing Lean to generate this display code when a datatype is defined, instead of at the last moment as part of `[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference")`, by adding `deriving [Repr](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Repr___mk "Documentation for Repr in Lean Language Reference")` to its definition:

##  1.6.5. Exercises[🔗](find/?domain=Verso.Genre.Manual.section&name=polymorphism-exercises "Permalink")
  * Write a function to find the last entry in a list. It should return an `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")`.
  * Write a function that finds the first entry in a list that satisfies a given predicate. Start the definition with `def List.findFirst? {α : Type} (xs : List α) (predicate : α → [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")) : [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") α := …`.
  * Write a function `Prod.switch` that switches the two fields in a pair for each other. Start the definition with `def Prod.switch {α β : Type} (pair : α × β) : β × α := …`.
  * Rewrite the `PetName` example to use a custom datatype and compare it to the version that uses `Sum`.
  * Write a function `zip` that combines two lists into a list of pairs. The resulting list should be as long as the shortest input list. Start the definition with `def zip {α β : Type} (xs : List α) (ys : List β) : List (α × β) := …`.
  * Write a polymorphic function `take` that returns the first `nnn` entries in a list, where `nnn` is a `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`. If the list contains fewer than `nnn` entries, then the resulting list should be the entire input list. `[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") take 3 ["bolete", "oyster"]` should yield `["bolete", "oyster"]`, and ``[[](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference")"bolete"[]](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference")`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") take 1 ["bolete", "oyster"]` should yield `["bolete"]`.
  * Using the analogy between types and arithmetic, write a function that distributes products over sums. In other words, it should have type `α × (β ⊕ γ) → (α × β) ⊕ (α × γ)`.
  * Using the analogy between types and arithmetic, write a function that turns multiplication by two into a sum. In other words, it should have type `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") × α → α ⊕ α`.
