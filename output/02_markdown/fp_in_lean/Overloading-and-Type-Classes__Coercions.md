#  3.6. Coercions[🔗](find/?domain=Verso.Genre.Manual.section&name=coercions "Permalink")
In mathematics, it is common to use the same symbol to stand for different aspects of some object in different contexts. For example, if a ring is referred to in a context where a set is expected, then it is understood that the ring's underlying set is what's intended. In programming languages, it is common to have rules to automatically translate values of one type into values of another type. Java allows a `byte` to be automatically promoted to an `int`, and Kotlin allows a non-nullable type to be used in a context that expects a nullable version of the type.
In Lean, both purposes are served by a mechanism called _coercions_. When Lean encounters an expression of one type in a context that expects a different type, it will attempt to coerce the expression before reporting a type error. Unlike Java, C, and Kotlin, the coercions are extensible by defining instances of type classes.
##  3.6.1. Strings and Paths[🔗](find/?domain=Verso.Genre.Manual.section&name=string-path-coercion "Permalink")
In the [source code to `feline`](Hello___-World___/Worked-Example___--cat/#handling-input), a `[String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")` is converted to a `[FilePath](https://lean-lang.org/doc/reference/4.26.0/IO/Files___-File-Handles___-and-Streams/#System___FilePath___mk "Documentation for System.FilePath in Lean Language Reference")` using the anonymous constructor syntax. ##  3.6.2. Positive Numbers[🔗](find/?domain=Verso.Genre.Manual.section&name=positive-number-coercion "Permalink")
Every positive number corresponds to a natural number. The function `Pos.toNat` that was defined earlier converts a `Pos` to the corresponding `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`:
`def Pos.toNat : Pos → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")   | Pos.one => 1   | [Pos.succ](Overloading-and-Type-Classes/Positive-Numbers/#Pos___succ "Definition of example") n => n.toNat + 1`
The function `[List.drop](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___drop "Documentation for List.drop in Lean Language Reference")`, with type `{α : Type} → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α`, removes a prefix of a list. Applying `[List.drop](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___drop "Documentation for List.drop in Lean Language Reference")` to a `Pos`, however, leads to a type error:
`[1, 2, 3, 4].[drop](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___drop "Documentation for List.drop in Lean Language Reference") (2 : Pos)`
```

  2
has type
  Pos
but is expected to have type
  [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")

##  3.6.3. Chaining Coercions[🔗](find/?domain=Verso.Genre.Manual.section&name=chaining-coercions "Permalink")
When searching for coercions, Lean will attempt to assemble a coercion out of a chain of smaller coercions. For example, there is already a coercion from `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` to `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")`. Because of that instance, combined with the `[Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference") Pos [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` instance, the following code is accepted:
`def oneInt : [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") := Pos.one`
This definition uses two coercions: from `Pos` to `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`, and then from `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` to `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")`.
The Lean compiler does not get stuck in the presence of circular coercions. For example, even if two types `A` and `B` can be coerced to one another, their mutual coercions can be used to find a path:
`inductive A where   | a  inductive B where   | b  instance : [Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference") A B where   [coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe.coe in Lean Language Reference") _ := [B.b](Overloading-and-Type-Classes/Coercions/#B___b "Definition of example")  instance : [Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference") B A where   [coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe.coe in Lean Language Reference") _ := [A.a](Overloading-and-Type-Classes/Coercions/#A___a "Definition of example")  instance : [Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference") [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") A where   [coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe.coe in Lean Language Reference") _ := [A.a](Overloading-and-Type-Classes/Coercions/#A___a "Definition of example")  def coercedToB : B := ()`
Remember: the double parentheses `()` is short for the constructor `[Unit.unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit___unit "Documentation for Unit.unit in Lean Language Reference")`. After deriving a `[Repr](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Repr___mk "Documentation for Repr in Lean Language Reference") B` instance with `[deriving](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") [instance](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") [Repr](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Repr___mk "Documentation for Repr in Lean Language Reference") [for](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") B`,
``B.b`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") [coercedToB](Overloading-and-Type-Classes/Coercions/#coercedToB "Definition of example")`
results in:

```
B.b
```

The `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")` type can be used similarly to nullable types in C# and Kotlin: the `[none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")` constructor represents the absence of a value. The Lean standard library defines a coercion from any type `α` to `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") α` that wraps the value in `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference")`. This allows option types to be used in a manner even more similar to nullable types, because `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference")` can be omitted. For instance, the function `List.last?` that finds the last entry in a list can be written without a `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference")` around the return value `x`:
`def List.last? : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") α   | [] => [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")   | [x] => x   | _ :: x :: xs => last? (x :: xs)`
Instance search finds the coercion, and inserts a call to `coe`, which wraps the argument in `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference")`. These coercions can be chained, so that nested uses of `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")` don't require nested `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference")` constructors:
`def perhapsPerhapsPerhaps : [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ([Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ([Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference"))) :=   "Please don't tell me"`

##  3.6.4. Non-Empty Lists and Dependent Coercions[🔗](find/?domain=Verso.Genre.Manual.section&name=CoeDep "Permalink")
An instance of `[Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference") α β` makes sense when the type `β` has a value that can represent each value from the type `α`. Coercing from `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` to `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` makes sense, because the type `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` contains all the natural numbers, but a coercion from `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` to `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` is a poor idea because `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` does not contain the negative numbers. Similarly, a coercion from non-empty lists to ordinary lists makes sense because the `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference")` type can represent every non-empty list:
`instance : [Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference") (NonEmptyList α) ([List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) where   [coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe.coe in Lean Language Reference")     | { [head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") := x, [tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") := xs } => x :: xs`
This allows non-empty lists to be used with the entire `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference")` API.
On the other hand, it is impossible to write an instance of `[Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference") ([List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) (NonEmptyList α)`, because there's no non-empty list that can represent the empty list. This limitation can be worked around by using another version of coercions, which are called _dependent coercions_. Dependent coercions can be used when the ability to coerce from one type to another depends on which particular value is being coerced. Just as the `OfNat` type class takes the particular `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` being overloaded as a parameter, dependent coercion takes the value being coerced as a parameter:
`class CoeDep (α : Type) (x : α) (β : Type) where   coe : β`
This is a chance to select only certain values, either by imposing further type class constraints on the value or by writing certain constructors directly. For example, any `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference")` that is not actually empty can be coerced to a `NonEmptyList`:
`instance : [CoeDep](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-Between-Types/#CoeDep___mk "Documentation for CoeDep in Lean Language Reference") ([List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) (x :: xs) (NonEmptyList α) where   [coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-Between-Types/#CoeDep___mk "Documentation for CoeDep.coe in Lean Language Reference") := { [head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") := x, [tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") := xs }`
##  3.6.5. Coercing to Types[🔗](find/?domain=Verso.Genre.Manual.section&name=CoeSort "Permalink")
In mathematics, it is common to have a concept that consists of a set equipped with additional structure. For example, a monoid is some set `SSS`, an element `sss` of `SSS`, and an associative binary operator on `SSS`, such that `sss` is neutral on the left and right of the operator. `SSS` is referred to as the “carrier set” of the monoid. The natural numbers with zero and addition form a monoid, because addition is associative and adding zero to any number is the identity. Similarly, the natural numbers with one and multiplication also form a monoid. Monoids are also widely used in functional programming: lists, the empty list, and the append operator form a monoid, as do strings, the empty string, and string append:

Given a monoid, it is possible to write the `foldMap` function that, in a single pass, transforms the entries in a list into a monoid's carrier set and then combines them using the monoid's operator. Because monoids have a neutral element, there is a natural result to return when the list is empty, and because the operator is associative, clients of the function don't have to care whether the recursive function combines elements from left to right or from right to left.
`def foldMap (M : Monoid) (f : α → M.[Carrier](Overloading-and-Type-Classes/Coercions/#Monoid___Carrier "Definition of example")) (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) : M.[Carrier](Overloading-and-Type-Classes/Coercions/#Monoid___Carrier "Definition of example") :=   let rec go (soFar : M.[Carrier](Overloading-and-Type-Classes/Coercions/#Monoid___Carrier "Definition of example")) : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → M.[Carrier](Overloading-and-Type-Classes/Coercions/#Monoid___Carrier "Definition of example")     | [] => soFar     | y :: ys => [go](Overloading-and-Type-Classes/Coercions/#MMM___foldMap___go "Definition of example") (M.[op](Overloading-and-Type-Classes/Coercions/#Monoid___op "Definition of example") soFar (f y)) ys   [go](Overloading-and-Type-Classes/Coercions/#MMM___foldMap___go "Definition of example") M.[neutral](Overloading-and-Type-Classes/Coercions/#Monoid___neutral "Definition of example") xs`
Even though a monoid consists of three separate pieces of information, it is common to just refer to the monoid's name in order to refer to its set. Instead of saying “Let A be a monoid and let _x_ and _y_ be elements of its carrier set”, it is common to say “Let _A_ be a monoid and let _x_ and _y_ be elements of _A_ ”. This practice can be encoded in Lean by defining a new kind of coercion, from the monoid to its carrier set.
The `[CoeSort](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-to-Sorts/#CoeSort___mk "Documentation for CoeSort in Lean Language Reference")` class is just like the `[Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference")` class, with the exception that the target of the coercion must be a _sort_ , namely `Type` or `Prop`. The term _sort_ in Lean refers to these types that classify other types—`Type` classifies types that themselves classify data, and `Prop` classifies propositions that themselves classify evidence of their truth. Just as `[Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference")` is checked when a type mismatch occurs, `[CoeSort](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-to-Sorts/#CoeSort___mk "Documentation for CoeSort in Lean Language Reference")` is used when something other than a sort is provided in a context where a sort would be expected.
The coercion from a monoid into its carrier set extracts the carrier:
`instance : [CoeSort](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-to-Sorts/#CoeSort___mk "Documentation for CoeSort in Lean Language Reference") Monoid Type where   [coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-to-Sorts/#CoeSort___mk "Documentation for CoeSort.coe in Lean Language Reference") m := m.[Carrier](Overloading-and-Type-Classes/Coercions/#Monoid___Carrier "Definition of example")`
With this coercion, the type signatures become less bureaucratic:
`def foldMap (M : Monoid) (f : α → M) (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) : M :=   let rec go (soFar : M) : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → M     | [] => soFar     | y :: ys => [go](Overloading-and-Type-Classes/Coercions/#foldMap___go "Definition of example") (M.[op](Overloading-and-Type-Classes/Coercions/#Monoid___op "Definition of example") soFar (f y)) ys   [go](Overloading-and-Type-Classes/Coercions/#foldMap___go "Definition of example") M.[neutral](Overloading-and-Type-Classes/Coercions/#Monoid___neutral "Definition of example") xs`
Another useful example of `[CoeSort](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-to-Sorts/#CoeSort___mk "Documentation for CoeSort in Lean Language Reference")` is used to bridge the gap between `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")` and `Prop`. As discussed in [the section on ordering and equality](Overloading-and-Type-Classes/Standard-Classes/#equality-and-ordering), Lean's `if` expression expects the condition to be a decidable proposition rather than a `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")`. Programs typically need to be able to branch based on Boolean values, however. Rather than have two kinds of `if` expression, the Lean standard library defines a coercion from `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")` to the proposition that the `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")` in question is equal to `[true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")`:
`instance : [CoeSort](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-to-Sorts/#CoeSort___mk "Documentation for CoeSort in Lean Language Reference") [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") Prop where   [coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-to-Sorts/#CoeSort___mk "Documentation for CoeSort.coe in Lean Language Reference") b := b = [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")`
In this case, the sort in question is `Prop` rather than `Type`.
##  3.6.6. Coercing to Functions[🔗](find/?domain=Verso.Genre.Manual.section&name=CoeFun "Permalink")
Many datatypes that occur regularly in programming consist of a function along with some extra information about it. For example, a function might be accompanied by a name to show in logs or by some configuration data. Additionally, putting a type in a field of a structure, similarly to the `Monoid` example, can make sense in contexts where there is more than one way to implement an operation and more manual control is needed than type classes would allow. For example, the specific details of values emitted by a JSON serializer may be important because another application expects a particular format. Sometimes, the function itself may be derivable from just the configuration data.
A type class called `CoeFun` can transform values from non-function types to function types. `CoeFun` has two parameters: the first is the type whose values should be transformed into functions, and the second is an output parameter that determines exactly which function type is being targeted.
`class CoeFun (α : Type) (makeFunctionType : [outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference") (α → Type)) where   coe : (x : α) → makeFunctionType x`
The second parameter is itself a function that computes a type. In Lean, types are first-class and can be passed to functions or returned from them, just like anything else.
For example, a function that adds a constant amount to its argument can be represented as a wrapper around the amount to add, rather than by defining an actual function:
`structure Adder where   howMuch : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`
A function that adds five to its argument has a `5` in the `howMuch` field:
`def add5 : Adder := ⟨5⟩`
This `Adder` type is not a function, and applying it to an argument results in an error:

When the value itself is needed to determine the right function type, then `[CoeFun](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-to-Function-Types/#CoeFun___mk "Documentation for CoeFun in Lean Language Reference")`'s second parameter is no longer ignored. ###  3.6.6.1. Aside: JSON as a String[🔗](find/?domain=Verso.Genre.Manual.section&name=json-string "Permalink")
It can be a bit difficult to understand JSON when encoded as Lean objects. To help make sure that the serialized response was what was expected, it can be convenient to write a simple converter from `JSON` to `[String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")`. The first step is to simplify the display of numbers. `JSON` doesn't distinguish between integers and floating point numbers, and the type `[Float](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Floating-Point-Numbers/#Float-next "Documentation for Float in Lean Language Reference")` is used to represent both. In Lean, `[Float.toString](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Floating-Point-Numbers/#Float___toString "Documentation for Float.toString in Lean Language Reference")` includes a number of trailing zeros:
``"5.000000"`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") (5 : [Float](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Floating-Point-Numbers/#Float-next "Documentation for Float in Lean Language Reference")).[toString](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Floating-Point-Numbers/#Float___toString "Documentation for Float.toString in Lean Language Reference")`
```
"5.000000"
```

The solution is to write a little function that cleans up the presentation by dropping all trailing zeros, followed by a trailing decimal point:
`def dropDecimals (numString : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") :=   [if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") numString.[contains](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___contains "Documentation for String.contains in Lean Language Reference") '.' The next step is to define a helper function to append a list of strings with a separator in between them:

This function is useful to account for comma-separated elements in JSON arrays and objects. `", ".[separate](Overloading-and-Type-Classes/Coercions/#String___separate "Definition of example") ["1", "2"]` yields `"1, 2"`, `", ".[separate](Overloading-and-Type-Classes/Coercions/#String___separate "Definition of example") ["1"]` yields `"1"`, and `", ".[separate](Overloading-and-Type-Classes/Coercions/#String___separate "Definition of example") []` yields `""`. Finally, a string escaping procedure is needed for JSON strings, so that the Lean string containing `"Hello!"` can be output as `"\"Hello!\""`. ##  3.6.8. Design Considerations[🔗](find/?domain=Verso.Genre.Manual.section&name=coercion-design-considerations "Permalink")
Coercions are a powerful tool that should be used responsibly. On the one hand, they can allow an API to naturally follow the everyday rules of the domain being modeled. This can be the difference between a bureaucratic mess of manual conversion functions and a clear program. As Abelson and Sussman wrote in the preface to _Structure and Interpretation of Computer Programs_ (MIT Press, 1996),
> Programs must be written for people to read, and only incidentally for machines to execute.
Coercions, used wisely, are a valuable means of achieving readable code that can serve as the basis for communication with domain experts. APIs that rely heavily on coercions have a number of important limitations, however. Think carefully about these limitations before using coercions in your own libraries.
First off, coercions are only applied in contexts where enough type information is available for Lean to know all of the types involved, because there are no output parameters in the coercion type classes. This means that a return type annotation on a function can be the difference between a type error and a successfully applied coercion. For example, the coercion from non-empty lists to lists makes the following program work:
`def lastSpider : [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") :=   [List.getLast?](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___getLast___ "Documentation for List.getLast? in Lean Language Reference") idahoSpiders`
On the other hand, if the type annotation is omitted, then the result type is unknown, so Lean is unable to find the coercion:
`def lastSpider :=   [List.getLast?](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___getLast___ "Documentation for List.getLast? in Lean Language Reference") `Application type mismatch: The argument   idahoSpiders has type   NonEmptyList [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") but is expected to have type   [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") ?m.3 in the application   [List.getLast?](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___getLast___ "Documentation for List.getLast? in Lean Language Reference") idahoSpiders`idahoSpiders`

More generally, when a coercion is not applied for some reason, the user receives the original type error, which can make it difficult to debug chains of coercions.
Finally, coercions are not applied in the context of field accessor notation. This means that there is still an important difference between expressions that need to be coerced and those that don't, and this difference is visible to users of your API.
