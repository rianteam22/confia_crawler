#  8.3. Arrays and Termination[🔗](find/?domain=Verso.Genre.Manual.section&name=array-termination "Permalink")
To write efficient code, it is important to select appropriate data structures. Linked lists have their place: in some applications, the ability to share the tails of lists is very important. However, most use cases for a variable-length sequential collection of data are better served by arrays, which have both less memory overhead and better locality.
Arrays, however, have two drawbacks relative to lists:
  1. Arrays are accessed through indexing, rather than by pattern matching, which imposes [proof obligations](Interlude___-Propositions___-Proofs___-and-Indexing/#props-proofs-indexing) in order to maintain safety.
  2. A loop that processes an entire array from left to right is a tail-recursive function, but it does not have an argument that decreases on each call.


Making effective use of arrays requires knowing how to prove to Lean that an array index is in bounds, and how to prove that an array index that approaches the size of the array also causes the program to terminate. Both of these are expressed using an inequality proposition, rather than propositional equality.
##  8.3.1. Inequality[🔗](find/?domain=Verso.Genre.Manual.section&name=inequality "Permalink")
Because different types have different notions of ordering, inequality is governed by two type classes, called `[LE](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LE___mk "Documentation for LE in Lean Language Reference")` and `[LT](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LT___mk "Documentation for LT in Lean Language Reference")`. The table in the section on [standard type classes](Overloading-and-Type-Classes/Standard-Classes/#equality-and-ordering) describes how these classes relate to the syntax:  
|  Expression  |  Desugaring  |  Class Name  |  
| --- | --- | --- |  
|  `x < y`  |  `[LT.lt](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LT___mk "Documentation for LT.lt in Lean Language Reference") x y`  |  `[LT](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LT___mk "Documentation for LT in Lean Language Reference")`  |  
|  `x ≤ y`  |  `[LE.le](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LE___mk "Documentation for LE.le in Lean Language Reference") x y`  |  `[LE](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LE___mk "Documentation for LE in Lean Language Reference")`  |  
|  `x > y`  |  `[LT.lt](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LT___mk "Documentation for LT.lt in Lean Language Reference") y x`  |  `[LT](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LT___mk "Documentation for LT in Lean Language Reference")`  |  
|  `x ≥ y`  |  `[LE.le](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LE___mk "Documentation for LE.le in Lean Language Reference") y x`  |  `[LE](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LE___mk "Documentation for LE in Lean Language Reference")`  |  
In other words, a type may customize the meaning of the `<` and `≤` operators, while `>` and `≥` derive their meanings from `<` and `≤`. The classes `[LT](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LT___mk "Documentation for LT in Lean Language Reference")` and `[LE](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LE___mk "Documentation for LE in Lean Language Reference")` have methods that return propositions rather than `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")`s:
`class LE (α : Type u) where   le : α → α → Prop  class LT (α : Type u) where   lt : α → α → Prop`
The instance of `[LE](Programming___-Proving___-and-Performance/Arrays-and-Termination/#Orders___LE "Definition of example")` for `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` delegates to `Nat.le`:
`instance : [LE](Programming___-Proving___-and-Performance/Arrays-and-Termination/#Orders___LE "Definition of example") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") where   [le](Programming___-Proving___-and-Performance/Arrays-and-Termination/#Orders___LE___le "Definition of example") := Nat.le`
Defining `Nat.le` requires a feature of Lean that has not yet been presented: it is an inductively-defined relation.
###  8.3.1.1. Inductively-Defined Propositions, Predicates, and Relations[🔗](find/?domain=Verso.Genre.Manual.section&name=inductive-props "Permalink")
`Nat.le` is an _inductively-defined relation_. Just as `inductive` can be used to create new datatypes, it can be used to create new propositions. When a proposition takes an argument, it is referred to as a _predicate_ that may be true for some, but not all, potential arguments. Propositions that take multiple arguments are called _relations_.
Each constructor of an inductively defined proposition is a way to prove it. In other words, the declaration of the proposition describes the different forms of evidence that it is true. A proposition with no arguments that has a single constructor can be quite easy to prove:
`inductive EasyToProve : Prop where   | heresTheProof : EasyToProve`
The proof consists of using its constructor:
`theorem fairlyEasy : EasyToProve := by⊢ EasyToProve   [constructor](https://lean-lang.org/doc/reference/4.26.0/Tactic-Proofs/Tactic-Reference/#constructor "Documentation for tactic in Lean Language Reference")All goals completed! 🐙`
In fact, the proposition `True`, which should always be easy to prove, is defined just like `EasyToProve`:
`inductive True : Prop where   | intro : True`
Inductively-defined propositions that don't take arguments are not nearly as interesting as inductively-defined datatypes. This is because data is interesting in its own right—the natural number `3` is different from the number `35`, and someone who has ordered 3 pizzas will be upset if 35 arrive at their door 30 minutes later. The constructors of a proposition describe ways in which the proposition can be true, but once a proposition has been proved, there is no need to know _which_ underlying constructors were used. This is why most interesting inductively-defined types in the `Prop` universe take arguments.
The inductively-defined predicate `IsThree` states that its argument is three:
`inductive IsThree : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → Prop where   | isThree : IsThree 3`
The mechanism used here is just like [indexed families such as `HasCol`](Programming-with-Dependent-Types/Worked-Example___-Typed-Queries/#column-pointers), except the resulting type is a proposition that can be proved rather than data that can be used.
Using this predicate, it is possible to prove that three is indeed three:
`theorem three_is_three : IsThree 3 := by⊢ IsThree 3   [constructor](https://lean-lang.org/doc/reference/4.26.0/Tactic-Proofs/Tactic-Reference/#constructor "Documentation for tactic in Lean Language Reference")All goals completed! 🐙`

The standard false proposition `[False](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Truth/#False "Documentation for False in Lean Language Reference")` has no constructors, making it impossible to provide direct evidence for. The only way to provide evidence for `[False](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Truth/#False "Documentation for False in Lean Language Reference")` is if an assumption is itself impossible, similarly to how `nomatch` can be used to mark code that the type system can see is unreachable. As described in [the initial Interlude on proofs](Interlude___-Propositions___-Proofs___-and-Indexing/#connectives), the negation `[Not](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Logical-Connectives/#Not "Documentation for Not in Lean Language Reference") A` is short for `A → [False](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Truth/#False "Documentation for False in Lean Language Reference")`. `[Not](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Logical-Connectives/#Not "Documentation for Not in Lean Language Reference") A` can also be written `¬A`.

###  8.3.1.2. Inequality of Natural Numbers[🔗](find/?domain=Verso.Genre.Manual.section&name=inequality-of-natural-numbers "Permalink")
The definition of `Nat.le` has a parameter and an index:
`inductive Nat.le (n : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → Prop   | refl : Nat.le n n   | step : Nat.le n m → Nat.le n (m + 1)`
The parameter `n` is the number that should be smaller, while the index is the number that should be greater than or equal to `n`. The `refl` constructor is used when both numbers are equal, while the `step` constructor is used when the index is greater than `n`.
From the perspective of evidence, a proof that `n≤kn \leq kn≤k` consists of finding some number `ddd` such that `n+d=mn + d = mn+d=m`. In Lean, the proof then consists of a `Nat.le.refl` constructor wrapped by `ddd` instances of `Nat.le.step`. Each `step` constructor adds one to its index argument, so `ddd` `step` constructors adds `ddd` to the larger number. For example, evidence that four is less than or equal to seven consists of three `step`s around a `refl`:
`theorem four_le_seven : 4 ≤ 7 :=   open Nat.le in   [step](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___le___refl "Documentation for Nat.le.step in Lean Language Reference") ([step](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___le___refl "Documentation for Nat.le.step in Lean Language Reference") ([step](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___le___refl "Documentation for Nat.le.step in Lean Language Reference") [refl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___le___refl "Documentation for Nat.le.refl in Lean Language Reference")))`
The strict less-than relation is defined by adding one to the number on the left:
`def Nat.lt (n m : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : Prop :=   Nat.le (n + 1) m  instance : [LT](Programming___-Proving___-and-Performance/Arrays-and-Termination/#Orders___LT "Definition of example") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") where   [lt](Programming___-Proving___-and-Performance/Arrays-and-Termination/#Orders___LT___lt "Definition of example") := [Nat.lt](Programming___-Proving___-and-Performance/Arrays-and-Termination/#Orders___Nat___lt "Definition of example")`
Evidence that four is strictly less than seven consists of two `[step](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___le___refl "Documentation for Nat.le.step in Lean Language Reference")`'s around a `[refl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___le___refl "Documentation for Nat.le.refl in Lean Language Reference")`:
`theorem four_lt_seven : 4 < 7 :=   open Nat.le in   [step](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___le___refl "Documentation for Nat.le.step in Lean Language Reference") ([step](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___le___refl "Documentation for Nat.le.step in Lean Language Reference") [refl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___le___refl "Documentation for Nat.le.refl in Lean Language Reference"))`
This is because `4 < 7` is equivalent to `5 ≤ 7`.
##  8.3.2. Proving Termination[🔗](find/?domain=Verso.Genre.Manual.section&name=proving-termination "Permalink")
The function `Array.map` transforms an array with a function, returning a new array that contains the result of applying the function to each element of the input array. Writing it as a tail-recursive function follows the usual pattern of delegating to a function that passes the output array in an accumulator. The accumulator is initialized with an empty array. The accumulator-passing helper function also takes an argument that tracks the current index into the array, which starts at `0`:
`def Array.map (f : α → β) (arr : [Array](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___mk "Documentation for Array in Lean Language Reference") α) : [Array](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___mk "Documentation for Array in Lean Language Reference") β :=   arrayMapHelper f arr [Array.empty](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___empty "Documentation for Array.empty in Lean Language Reference") 0`
The helper should, at each iteration, check whether the index is still in bounds. Lean accepts the modified program, even though the recursive call is not made on an argument to one of the input constructors. In fact, both the accumulator and the index grow, rather than shrinking.
Behind the scenes, Lean's proof automation constructs a termination proof. Reconstructing this proof can make it easier to understand the cases that Lean cannot automatically recognize.
Why does `arrayMapHelper` terminate? Each iteration checks whether the index `i` is still in bounds for the array `arr`. If so, `i` is incremented and the loop repeats. If not, the program terminates. Because `arr.[size](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___size "Documentation for Array.size in Lean Language Reference")` is a finite number, `i` can be incremented only a finite number of times. Even though no argument to the function decreases on each call, `arr.[size](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___size "Documentation for Array.size in Lean Language Reference") - i` decreases toward zero.
The value that decreases at each recursive call is called a _measure_. Lean can be instructed to use a specific expression as the measure of termination by providing a `termination_by` clause at the end of a definition. For `arrayMapHelper`, the explicit measure looks like this:
`def arrayMapHelper (f : α → β) (arr : [Array](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___mk "Documentation for Array in Lean Language Reference") α)     (soFar : [Array](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___mk "Documentation for Array in Lean Language Reference") β) (i : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Array](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___mk "Documentation for Array in Lean Language Reference") β :=   [if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference") inBounds : i < arr.[size](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___size "Documentation for Array.size in Lean Language Reference") [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference")     arrayMapHelper f arr (soFar.[push](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___push "Documentation for Array.push in Lean Language Reference") (f arr[i])) (i + 1)   [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference") soFar termination_by arr.[size](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___size "Documentation for Array.size in Lean Language Reference") - i`
