#  4.5. The IO Monad[🔗](find/?domain=Verso.Genre.Manual.section&name=io-monad "Permalink")
`[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` as a monad can be understood from two perspectives, which were described in the section on [running programs](Hello___-World___/Running-a-Program/#running-a-program). Each can help to understand the meanings of `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` and `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` for `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")`.
From the first perspective, an `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` action is an instruction to Lean's run-time system. For example, the instruction might be “read a string from this file descriptor, then re-invoke the pure Lean code with the string”. This perspective is an _exterior_ one, viewing the program from the perspective of the operating system. In this case, `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` is an `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` action that does not request any effects from the RTS, and `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` instructs the RTS to first carry out one potentially-effectful operation and then invoke the rest of the program with the resulting value.
From the second perspective, an `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` action transforms the whole world. `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` actions are actually pure, because they receive a unique world as an argument and then return the changed world. This perspective is an _interior_ one that matches how `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` is represented inside of Lean. The world is represented in Lean as a token, and the `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` monad is structured to make sure that each token is used exactly once.

`[EIO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#EIO "Documentation for EIO in Lean Language Reference") ε α` represents `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` actions that will either terminate with an error of type `ε` or succeed with a value of type `α`. This means that, like the `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference") ε` monad, the `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` monad includes the ability to define error handling and exceptions.
Peeling back another layer, `[EIO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#EIO "Documentation for EIO in Lean Language Reference")` is itself defined in terms of a simpler structure:
``def EIO : Type → Type → Type := fun ε α => [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε IO.RealWorld α`#print [EIO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#EIO "Documentation for EIO in Lean Language Reference")`
```
def EIO : Type → Type → Type :=
fun ε α => [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε IO.RealWorld α
```

The `[EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference")` monad includes both errors and state—it's similar to a combination of `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")` and `State`. It is defined using another type, `EST.Out`:
``def EST : Type → Type → Type → Type := fun ε σ α => Void σ → EST.Out ε σ α`#print [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference")`
```
def EST : Type → Type → Type → Type :=
fun ε σ α => Void σ → EST.Out ε σ α
```

In other words, a program with type `[EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε σ α` is a function that accepts an initial state of type `σ` and returns an `EST.Out ε σ α`. The state is wrapped in the type `Void`, which is an internal primitive that causes a value to be erased from compiled code; `Void σ` has the same representation as `[Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")`.
`EST.Out` is very much like the definition of `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")`, with one constructor that indicates a successful termination and one constructor that indicates an error:
``inductive EST.Out : Type → Type → Type → Type number of parameters: 3 constructors: EST.Out.ok : {ε σ α : Type} → α → Void σ → EST.Out ε σ α EST.Out.error : {ε σ α : Type} → ε → Void σ → EST.Out ε σ α`#print EST.Out`
```
inductive EST.Out : Type → Type → Type → Type
number of parameters: 3
constructors:
EST.Out.ok : {ε σ α : Type} → α → Void σ → EST.Out ε σ α
EST.Out.error : {ε σ α : Type} → ε → Void σ → EST.Out ε σ α
```

Just like `Except ε α`, the `ok` constructor includes a result of type `α`, and the `error` constructor includes an exception of type `ε`. Unlike `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")`, both constructors have an additional state field that includes the final state of the computation.
The `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance for `[EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε σ` requires `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` and `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")`. Just as with `State`, the implementation of `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` for `[EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference")` accepts an initial state and returns it unchanged, and just as with `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")`, it returns its argument in the `ok` constructor:
``protected def EST.pure : {α ε σ : Type} → α → [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε σ α := fun {α ε σ} a s => EST.Out.ok a s`#print EST.pure`
```
protected def EST.pure : {α ε σ : Type} → α → [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε σ α :=
fun {α ε σ} a s => EST.Out.ok a s
```

`protected` means that the full name `EST.pure` is needed even if the `[EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference")` namespace has been opened.
Similarly, `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` for `[EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference")` takes an initial state as an argument. It passes this initial state to its first action. Like `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` for `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")`, it then checks whether the result is an error. If so, the error is returned unchanged and the second argument to `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` remains unused. If the result was a success, then the second argument is applied to both the returned value and to the resulting state.
``protected def EST.bind : {ε σ α β : Type} → [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε σ α → (α → [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε σ β) → [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε σ β := fun {ε σ α β} x f s =>   match x s with   | EST.Out.ok a s => f a s   | EST.Out.error e s => EST.Out.error e s`#print EST.bind`
```
protected def EST.bind : {ε σ α β : Type} → [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε σ α → (α → [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε σ β) → [EST](https://lean-lang.org/doc/reference/4.26.0/IO/Mutable-References/#EST "Documentation for EST in Lean Language Reference") ε σ β :=
fun {ε σ α β} x f s =>
  match x s with
  | EST.Out.ok a s => f a s
  | EST.Out.error e s => EST.Out.error e s
```

Putting all of this together, `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` is a monad that tracks state and errors at the same time. The collection of available errors is that given by the datatype `[IO.Error](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO___Error___alreadyExists "Documentation for IO.Error in Lean Language Reference")`, which has constructors that describe many things that can go wrong in a program. The state is a type that represents the real world, called `IO.RealWorld`. Each basic `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` action receives this real world and returns another one, paired either with an error or a result. In `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")`, `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` returns the world unchanged, while `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` passes the modified world from one action into the next action.
Because the entire universe doesn't fit in a computer's memory, the world being passed around is just a representation. So long as world tokens are not re-used, the representation is safe. The type `IO.RealWorld` is a trivial primitive type that does not need any representation at all, because it is only used inside of `Void`.
