#  6.1. Combining IO and Reader[🔗](find/?domain=Verso.Genre.Manual.section&name=io-reader "Permalink")
One case where a reader monad can be useful is when there is some notion of the “current configuration” of the application that is passed through many recursive calls. An example of such a program is `tree`, which recursively prints the files in the current directory and its subdirectories, indicating their tree structure using characters. The version of `tree` in this chapter, called `doug` after the mighty Douglas Fir tree that adorns the west coast of North America, provides the option of Unicode box-drawing characters or their ASCII equivalents when indicating directory structure.
For example, the following commands create a directory structure and some empty files in a directory called `doug-demo`:
`cd doug-demo``mkdir -p a/b/c``mkdir -p a/d``mkdir -p a/e/f``touch a/b/hello``touch a/d/another-file``touch a/e/still-another-file-again`
Running `doug` results in the following:
`doug``├── doug-demo/ │   ├── a/ │   │   ├── b/ │   │   │   ├── c/ │   │   │   ├── hello │   │   ├── d/ │   │   │   ├── another-file │   │   ├── e/ │   │   │   ├── f/ │   │   │   ├── still-another-file-again `
##  6.1.1. Implementation[🔗](find/?domain=Verso.Genre.Manual.section&name=reader-io-implementation "Permalink")
Internally, `doug` passes a configuration value downwards as it recursively traverses the directory structure. This configuration contains two fields: `useASCII` determines whether to use Unicode box-drawing characters or ASCII vertical line and dash characters to indicate structure, and `currentPrefix` contains a string to prepend to each line of output. As the current directory deepens, the prefix string accumulates indicators of being in a directory. The configuration is a structure:
`structure Config where   useASCII : [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") := [false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")   currentPrefix : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") := ""`
This structure has default definitions for both fields. The default `Config` uses Unicode display with no prefix.
Users who invoke `doug` will need to be able to provide command-line arguments. The usage information is as follows:
`def usage : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") :=   "Usage: doug [--ascii] Options: \t--ascii\tUse ASCII characters to display the directory structure"`
Accordingly, a configuration can be constructed by examining a list of command-line arguments:
`def configFromArgs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") → [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") Config   | [] => [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") {} -- both fields default   | ["--ascii"] => [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") {useASCII := [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")}   | _ => [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`

The call to `toEntry` is a [nested action](Hello___-World___/Additional-Conveniences/#nested-actions)—the parentheses are optional in positions where the arrow couldn't have any other meaning, such as `match`. When the filename doesn't correspond to an entry in the tree (e.g. because it is `..`), `dirTree` does nothing. When the filename points to an ordinary file, `dirTree` calls a helper to show it with the current configuration. When the filename points to a directory, it is shown with a helper, and then its contents are recursively shown in a new configuration in which the prefix has been extended to account for being in a new directory. The contents of the directory are sorted in order to make the output deterministic, compared according to `dirLT`.
`def dirLT (e1 : [IO.FS.DirEntry](https://lean-lang.org/doc/reference/4.26.0/IO/Files___-File-Handles___-and-Streams/#IO___FS___DirEntry___mk "Documentation for IO.FS.DirEntry in Lean Language Reference")) (e2 : [IO.FS.DirEntry](https://lean-lang.org/doc/reference/4.26.0/IO/Files___-File-Handles___-and-Streams/#IO___FS___DirEntry___mk "Documentation for IO.FS.DirEntry in Lean Language Reference")) : [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") :=   e1.[fileName](https://lean-lang.org/doc/reference/4.26.0/IO/Files___-File-Handles___-and-Streams/#IO___FS___DirEntry___mk "Documentation for IO.FS.DirEntry.fileName in Lean Language Reference") < e2.[fileName](https://lean-lang.org/doc/reference/4.26.0/IO/Files___-File-Handles___-and-Streams/#IO___FS___DirEntry___mk "Documentation for IO.FS.DirEntry.fileName in Lean Language Reference")`

Iterating an IO action over a list of directory contents is achieved using `doList`. Because `doList` carries out all the actions in a list and does not base control-flow decisions on the values returned by any of the actions, the full power of `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` is not necessary, and it will work for any `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")`:
`def doList [[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference") f] : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → (α → f [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")) → f [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference")   | [], _ => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") ()   | x :: xs, action =>     action x *>     doList xs action`
##  6.1.2. Using a Custom Monad[🔗](find/?domain=Verso.Genre.Manual.section&name=reader-io-custom-monad "Permalink")
While this implementation of `doug` works, manually passing the configuration around is verbose and error-prone. The type system will not catch it if the wrong configuration is passed downwards, for instance. A reader effect ensures that the same configuration is passed to all recursive calls, unless it is manually overridden, and it helps make the code less verbose.
To create a version of `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` that is also a reader of `Config`, first define the type and its `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance, following the recipe from [the evaluator example](Monads/Example___-Arithmetic-in-Monads/#custom-environments):
`def ConfigIO (α : Type) : Type :=   Config → [IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference") α  instance : [Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") ConfigIO where   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x := fun _ => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x   [bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") result next := fun cfg => [do](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Syntax/#Lean___Parser___Term___do "Documentation for syntax in Lean Language Reference")     let v ← result cfg     next v cfg`
The difference between this `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance and the one for `Reader` is that this one uses `do`-notation in the `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` monad as the body of the function that `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` returns, rather than applying `next` directly to the value returned from `result`. Any `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` effects performed by `result` must occur before `next` is invoked, which is ensured by the `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` monad's `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` operator. `ConfigIO` is not universe polymorphic because the underlying `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` type is also not universe polymorphic.
Running a `ConfigIO` action involves transforming it into an `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` action by providing it with a configuration:
`def ConfigIO.run (action : ConfigIO α) (cfg : Config) : [IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference") α :=   action cfg`
This function is not really necessary, as a caller could simply provide the configuration directly. However, naming the operation can make it easier to see which parts of the code are intended to run in which monad.
The next step is to define a means of accessing the current configuration as part of `ConfigIO`:
`def currentConfig : ConfigIO Config :=   fun cfg => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") cfg`
This is just like `read` from [the evaluator example](Monads/Example___-Arithmetic-in-Monads/#custom-environments), except it uses `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")`'s `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` to return its value rather than doing so directly. Because entering a directory modifies the current configuration for the scope of a recursive call, it will be necessary to have a way to override a configuration:
`def locally (change : Config → Config) (action : ConfigIO α) : ConfigIO α :=   fun cfg => action (change cfg)`
Much of the code used in `doug` has no need for configurations, and `doug` calls ordinary Lean `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` actions from the standard library that certainly don't need a `Config`. Ordinary `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` actions can be run using `runIO`, which ignores the configuration argument:
`def runIO (action : [IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference") α) : ConfigIO α :=   fun _ => action`

This custom monad has a number of advantages over passing configurations manually:
  1. It is easier to ensure that configurations are passed down unchanged, except when changes are desired
  2. The concern of passing the configuration onwards is more clearly separated from the concern of printing directory contents
  3. As the program grows, there will be more and more intermediate layers that do nothing with configurations except propagate them, and these layers don't need to be rewritten as the configuration logic changes


However, there are also some clear downsides:
  1. As the program evolves and the monad requires more features, each of the basic operators such as `locally` and `currentConfig` will need to be updated
  2. Wrapping ordinary `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` actions in `runIO` is noisy and distracts from the flow of the program
  3. Writing monads instances by hand is repetitive, and the technique for adding a reader effect to another monad is a design pattern that requires documentation and communication overhead


Using a technique called _monad transformers_ , all of these downsides can be addressed. A monad transformer takes a monad as an argument and returns a new monad. Monad transformers consist of:
  1. A definition of the transformer itself, which is typically a function from types to types
  2. A `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance that assumes the inner type is already a monad
  3. An operator to “lift” an action from the inner monad to the transformed monad, akin to `runIO`


##  6.1.3. Adding a Reader to Any Monad[🔗](find/?domain=Verso.Genre.Manual.section&name=ReaderT "Permalink")
Adding a reader effect to `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` was accomplished in `ConfigIO` by wrapping `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference") α` in a function type. The Lean standard library contains a function that can do this to _any_ polymorphic type, called `ReaderT`:
`def ReaderT (ρ : Type u) (m : Type u → Type v) (α : Type u) :     Type (max u v) :=   ρ → m α`
Its arguments are as follows:
  * `ρ` is the environment that is accessible to the reader
  * `m` is the monad that is being transformed, such as `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")`
  * `α` is the type of values being returned by the monadic computation Both `α` and `ρ` are in the same universe because the operator that retrieves the environment in the monad will have type `m ρ`.


With `ReaderT`, `ConfigIO` becomes:
`abbrev ConfigIO (α : Type) : Type := ReaderT Config [IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference") α`
It is an `abbrev` because `ReaderT` has many useful features defined in the standard library that a non-reducible definition would hide. Rather than taking responsibility for making these work directly for `ConfigIO`, it's easier to simply have `ConfigIO` behave identically to `ReaderT Config [IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")`.
The manually-written `currentConfig` obtained the environment out of the reader. This effect can be defined in a generic form for all uses of `ReaderT`, under the name `read`:
`def read [[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") m] : ReaderT ρ m ρ :=    fun env => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") env`
However, not every monad that provides a reader effect is built with `ReaderT`. The type class `MonadReader` allows any monad to provide a `read` operator:
`class MonadReader (ρ : [outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference") (Type u)) (m : Type u → Type v) :     Type (max (u + 1) v) where   read : m ρ  instance [[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") m] : MonadReader ρ (ReaderT ρ m) where   read := fun env => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") env  [export](https://lean-lang.org/doc/reference/4.26.0/Namespaces-and-Sections/#Lean___Parser___Command___export "Documentation for syntax in Lean Language Reference") MonadReader (read)`
The type `ρ` is an output parameter because any given monad typically only provides a single type of environment through a reader, so automatically selecting it when the monad is known makes programs more convenient to write.
The `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance for `ReaderT` is essentially the same as the `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance for `ConfigIO`, except `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` has been replaced by some arbitrary monad argument `m`:
`instance [[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") m] : [Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") (ReaderT ρ m) where   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x := fun _ => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x   [bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") result next := fun env => [do](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Syntax/#Lean___Parser___Term___do "Documentation for syntax in Lean Language Reference")     let v ← result env     next v env`
The next step is to eliminate uses of `runIO`. When Lean encounters a mismatch in monad types, it automatically attempts to use a type class called `MonadLift` to transform the actual monad into the expected monad. This process is similar to the use of coercions. `MonadLift` is defined as follows:
`class MonadLift (m : Type u → Type v) (n : Type u → Type w) where   monadLift : {α : Type u} → m α → n α`
The method `monadLift` translates from the monad `m` to the monad `n`. The process is called “lifting” because it takes an action in the embedded monad and makes it into an action in the surrounding monad. In this case, it will be used to “lift” from `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` to `ReaderT Config [IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")`, though the instance works for _any_ inner monad `m`:
`instance : [MonadLift](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Lifting-Monads/#MonadLift___mk "Documentation for MonadLift in Lean Language Reference") m (ReaderT ρ m) where   [monadLift](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Lifting-Monads/#MonadLift___mk "Documentation for MonadLift.monadLift in Lean Language Reference") action := fun _ => action`
The implementation of `[monadLift](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Lifting-Monads/#MonadLift___mk "Documentation for MonadLift.monadLift in Lean Language Reference")` is very similar to that of `runIO`. Indeed, it is enough to define `showFileName` and `showDirName` without using `runIO`:
`def showFileName (file : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) : ConfigIO [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") := [do](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Syntax/#Lean___Parser___Term___do "Documentation for syntax in Lean Language Reference")   [IO.println](https://lean-lang.org/doc/reference/4.26.0/IO/Console-Output/#IO___println "Documentation for IO.println in Lean Language Reference") s!"{(← read).currentPrefix} {file}"  def showDirName (dir : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) : ConfigIO [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") := [do](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Syntax/#Lean___Parser___Term___do "Documentation for syntax in Lean Language Reference")   [IO.println](https://lean-lang.org/doc/reference/4.26.0/IO/Console-Output/#IO___println "Documentation for IO.println in Lean Language Reference") s!"{(← read).currentPrefix} {dir}/"`
One final operation from the original `ConfigIO` remains to be translated to a use of `ReaderT`: `locally`. The definition can be translated directly to `ReaderT`, but the Lean standard library provides a more general version. The standard version is called `withReader`, and it is part of a type class called `MonadWithReader`:
`class MonadWithReader (ρ : [outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference") (Type u)) (m : Type u → Type v) where   withReader {α : Type u} : (ρ → ρ) → m α → m α`
Just as in `MonadReader`, the environment `ρ` is an `[outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference")`. The `withReader` operation is exported, so that it doesn't need to be written with the type class name before it:
`[export](https://lean-lang.org/doc/reference/4.26.0/Namespaces-and-Sections/#Lean___Parser___Command___export "Documentation for syntax in Lean Language Reference") MonadWithReader (withReader)`
The instance for `ReaderT` is essentially the same as the definition of `locally`:
`instance : MonadWithReader ρ (ReaderT ρ m) where   withReader change action :=     fun cfg => action (change cfg)`

Replacing the custom `ConfigIO` type with `ReaderT` did not save a large number of lines of code in this section. However, rewriting the code using components from the standard library does have long-term benefits. First, readers who know about `ReaderT` don't need to take time to understand the `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance for `ConfigIO`, working backwards to the meaning of monad itself. Instead, they can be confident in their initial understanding. Next, adding further effects to the monad (such as a state effect to count the files in each directory and display a count at the end) requires far fewer changes to the code, because the monad transformers and `[MonadLift](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Lifting-Monads/#MonadLift___mk "Documentation for MonadLift in Lean Language Reference")` instances provided in the library work well together. Finally, using a set of type classes included in the standard library, polymorphic code can be written in such a way that it can work with a variety of monads without having to care about details like the order in which the monad transformers were applied. Just as some functions work in any monad, others can work in any monad that provides a certain type of state, or a certain type of exceptions, without having to specifically describe the _way_ in which a particular concrete monad provides the state or exceptions.
