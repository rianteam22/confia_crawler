#  5.2. Applicative Functors[🔗](find/?domain=Verso.Genre.Manual.section&name=applicative "Permalink")
An _applicative functor_ is a functor that has two additional operations available: `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` and `[seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference")`. `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` is the same operator used in `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")`, because `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` in fact inherits from `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")`. `[seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference")` is much like `[map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")`: it allows a function to be used in order to transform the contents of a datatype. However, with `[seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference")`, the function is itself contained in the datatype: `f (α → β) → ([Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") → f α) → f β`. Having the function under the type `f` allows the `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` instance to control how the function is applied, while `[Functor.map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")` unconditionally applies a function. The second argument has a type that begins with `[Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") →` to allow the definition of `[seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference")` to short-circuit in cases where the function will never be applied.
The value of this short-circuiting behavior can be seen in the instance of `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference") [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")`:
`instance : [Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference") [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") where   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x := [.some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x   [seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference") f x :=     [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") f [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")     | [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference") => [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")     | [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") g => g <$> x ()`
In this case, if there is no function for `[seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference")` to apply, then there is no need to compute its argument, so `x` is never called. The same consideration informs the instance of `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` for `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")`:
`instance : [Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference") ([Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference") ε) where   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x := [.ok](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except.ok in Lean Language Reference") x   [seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference") f x :=     [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") f [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")     | [.error](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except.error in Lean Language Reference") e => [.error](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except.error in Lean Language Reference") e     | [.ok](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except.ok in Lean Language Reference") g => g <$> x ()`
This short-circuiting behavior depends only on the `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")` or `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")` structures that _surround_ the function, rather than on the function itself.
Monads can be seen as a way of capturing the notion of sequentially executing statements into a pure functional language. The result of one statement can affect which further statements run. This can be seen in the type of `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")`: `m α → (α → m β) → m β`. The first statement's resulting value is an input into a function that computes the next statement to execute. Successive uses of `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` are like a sequence of statements in an imperative programming language, and `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` is powerful enough to implement control structures like conditionals and loops.
Following this analogy, `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` captures function application in a language that has side effects. The arguments to a function in languages like Kotlin or C# are evaluated from left to right. Side effects performed by earlier arguments occur before those performed by later arguments. A function is not powerful enough to implement custom short-circuiting operators that depend on the specific _value_ of an argument, however.
Typically, `seq` is not invoked directly. Instead, the operator `<*>` is used. This operator wraps its second argument in `fun () => ...`, simplifying the call site. In other words, `E1 <*> E2` is syntactic sugar for `[Seq.seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference") E1 (fun () => E2)`.
The key feature that allows `seq` to be used with multiple arguments is that a multiple-argument Lean function is really a single-argument function that returns another function that's waiting for the rest of the arguments. In other words, if the first argument to `seq` is awaiting multiple arguments, then the result of the `seq` will be awaiting the rest. For example, `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") Plus.plus` can have the type `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference"))`. Providing one argument, `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") Plus.plus <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") 4`, results in the type `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference"))`. This can itself be used with `seq`, so `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") Plus.plus <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") 4 <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") 7` has the type `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`.
Not every functor is applicative. `Pair` is like the built-in product type `[Prod](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod in Lean Language Reference")`:
`structure Pair (α β : Type) : Type where   first : α   second : β`
Like `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")`, `Pair` has type `Type → Type → Type`. This means that `Pair α` has type `Type → Type`, and a `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` instance is possible:
`instance : [Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference") (Pair α) where   [map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") f x := ⟨x.[first](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Pair___first "Definition of example"), f x.[second](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Pair___second "Definition of example")⟩`
This instance obeys the `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` contract.
The two properties to check are that `id <$> Pair.mk x y`` = ``Pair.mk x y` and that `f <$> g <$> Pair.mk x y`` = ``(f ∘ g) <$> Pair.mk x y`. The first property can be checked by just stepping through the evaluation of the left side, and noticing that it evaluates to the right side:
`id <$> Pair.mk x y``Pair.mk x (id y)``Pair.mk x y`
The second can be checked by stepping through both sides, and noting that they yield the same result:
`f <$> g <$> Pair.mk x y``f <$> Pair.mk x (g y)``Pair.mk x (f (g y))`
`(f ∘ g) <$> Pair.mk x y``Pair.mk x ((f ∘ g) y)``Pair.mk x (f (g y))`

##  5.2.1. A Non-Monadic Applicative[🔗](find/?domain=Verso.Genre.Manual.section&name=validate "Permalink")
When validating user input to a form, it's generally considered to be best to provide many errors at once, rather than one error at a time. This allows the user to have an overview of what is needed to please the computer, rather than feeling badgered as they correct the errors field by field.
Ideally, validating user input will be visible in the type of the function that's doing the validating. It should return a datatype that is specific—checking that a text box contains a number should return an actual numeric type, for instance. A validation routine could throw an exception when the input does not pass validation. Exceptions have a major drawback, however: they terminate the program at the first error, making it impossible to accumulate a list of errors.
On the other hand, the common design pattern of accumulating a list of errors and then failing when it is non-empty is also problematic. A long nested sequences of `if` statements that validate each sub-section of the input data is hard to maintain, and it's easy to lose track of an error message or two. Ideally, validation can be performed using an API that enables a new value to be returned yet automatically tracks and accumulates error messages.
An applicative functor called `Validate` provides one way to implement this style of API. Like the `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")` monad, `Validate` allows a new value to be constructed that characterizes the validated data accurately. Unlike `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")`, it allows multiple errors to be accumulated, without a risk of forgetting to check whether the list is empty.
###  5.2.1.1. User Input[🔗](find/?domain=Verso.Genre.Manual.section&name=user-input "Permalink")
As an example of user input, take the following structure:
`structure RawInput where   name : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")   birthYear : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")`
The business logic to be implemented is the following:
  1. The name may not be empty
  2. The birth year must be numeric and non-negative
  3. The birth year must be greater than 1900, and less than or equal to the year in which the form is validated


Representing these as a datatype will require a new feature, called _subtypes_. With this tool in hand, a validation framework can be written that uses an applicative functor to track errors, and these rules can be implemented in the framework.
###  5.2.1.2. Subtypes[🔗](find/?domain=Verso.Genre.Manual.section&name=subtypes "Permalink")
Representing these conditions is easiest with one additional Lean type, called `Subtype`:
`structure Subtype {α : Type} (p : α → Prop) where   val : α   property : p val`
This structure has two type parameters: an implicit parameter that is the type of data `α`, and an explicit parameter `p` that is a predicate over `α`. A _predicate_ is a logical statement with a variable in it that can be replaced with a value to yield an actual statement, like the [parameter to ``](Overloading-and-Type-Classes/Arrays-and-Indexing/#overloading-indexing)`[GetElem](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#GetElem___mk "Documentation for GetElem in Lean Language Reference")` that describes what it means for an index to be in bounds for a lookup. In the case of `Subtype`, the predicate slices out some subset of the values of `α` for which the predicate holds. The structure's two fields are, respectively, a value from `α` and evidence that the value satisfies the predicate `p`. Lean has special syntax for `Subtype`. If `p` has type `α → Prop`, then the type `Subtype p` can also be written `{x : α // p x}`, or even `{x // p x}` when the type `α` can be inferred automatically.
[Representing positive numbers as inductive types](Overloading-and-Type-Classes/Positive-Numbers/#positive-numbers) is clear and easy to program with. However, it has a key disadvantage. While `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` and `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` have the structure of ordinary inductive types from the perspective of Lean programs, the compiler treats them specially and uses fast arbitrary-precision number libraries to implement them. This is not the case for additional user-defined types. However, a subtype of `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` that restricts it to non-zero numbers allows the new type to use the efficient representation while still ruling out zero at compile time:
`def FastPos : Type := {x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") // x > 0}`
The smallest fast positive number is still one. Now, instead of being a constructor of an inductive type, it's an instance of a structure that's constructed with angle brackets. The first argument is the underlying `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`, and the second argument is the evidence that said `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` is greater than zero:
`def one : [FastPos](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#FastPos___FastPos "Definition of example") := ⟨1, by⊢ 1 > 0 [decide](https://lean-lang.org/doc/reference/4.26.0/Tactic-Proofs/Tactic-Reference/#decide "Documentation for tactic in Lean Language Reference")All goals completed! 🐙⟩`
The proposition `1 > 0` is decidable, so the `[decide](https://lean-lang.org/doc/reference/4.26.0/Tactic-Proofs/Tactic-Reference/#decide "Documentation for tactic in Lean Language Reference")` tactic produces the necessary evidence. The `[OfNat](https://lean-lang.org/doc/reference/4.26.0/Terms/Literals/#OfNat___mk "Documentation for OfNat in Lean Language Reference")` instance is very much like that for `Pos`, except it uses a short tactic proof to provide evidence that `n + 1 > 0`:
`instance : [OfNat](https://lean-lang.org/doc/reference/4.26.0/Terms/Literals/#OfNat___mk "Documentation for OfNat in Lean Language Reference") [FastPos](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#FastPos___FastPos "Definition of example") (n + 1) where   [ofNat](https://lean-lang.org/doc/reference/4.26.0/Terms/Literals/#OfNat___mk "Documentation for OfNat.ofNat in Lean Language Reference") := ⟨n + 1, byn:[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")⊢ n [+](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAdd___mk "Documentation for HAdd.hAdd in Lean Language Reference") 1 > 0 [simp](https://lean-lang.org/doc/reference/4.26.0/Tactic-Proofs/Tactic-Reference/#simp "Documentation for tactic in Lean Language Reference")All goals completed! 🐙⟩`
Here, `[simp](https://lean-lang.org/doc/reference/4.26.0/Tactic-Proofs/Tactic-Reference/#simp "Documentation for tactic in Lean Language Reference")` is needed because `[decide](https://lean-lang.org/doc/reference/4.26.0/Tactic-Proofs/Tactic-Reference/#decide "Documentation for tactic in Lean Language Reference")` requires concrete values, but the proposition in question is `n + 1 > 0`.
Subtypes are a two-edged sword. They allow efficient representation of validation rules, but they transfer the burden of maintaining these rules to the users of the library, who have to _prove_ that they are not violating important invariants. Generally, it's a good idea to use them internally to a library, providing an API to users that automatically ensures that all invariants are satisfied, with any necessary proofs being internal to the library.
Checking whether a value of type `α` is in the subtype `{x : α // p x}` usually requires that the proposition `p x` be decidable. The [section on equality and ordering classes](Overloading-and-Type-Classes/Standard-Classes/#equality-and-ordering) describes how decidable propositions can be used with `if`. When `if` is used with a decidable proposition, a name can be provided. In the `then` branch, the name is bound to evidence that the proposition is true, and in the `else` branch, it is bound to evidence that the proposition is false. This comes in handy when checking whether a given `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` is positive:
`def Nat.asFastPos? In the `then` branch, `h` is bound to evidence that `n > 0`, and this evidence can be used as the second argument to `Subtype`'s constructor.
###  5.2.1.3. Validated Input[🔗](find/?domain=Verso.Genre.Manual.section&name=validated-input "Permalink")
The validated user input is a structure that expresses the business logic using multiple techniques:
  * The structure type itself encodes the year in which it was checked for validity, so that `CheckedInput 2019` is not the same type as `CheckedInput 2020`
  * The birth year is represented as a `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` rather than a `[String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")`
  * Subtypes are used to constrain the allowed values in the name and birth year fields

`structure CheckedInput (thisYear : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : Type where   name : {n : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") // n ≠ ""}   birthYear : {y : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") // y > 1900 ∧ y ≤ thisYear}`
An input validator should take the current year and a `RawInput` as arguments, returning either a checked input or at least one validation failure. This is represented by the `Validate` type:
`inductive Validate (ε α : Type) : Type where   | ok : α → Validate ε α   | errors : NonEmptyList ε → Validate ε α`
It looks very much like `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")`. The only difference is that the `errors` constructor may contain more than one failure.
`Validate` is a functor. Mapping a function over it transforms any successful value that might be present, just as in the `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` instance for `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")`:
`instance : [Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference") (Validate ε) where   [map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") f    | [.ok](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Validate___ok "Definition of example") x => [.ok](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Validate___ok "Definition of example") (f x)    | .errors errs => .errors errs`
The `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` instance for `Validate` has an important difference from the instance for `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")`: while the instance for `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")` terminates at the first error encountered, the instance for `Validate` is careful to accumulate all errors from _both_ the function and the argument branches:
`instance : [Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference") (Validate ε) where   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") := [.ok](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Validate___ok "Definition of example")   [seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference") f x :=     [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") f [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")     | [.ok](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Validate___ok "Definition of example") g => g <$> (x ())     | .errors errs =>       [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") x () [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")       | [.ok](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Validate___ok "Definition of example") _ => .errors errs       | .errors errs' => .errors (errs ++ errs')`
Using `.errors` together with the constructor for `NonEmptyList` is a bit verbose. Helpers like `reportError` make code more readable. In this application, error reports will consist of field names paired with messages:
`def Field := [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")``def reportError (f : [Field](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Field "Definition of example")) (msg : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) : Validate ([Field](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Field "Definition of example") × [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) α :=   .errors { [head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") := (f, msg), [tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") := [] }`
The `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` instance for `Validate` allows the checking procedures for each field to be written independently and then composed. Checking a name consists of ensuring that a string is non-empty, then returning evidence of this fact in the form of a `Subtype`. This uses the evidence-binding version of `if`:
`def checkName (name : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) :     Validate ([Field](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Field "Definition of example") × [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) {n : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") // n ≠ ""} :=   [if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference") h : name = "" [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference")     reportError "name" "Required"   [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference") [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") ⟨name, h⟩`
In the `then` branch, `h` is bound to evidence that `name = ""`, while it is bound to evidence that `¬name = ""` in the `else` branch.
It's certainly the case that some validation errors make other checks impossible. For example, it makes no sense to check whether the birth year field is greater than 1900 if a confused user wrote the word `"syzygy"` instead of a number. Checking the allowed range of the number is only meaningful after ensuring that the field in fact contains a number. This can be expressed using the function `andThen`:
`def Validate.andThen (val : Validate ε α)     (next : α → Validate ε β) : Validate ε β :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") val [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | .errors errs => .errors errs   | [.ok](Functors___-Applicative-Functors___-and-Monads/Applicative-Functors/#Validate___ok "Definition of example") x => next x`

They are described [in the section that describes the ``](Functors___-Applicative-Functors___-and-Monads/The-Applicative-Contract/#additional-stipulations)`[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` contract.

Form validation with `checkInput` illustrates a key advantage of `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` over `Monad`. Because `>>=` provides enough power to modify the rest of the program's execution based on the value from the first step, it _must_ receive a value from the first step to pass on. If no value is received (e.g. because an error has occurred), then `>>=` cannot execute the rest of the program. `Validate` demonstrates why it can be useful to run the rest of the program anyway: in cases where the earlier data isn't needed, running the rest of the program can yield useful information (in this case, more validation errors). `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")`'s `<*>` may run both of its arguments before recombining the results. Similarly, `>>=` forces sequential execution. Each step must complete before the next may run. This is generally useful, but it makes it impossible to have parallel execution of different threads that naturally emerges from the program's actual data dependencies. A more powerful abstraction like `Monad` increases the flexibility that's available to the API consumer, but it decreases the flexibility that is available to the API implementor.
