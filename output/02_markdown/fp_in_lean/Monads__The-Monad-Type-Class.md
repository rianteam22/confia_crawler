#  4.2. The Monad Type Class[🔗](find/?domain=Verso.Genre.Manual.section&name=monad-type-class "Permalink")
Rather than having to import an operator like `ok` or `andThen` for each type that is a monad, the Lean standard library contains a type class that allow them to be overloaded, so that the same operators can be used for _any_ monad. Monads have two operations, which are the equivalent of `ok` and `andThen`:
`class Monad (m : Type → Type) where   pure : α → m α   bind : m α → (α → m β) → m β`
This definition is slightly simplified. The actual definition in the Lean library is somewhat more involved, and will be presented later.
The `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instances for `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")` and `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference") ε` can be created by adapting the definitions of their respective `andThen` operations:

##  4.2.1. General Monad Operations[🔗](find/?domain=Verso.Genre.Manual.section&name=monad-class-polymorphism "Permalink")
Because many different types are monads, functions that are polymorphic over _any_ monad are very powerful. For example, the function `mapM` is a version of `[map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")` that uses a `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` to sequence and combine the results of applying a function:
`def mapM [[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") m] (f : α → m β) : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → m ([List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") β)   | [] => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") []   | x :: xs =>     f x >>= fun hd =>     mapM f xs >>= fun tl =>     [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (hd :: tl)`
The return type of the function argument `f` determines which `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance will be used. In other words, `mapM` can be used for functions that produce logs, for functions that can fail, or for functions that use mutable state. Because `f`'s type determines the available effects, they can be tightly controlled by API designers.
As described in [this chapter's introduction](Monads/One-API___-Many-Applications/#numbering-tree-nodes), `State σ α` represents programs that make use of a mutable variable of type `σ` and return a value of type `α`. These programs are actually functions from a starting state to a pair of a value and a final state. The `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` class requires that its parameter expect a single type argument—that is, it should be a `Type → Type`. This means that the instance for `State` should mention the state type `σ`, which becomes a parameter to the instance:
`instance : [Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") (State σ) where   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x := fun s => (s, x)   [bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") first next :=     fun s =>       let (s', x) := first s       next x s'`
This means that the type of the state cannot change between calls to `get` and `set` that are sequenced using `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")`, which is a reasonable rule for stateful computations. The operator `increment` increases a saved state by a given amount, returning the old value:
`def increment (howMuch : [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")) : State [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") :=   get >>= fun i =>   set (i + howMuch) >>= fun () =>   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") i`
Using `mapM` with `increment` results in a program that computes the sum of the entries in a list. More specifically, the mutable variable contains the sum so far, while the resulting list contains a running sum. In other words, `mapM increment` has type `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") → State [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") ([List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference"))`, and expanding the definition of `State` yields `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") → [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") → ([Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") × [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference"))`. ##  4.2.2. The Identity Monad[🔗](find/?domain=Verso.Genre.Manual.section&name=Id-monad "Permalink")
Monads encode programs with effects, such as failure, exceptions, or logging, into explicit representations as data and functions. Sometimes, however, an API will be written to use a monad for flexibility, but the API's client may not require any encoded effects. The _identity monad_ is a monad that has no effects. It allows pure code to be used with monadic APIs:
`def Id (t : Type) : Type := t  instance : [Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") Id where   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x := x   [bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") x f := f x`
The type of `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` should be `α → [Id](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Id "Documentation for Id in Lean Language Reference") α`, but `[Id](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Id "Documentation for Id in Lean Language Reference") α` reduces to just `α`. Similarly, the type of `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` should be `α → (α → [Id](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Id "Documentation for Id in Lean Language Reference") β) → [Id](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Id "Documentation for Id in Lean Language Reference") β`. Because this reduces to `α → (α → β) → β`, the second argument can be applied to the first to find the result.
With the identity monad, `mapM` becomes equivalent to `[map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")` To call it this way, however, Lean requires a hint that the intended monad is `[Id](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Id "Documentation for Id in Lean Language Reference")`:
`def numbers := mapM (m := [Id](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Id "Documentation for Id in Lean Language Reference")) ([do](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Syntax/#Lean___Parser___Term___do "Documentation for syntax in Lean Language Reference") return · + 1) [1, 2, 3, 4, 5]`

##  4.2.3. The Monad Contract[🔗](find/?domain=Verso.Genre.Manual.section&name=monad-contract "Permalink")
Just as every pair of instances of `[BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference")` and `[Hashable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable in Lean Language Reference")` should ensure that any two equal values have the same hash, there is a contract that each instance of `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` should obey. First, `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` should be a left identity of `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")`. That is, `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") ([pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") v) f` should be the same as `f v`. Secondly, `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` should be a right identity of `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")`, so `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") v [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` is the same as `v`. Finally, `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` should be associative, so `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") ([bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") v f) g` is the same as `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") v (fun x => [bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") (f x) g)`.
This contract specifies the expected properties of programs with effects more generally. Because `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` has no effects, sequencing its effects with `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` shouldn't change the result. The associative property of `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` basically says that the sequencing bookkeeping itself doesn't matter, so long as the order in which things are happening is preserved.
