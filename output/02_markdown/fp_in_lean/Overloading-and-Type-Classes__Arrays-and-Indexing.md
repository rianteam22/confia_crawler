#  3.4. Arrays and Indexing
The [Interlude](Interlude___-Propositions___-Proofs___-and-Indexing/#props-proofs-indexing) describes how to use indexing notation in order to look up entries in a list by their position. This syntax is also governed by a type class, and it can be used for a variety of different types.
##  3.4.1. Arrays[🔗](find/?domain=Verso.Genre.Manual.section&name=array-indexing "Permalink")
For instance, Lean arrays are much more efficient than linked lists for most purposes. In Lean, the type `[Array](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___mk "Documentation for Array in Lean Language Reference") α` is a dynamically-sized array holding values of type `α`, much like a Java `ArrayList`, a C++ `std::vector`, or a Rust `Vec`. Unlike `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference")`, which has a pointer indirection on each use of the `[cons](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference")` constructor, arrays occupy a contiguous region of memory, which is much better for processor caches. Also, looking up a value in an array takes constant time, while lookup in a linked list takes time proportional to the index being accessed.
In pure functional languages like Lean, it is not possible to mutate a given position in a data structure. Instead, a copy is made that has the desired modifications. However, copying is not always necessary: the Lean compiler and runtime contain an optimization that can allow modifications to be implemented as mutations behind the scenes when there is only a single unique reference to an array.
Arrays are written similarly to lists, but with a leading `#`:
`def northernTrees : [Array](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___mk "Documentation for Array in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") :=   #["sloe", "birch", "elm", "oak"]`
The number of values in an array can be found using `[Array.size](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___size "Documentation for Array.size in Lean Language Reference")`. For instance, `[northernTrees](Overloading-and-Type-Classes/Arrays-and-Indexing/#northernTrees "Definition of example").[size](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___size "Documentation for Array.size in Lean Language Reference")` evaluates to `4`. For indices that are smaller than an array's size, indexing notation can be used to find the corresponding value, just as with lists. That is, `[northernTrees](Overloading-and-Type-Classes/Arrays-and-Indexing/#northernTrees "Definition of example")[2]` evaluates to `"elm"`. Similarly, the compiler requires a proof that an index is in bounds, and attempting to look up a value outside the bounds of the array results in a compile-time error, just as with lists. For instance, `[northernTrees](Overloading-and-Type-Classes/Arrays-and-Indexing/#northernTrees "Definition of example")[8]` results in:

```
failed to prove index is valid, possible solutions:
  - Use `have`-expressions to prove the index is valid
  - Use `a[i]!` notation instead, runtime check is performed, and 'Panic' error message is produced if index is not valid
  - Use `a[i]?` notation instead, result is an `Option` type
  - Use `a[i]'h` notation instead, where `h` is a proof that index is valid
⊢ 8 [<](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LT___mk "Documentation for LT.lt in Lean Language Reference") [northernTrees](Overloading-and-Type-Classes/Arrays-and-Indexing/#northernTrees "Definition of example").[size](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___size "Documentation for Array.size in Lean Language Reference")
```

##  3.4.2. Non-Empty Lists[🔗](find/?domain=Verso.Genre.Manual.section&name=non-empty-list-indexing "Permalink")
A datatype that represents non-empty lists can be defined as a structure with a field for the head of the list and a field for the tail, which is an ordinary, potentially empty list:
`structure NonEmptyList (α : Type) : Type where   head : α   tail : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α`
For example, the non-empty list `idahoSpiders` (which contains some spider species native to the US state of Idaho) consists of `"Banded Garden Spider"` followed by four other spiders, for a total of five spiders:
`def idahoSpiders : NonEmptyList [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") := {   [head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") := "Banded Garden Spider",   [tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") := [     "Long-legged Sac Spider",     "Wolf Spider",     "Hobo Spider",     "Cat-faced Spider"   ] }`
Looking up the value at a specific index in this list with a recursive function should consider three possibilities:
  1. The index is `0`, in which case the head of the list should be returned.
  2. The index is `n + 1` and the tail is empty, in which case the index is out of bounds.
  3. The index is `n + 1` and the tail is non-empty, in which case the function can be called recursively on the tail and `n`.


For example, a lookup function that returns an `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")` can be written as follows:
`def NonEmptyList.get? : NonEmptyList α → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") α   | xs, 0 => [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") xs.[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example")   | {[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") := _, [tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") := []}, _ + 1 => [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")   | {[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") := _, [tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") := h :: t}, n + 1 => [get?](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___get___ "Definition of example") {[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") := h, [tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") := t} n`
Each case in the pattern match corresponds to one of the possibilities above. The recursive call to `[get?](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___get___ "Definition of example")` does not require a `NonEmptyList` namespace qualifier because the body of the definition is implicitly in the definition's namespace. Another way to write this function uses a list lookup `xs.[tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example")[n]?` when the index is greater than zero:
`def NonEmptyList.get? : NonEmptyList α → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") α   | xs, 0 => [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") xs.[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example")   | xs, n + 1 => xs.[tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example")[n]?`
If the list contains one entry, then only `0` is a valid index. If it contains two entries, then both `0` and `1` are valid indices. If it contains three entries, then `0`, `1`, and `2` are valid indices. In other words, the valid indices into a non-empty list are natural numbers that are strictly less than the length of the list, which are less than or equal to the length of the tail.
The definition of what it means for an index to be in bounds should be written as an `abbrev` because the tactics used to find evidence that indices are acceptable are able to solve inequalities of numbers, but they don't know anything about the name `NonEmptyList.inBounds`:
`abbrev NonEmptyList.inBounds (xs : NonEmptyList α) (i : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : Prop :=   i ≤ xs.[tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example").[length](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___length "Documentation for List.length in Lean Language Reference")`
This function returns a proposition that might be true or false. For instance, `2` is in bounds for `idahoSpiders`, while `5` is not:
`theorem atLeastThreeSpiders : idahoSpiders.inBounds 2 := by⊢ idahoSpiders.inBounds 2 [decide](https://lean-lang.org/doc/reference/4.26.0/Tactic-Proofs/Tactic-Reference/#decide "Documentation for tactic in Lean Language Reference")All goals completed! 🐙  theorem notSixSpiders : ¬idahoSpiders.inBounds 5 := by⊢ [¬](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Logical-Connectives/#Not "Documentation for Not in Lean Language Reference")idahoSpiders.inBounds 5 [decide](https://lean-lang.org/doc/reference/4.26.0/Tactic-Proofs/Tactic-Reference/#decide "Documentation for tactic in Lean Language Reference")All goals completed! 🐙`
The logical negation operator has a very low precedence, which means that `¬idahoSpiders.inBounds 5` is equivalent to `¬(idahoSpiders.inBounds 5)`.
This fact can be used to write a lookup function that requires evidence that the index is valid, and thus need not return `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")`, by delegating to the version for lists that checks the evidence at compile time:
`def NonEmptyList.get (xs : NonEmptyList α)     (i : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) (ok : xs.inBounds i) : α :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") i [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | 0 => xs.[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example")   | n + 1 => xs.[tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example")[n]`
It is, of course, possible to write this function to use the evidence directly, rather than delegating to a standard library function that happens to be able to use the same evidence. This requires techniques for working with proofs and propositions that are described later in this book.
##  3.4.3. Overloading Indexing[🔗](find/?domain=Verso.Genre.Manual.section&name=overloading-indexing "Permalink")
Indexing notation for a collection type can be overloaded by defining an instance of the `GetElem` type class. For the sake of flexibility, `GetElem` has four parameters:
  * The type of the collection
  * The type of the index
  * The type of elements that are extracted from the collection
  * A function that determines what counts as evidence that the index is in bounds


The element type and the evidence function are both output parameters. `GetElem` has a single method, `getElem`, which takes a collection value, an index value, and evidence that the index is in bounds as arguments, and returns an element:
`class GetElem     (coll : Type)     (idx : Type)     (item : [outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference") Type)     (inBounds : [outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference") (coll → idx → Prop)) where   getElem : (c : coll) → (i : idx) → inBounds c i → item`
In the case of `NonEmptyList α`, these parameters are:
  * The collection is `NonEmptyList α`
  * Indices have type `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`
  * The type of elements is `α`
  * An index is in bounds if it is less than or equal to the length of the tail


In fact, the `[GetElem](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#GetElem___mk "Documentation for GetElem in Lean Language Reference")` instance can delegate directly to `[NonEmptyList.get](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___get "Definition of example")`:
`instance : [GetElem](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#GetElem___mk "Documentation for GetElem in Lean Language Reference") (NonEmptyList α) [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") α NonEmptyList.inBounds where   [getElem](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#GetElem___mk "Documentation for GetElem.getElem in Lean Language Reference") := [NonEmptyList.get](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___get "Definition of example")`
With this instance, `NonEmptyList` becomes just as convenient to use as `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference")`. Evaluating `idahoSpiders.[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example")` yields `"Banded Garden Spider"`, while `idahoSpiders[9]` leads to the compile-time error:

```
failed to prove index is valid, possible solutions:
  - Use `have`-expressions to prove the index is valid
  - Use `a[i]!` notation instead, runtime check is performed, and 'Panic' error message is produced if index is not valid
  - Use `a[i]?` notation instead, result is an `Option` type
  - Use `a[i]'h` notation instead, where `h` is a proof that index is valid
⊢ idahoSpiders.inBounds 9
```

Because both the collection type and the index type are input parameters to the `[GetElem](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#GetElem___mk "Documentation for GetElem in Lean Language Reference")` type class, new types can be used to index into existing collections. The positive number type `Pos` is a perfectly reasonable index into a `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference")`, with the caveat that it cannot point at the first entry. The following instance of `[GetElem](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#GetElem___mk "Documentation for GetElem in Lean Language Reference")` allows `Pos` to be used just as conveniently as `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` to find a list entry:
`instance : [GetElem](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#GetElem___mk "Documentation for GetElem in Lean Language Reference") ([List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) Pos α     (fun list n => list.[length](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___length "Documentation for List.length in Lean Language Reference") > n.toNat) where   [getElem](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#GetElem___mk "Documentation for GetElem.getElem in Lean Language Reference") (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) (i : Pos) ok := xs[i.toNat]`
Indexing can also make sense for non-numeric indices. For example, `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")` can be used to select between the fields in a point, with `[false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")` corresponding to `x` and `[true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")` corresponding to `y`:
`instance : [GetElem](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#GetElem___mk "Documentation for GetElem in Lean Language Reference") (PPoint α) [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") α (fun _ _ => [True](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Truth/#True___intro "Documentation for True in Lean Language Reference")) where   [getElem](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#GetElem___mk "Documentation for GetElem.getElem in Lean Language Reference") (p : PPoint α) (i : [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")) _ :=     [if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") [not](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___not "Documentation for Bool.not in Lean Language Reference") i [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") p.x [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") p.y`
In this case, both Booleans are valid indices. Because every possible `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")` is in bounds, the evidence is simply the true proposition `[True](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Truth/#True___intro "Documentation for True in Lean Language Reference")`.
