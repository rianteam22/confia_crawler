#  3.5. Standard Classes[🔗](find/?domain=Verso.Genre.Manual.section&name=standard-classes "Permalink")
This section presents a variety of operators and functions that can be overloaded using type classes in Lean. Each operator or function corresponds to a method of a type class. Unlike C++, infix operators in Lean are defined as abbreviations for named functions; this means that overloading them for new types is not done using the operator itself, but rather using the underlying name (such as `[HAdd.hAdd](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAdd___mk "Documentation for HAdd.hAdd in Lean Language Reference")`).
##  3.5.1. Arithmetic[🔗](find/?domain=Verso.Genre.Manual.section&name=arithmetic-classes "Permalink")
Most arithmetic operators are available in a heterogeneous form, where the arguments may have different type and an output parameter decides the type of the resulting expression. For each heterogeneous operator, there is a corresponding homogeneous version that can found by removing the letter `h`, so that `[HAdd.hAdd](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAdd___mk "Documentation for HAdd.hAdd in Lean Language Reference")` becomes `[Add.add](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Add___mk "Documentation for Add.add in Lean Language Reference")`. The following arithmetic operators are overloaded:

##  3.5.2. Bitwise Operators[🔗](find/?domain=Verso.Genre.Manual.section&name=bitwise-classes "Permalink")
Lean contains a number of standard bitwise operators that are overloaded using type classes. There are instances for fixed-width types such as `[UInt8](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#UInt8___ofBitVec "Documentation for UInt8 in Lean Language Reference")`, `[UInt16](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#UInt16___ofBitVec "Documentation for UInt16 in Lean Language Reference")`, `[UInt32](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#UInt32___ofBitVec "Documentation for UInt32 in Lean Language Reference")`, `[UInt64](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#UInt64___ofBitVec "Documentation for UInt64 in Lean Language Reference")`, and `[USize](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#USize___ofBitVec "Documentation for USize in Lean Language Reference")`. The latter is the size of words on the current platform, typically 32 or 64 bits. The following bitwise operators are overloaded:

Because the names `[And](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Logical-Connectives/#And___intro "Documentation for And in Lean Language Reference")` and `[Or](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Logical-Connectives/#Or___inl "Documentation for Or in Lean Language Reference")` are already taken as the names of logical connectives, the homogeneous versions of `[HAnd](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAnd___mk "Documentation for HAnd in Lean Language Reference")` and `[HOr](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HOr___mk "Documentation for HOr in Lean Language Reference")` are called `AndOp` and `OrOp` rather than `[And](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Logical-Connectives/#And___intro "Documentation for And in Lean Language Reference")` and `[Or](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Logical-Connectives/#Or___inl "Documentation for Or in Lean Language Reference")`.
##  3.5.3. Equality and Ordering[🔗](find/?domain=Verso.Genre.Manual.section&name=equality-and-ordering "Permalink")
Testing equality of two values typically uses the `[BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference")` class, which is short for “Boolean equality”. Due to Lean's use as a theorem prover, there are really two kinds of equality operators in Lean:
  * _Boolean equality_ is the same kind of equality that is found in other programming languages. It is a function that takes two values and returns a `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")`. Boolean equality is written with two equals signs, just as in Python and C#. Because Lean is a pure functional language, there's no separate notions of reference vs value equality—pointers cannot be observed directly.
  * _Propositional equality_ is the mathematical statement that two things are equal. Propositional equality is not a function; rather, it is a mathematical statement that admits proof. It is written with a single equals sign. A statement of propositional equality is like a type that classifies evidence of this equality.


Both notions of equality are important, and used for different purposes. Boolean equality is useful in programs, when a decision needs to be made about whether two values are equal. For example, `"Octopus" ==  "Cuttlefish"` evaluates to `[false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")`, and `"Octopodes" ==  "Octo".[append](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___append "Documentation for String.append in Lean Language Reference") "podes"` evaluates to `[true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")`. Some values, such as functions, cannot be checked for equality. For example, `(fun (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) => 1 + x) == ([Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") ·)` yields the error:

```
failed to synthesize
  [BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference") ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference"))

Hint: Additional diagnostic information may be available using the `set_option diagnostics true` command.
```

As this message indicates, `==` is overloaded using a type class. The expression `x == y` is actually shorthand for `[BEq.beq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq.beq in Lean Language Reference") x y`.
Propositional equality is a mathematical statement rather than an invocation of a program. Because propositions are like types that describe evidence for some statement, propositional equality has more in common with types like `[String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")` and `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` than it does with Boolean equality. This means that it can't automatically be checked. However, the equality of any two expressions can be stated in Lean, so long as they have the same type. The statement `(fun (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) => 1 + x) = ([Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") ·)` is a perfectly reasonable statement. From the perspective of mathematics, two functions are equal if they map equal inputs to equal outputs, so this statement is even true, though it requires a one-line proof to convince Lean of this fact.
Generally speaking, when using Lean as a programming language, it's easiest to stick to Boolean functions rather than propositions. However, as the names `[true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")` and `[false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")` for `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")`'s constructors suggest, this difference is sometimes blurred. Some propositions are _decidable_ , which means that they can be checked just like a Boolean function. The function that checks whether the proposition is true or false is called a _decision procedure_ , and it returns _evidence_ of the truth or falsity of the proposition. Some examples of decidable propositions include equality and inequality of natural numbers, equality of strings, and “ands” and “ors” of propositions that are themselves decidable.
In Lean, `if` works with decidable propositions. For example, `2 < 4` is a proposition:
``2 [<](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LT___mk "Documentation for LT.lt in Lean Language Reference") 4 : Prop`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") 2 < 4`
```
2 [<](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#LT___mk "Documentation for LT.lt in Lean Language Reference") 4 : Prop
```

Nonetheless, it is perfectly acceptable to write it as the condition in an `if`. For example, `[if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") 2 < 4 [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") 1 [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") 2` has type `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` and evaluates to `1`.
Not all propositions are decidable. If they were, then computers would be able to prove any true proposition just by running the decision procedure, and mathematicians would be out of a job. More specifically, decidable propositions have an instance of the `[Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference")` type class, which contains the decision procedure. Trying to use a proposition that isn't decidable as if it were a `[Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")` results in a failure to find the `[Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference")` instance. For example, `[if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") (fun (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) => 1 + x) = ([Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") ·) [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") "yes" [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") "no"` results in:

```
failed to synthesize
  [Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference") [(](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Propositional-Equality/#Eq___refl "Documentation for Eq in Lean Language Reference")(fun x => 1 [+](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAdd___mk "Documentation for HAdd.hAdd in Lean Language Reference") x) [=](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Propositional-Equality/#Eq___refl "Documentation for Eq in Lean Language Reference") fun x => x.[succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference")[)](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Propositional-Equality/#Eq___refl "Documentation for Eq in Lean Language Reference")

Hint: Additional diagnostic information may be available using the `set_option diagnostics true` command.
```

Comparing values using `<`, `==`, and `>` can be inefficient. Checking first whether one value is less than another, and then whether they are equal, can require two traversals over large data structures. To solve this problem, Java and C# have standard `compareTo` and `CompareTo` methods (respectively) that can be overridden by a class in order to implement all three operations at the same time. These methods return a negative integer if the receiver is less than the argument, zero if they are equal, and a positive integer if the receiver is greater than the argument. Rather than overloading the meaning of integers, Lean has a built-in inductive type that describes these three possibilities:
`inductive Ordering where   | lt   | eq   | gt`
The `[Ord](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ord___mk "Documentation for Ord in Lean Language Reference")` type class can be overloaded to produce these comparisons. For `Pos`, an implementation can be:
`def Pos.comp : Pos → Pos → [Ordering](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ordering___lt "Documentation for Ordering in Lean Language Reference")   | Pos.one, Pos.one => [Ordering.eq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ordering___lt "Documentation for Ordering.eq in Lean Language Reference")   | Pos.one, [Pos.succ](Overloading-and-Type-Classes/Positive-Numbers/#Pos___succ "Definition of example") _ => [Ordering.lt](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ordering___lt "Documentation for Ordering.lt in Lean Language Reference")   | [Pos.succ](Overloading-and-Type-Classes/Positive-Numbers/#Pos___succ "Definition of example") _, Pos.one => [Ordering.gt](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ordering___lt "Documentation for Ordering.gt in Lean Language Reference")   | [Pos.succ](Overloading-and-Type-Classes/Positive-Numbers/#Pos___succ "Definition of example") n, [Pos.succ](Overloading-and-Type-Classes/Positive-Numbers/#Pos___succ "Definition of example") k => [comp](Overloading-and-Type-Classes/Standard-Classes/#Pos___comp "Definition of example") n k  instance : [Ord](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ord___mk "Documentation for Ord in Lean Language Reference") Pos where   [compare](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ord___mk "Documentation for Ord.compare in Lean Language Reference") := [Pos.comp](Overloading-and-Type-Classes/Standard-Classes/#Pos___comp "Definition of example")`
In situations where `compareTo` would be the right approach in Java, use `[Ord.compare](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ord___mk "Documentation for Ord.compare in Lean Language Reference")` in Lean.
##  3.5.4. Hashing[🔗](find/?domain=Verso.Genre.Manual.section&name=hashing "Permalink")
Java and C# have `hashCode` and `GetHashCode` methods, respectively, that compute a hash of a value for use in data structures such as hash tables. The Lean equivalent is a type class called `Hashable`:
`class Hashable (α : Type) where   hash : α → [UInt64](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#UInt64___ofBitVec "Documentation for UInt64 in Lean Language Reference")`
If two values are considered equal according to a `[BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference")` instance for their type, then they should have the same hashes. In other words, if `x == y` then `[hash](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable.hash in Lean Language Reference") x == [hash](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable.hash in Lean Language Reference") y`. If `x ≠ y`, then `[hash](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable.hash in Lean Language Reference") x` won't necessarily differ from `[hash](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable.hash in Lean Language Reference") y` (after all, there are infinitely more `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` values than there are `[UInt64](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#UInt64___ofBitVec "Documentation for UInt64 in Lean Language Reference")` values), but data structures built on hashing will have better performance if unequal values are likely to have unequal hashes. This is the same expectation as in Java and C#.
The standard library contains a function `[mixHash](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#mixHash "Documentation for mixHash in Lean Language Reference")` with type `[UInt64](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#UInt64___ofBitVec "Documentation for UInt64 in Lean Language Reference") → [UInt64](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#UInt64___ofBitVec "Documentation for UInt64 in Lean Language Reference") → [UInt64](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#UInt64___ofBitVec "Documentation for UInt64 in Lean Language Reference")` that can be used to combine hashes for different fields for a constructor. A reasonable hash function for an inductive datatype can be written by assigning a unique number to each constructor, and then mixing that number with the hashes of each field. For example, a `[Hashable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable in Lean Language Reference")` instance for `Pos` can be written:
`def hashPos : Pos → [UInt64](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Fixed-Precision-Integers/#UInt64___ofBitVec "Documentation for UInt64 in Lean Language Reference")   | Pos.one => 0   | [Pos.succ](Overloading-and-Type-Classes/Positive-Numbers/#Pos___succ "Definition of example") n => [mixHash](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#mixHash "Documentation for mixHash in Lean Language Reference") 1 ([hashPos](Overloading-and-Type-Classes/Standard-Classes/#hashPos "Definition of example") n)  instance : [Hashable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable in Lean Language Reference") Pos where   [hash](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable.hash in Lean Language Reference") := [hashPos](Overloading-and-Type-Classes/Standard-Classes/#hashPos "Definition of example")`

##  3.5.5. Deriving Standard Classes[🔗](find/?domain=Verso.Genre.Manual.section&name=deriving-standard-classes "Permalink")
Instance of classes like `[BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference")` and `Hashable` are often quite tedious to implement by hand. Lean includes a feature called _instance deriving_ that allows the compiler to automatically construct well-behaved instances of many type classes. In fact, the `deriving [Repr](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Repr___mk "Documentation for Repr in Lean Language Reference")` phrase in the definition of `Firewood` in the [first section on polymorphism](Getting-to-Know-Lean/Polymorphism/#polymorphism) is an example of instance deriving.
Instances can be derived in two ways. The first can be used when defining a structure or inductive type. In this case, add `deriving` to the end of the type declaration followed by the names of the classes for which instances should be derived. For a type that is already defined, a standalone `deriving` command can be used. Write `deriving instance`` C1, C2, ... ``for`` T` to derive instances of `C1, C2, ...` for the type `T` after the fact.
`[BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference")` and `Hashable` instances can be derived for `Pos` and `NonEmptyList` using a very small amount of code:
`[deriving](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") [instance](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") [BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference"), [Hashable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable in Lean Language Reference") [for](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") Pos [deriving](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") [instance](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") [BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference"), [Hashable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable in Lean Language Reference") [for](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") NonEmptyList`
Instances can be derived for at least the following classes:
  * `[Inhabited](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Inhabited___mk "Documentation for Inhabited in Lean Language Reference")`
  * `[BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference")`
  * `[Repr](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Repr___mk "Documentation for Repr in Lean Language Reference")`
  * `Hashable`
  * `[Ord](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ord___mk "Documentation for Ord in Lean Language Reference")`


In some cases, however, the derived `[Ord](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ord___mk "Documentation for Ord in Lean Language Reference")` instance may not produce precisely the ordering desired in an application. When this is the case, it's fine to write an `[Ord](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Ord___mk "Documentation for Ord in Lean Language Reference")` instance by hand. The collection of classes for which instances can be derived can be extended by advanced users of Lean.
Aside from the clear advantages in programmer productivity and code readability, deriving instances also makes code easier to maintain, because the instances are updated as the definitions of types evolve. When reviewing changes to code, modifications that involve updates to datatypes are much easier to read without line after line of formulaic modifications to equality tests and hash computation.
##  3.5.6. Appending[🔗](find/?domain=Verso.Genre.Manual.section&name=append-class "Permalink")
Many datatypes have some sort of append operator. In Lean, appending two values is overloaded with the type class `HAppend`, which is a heterogeneous operation like that used for arithmetic operations:
`class HAppend (α : Type) (β : Type) (γ : [outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference") Type) where   hAppend : α → β → γ`
The syntax `xs ++ ys` desugars to `[HAppend.hAppend](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAppend___mk "Documentation for HAppend.hAppend in Lean Language Reference") xs ys`. For homogeneous cases, it's enough to implement an instance of `[Append](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Append___mk "Documentation for Append in Lean Language Reference")`, which follows the usual pattern:
`instance : [Append](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Append___mk "Documentation for Append in Lean Language Reference") (NonEmptyList α) where   [append](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Append___mk "Documentation for Append.append in Lean Language Reference") xs ys :=     { [head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") := xs.[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example"), [tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") := xs.[tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") ++ ys.[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") :: ys.[tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") }`

##  3.5.7. Functors[🔗](find/?domain=Verso.Genre.Manual.section&name=Functor "Permalink")
A polymorphic type is a _functor_ if it has an overload for a function named `map` that transforms every element contained in it by a function. While most languages use this terminology, C#'s equivalent of `map` is called `System.Linq.Enumerable.Select`. For example, mapping a function over a list constructs a new list in which each entry from the starting list has been replaced by the result of the function on that entry. Mapping a function `f` over an `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")` leaves `[none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")` untouched, and replaces `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x` with `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (f x)`.
Here are some examples of functors and how their `Functor` instances overload `map`:
  * `[Functor.map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") (· + 5) [1, 2, 3]` evaluates to `[6, 7, 8]`
  * `[Functor.map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") toString ([some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") ([List.cons](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference") 5 [List.nil](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.nil in Lean Language Reference")))` evaluates to `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") "[5]"`
  * `[Functor.map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") [List.reverse](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___reverse "Documentation for List.reverse in Lean Language Reference") [[1, 2, 3], [4, 5, 6]]` evaluates to `[[3, 2, 1], [6, 5, 4]]`


Because `[Functor.map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")` is a bit of a long name for this common operation, Lean also provides an infix operator for mapping a function, namely `<$>`. The prior examples can be rewritten as follows:
  * `(· + 5) <$> [1, 2, 3]` evaluates to `[6, 7, 8]`
  * `toString <$> ([some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") ([List.cons](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference") 5 [List.nil](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.nil in Lean Language Reference")))` evaluates to `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") "[5]"`
  * `[List.reverse](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___reverse "Documentation for List.reverse in Lean Language Reference") <$> [[1, 2, 3], [4, 5, 6]]` evaluates to `[[3, 2, 1], [6, 5, 4]]`


An instance of `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` for `NonEmptyList` requires specifying the `[map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")` function.
`instance : [Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference") NonEmptyList where   [map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") f xs := { [head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") := f xs.[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example"), [tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") := f <$> xs.[tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example") }`
Here, `[map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")` uses the `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` instance for `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference")` to map the function over the tail. This instance is defined for `NonEmptyList` rather than for `NonEmptyList α` because the argument type `α` plays no role in resolving the type class. A `NonEmptyList` can have a function mapped over it _no matter what the type of entries is_. If `α` were a parameter to the class, then it would be possible to make versions of `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` that only worked for `NonEmptyList [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`, but part of being a functor is that `[map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")` works for any entry type.
Here is an instance of `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` for `PPoint`:
`instance : [Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference") PPoint where   [map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") f p := { x := f p.x, y := f p.y }`
In this case, `f` has been applied to both `x` and `y`.
Even when the type contained in a functor is itself a functor, mapping a function only goes down one layer. That is, when using `[map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")` on a `NonEmptyList (PPoint [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference"))`, the function being mapped should take `PPoint [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` as its argument rather than `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`.
The definition of the `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` class uses one more language feature that has not yet been discussed: default method definitions. Normally, a class will specify some minimal set of overloadable operations that make sense together, and then use polymorphic functions with instance implicit arguments that build on the overloaded operations to provide a larger library of features. For example, the function `concat` can concatenate any non-empty list whose entries are appendable:
`def concat [[Append](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Append___mk "Documentation for Append in Lean Language Reference") α] (xs : NonEmptyList α) : α :=   let rec catList (start : α) : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → α     | [] => start     | (z :: zs) => [catList](Overloading-and-Type-Classes/Standard-Classes/#concat___catList "Definition of example") (start ++ z) zs   [catList](Overloading-and-Type-Classes/Standard-Classes/#concat___catList "Definition of example") xs.[head](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___head "Definition of example") xs.[tail](Overloading-and-Type-Classes/Arrays-and-Indexing/#NonEmptyList___tail "Definition of example")`
However, for some classes, there are operations that can be more efficiently implemented with knowledge of the internals of a datatype.
In these cases, a default method definition can be provided. A default method definition provides a default implementation of a method in terms of the other methods. However, instance implementors may choose to override this default with something more efficient. Default method definitions contain `:=` in a `class` definition.
In the case of `Functor`, some types have a more efficient way of implementing `map` when the function being mapped ignores its argument. Functions that ignore their arguments are called _constant functions_ because they always return the same value. Here is the definition of `Functor`, in which `mapConst` has a default implementation:
`class Functor (f : Type → Type) where   map : {α β : Type} → (α → β) → f α → f β    mapConst {α β : Type} (x : α) (coll : f β) : f α :=     map (fun _ => x) coll`
Just as a `[Hashable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable in Lean Language Reference")` instance that doesn't respect `[BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference")` is buggy, a `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` instance that moves around the data as it maps the function is also buggy. For example, a buggy `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` instance for `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference")` might throw away its argument and always return the empty list, or it might reverse the list. A bad `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` instance for `PPoint` might place `f x` in both the `x` and the `y` fields, or swap them. Specifically, `Functor` instances should follow two rules:
  1. Mapping the identity function should result in the original argument.
  2. Mapping two composed functions should have the same effect as composing their mapping.


More formally, the first rule says that `id <$> x` equals `x`. The second rule says that `[map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") (fun y => f (g y)) x` equals `[map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") f ([map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") g x)`. The composition `f ∘ g` can also be written `fun y => f (g y)`. These rules prevent implementations of `map` that move the data around or delete some of it.
##  3.5.8. Messages You May Meet[🔗](find/?domain=Verso.Genre.Manual.section&name=standard-classes-messages "Permalink")
Lean is not able to derive instances for all classes. For example, the code
`[deriving](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") [instance](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") `No deriving handlers have been implemented for class `ToString``ToString [for](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") NonEmptyList`
results in the following error:

```
No deriving handlers have been implemented for class `ToString`
```

Invoking `[deriving](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference") [instance](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Deriving-Instances/#Lean___Parser___Command___deriving-next "Documentation for syntax in Lean Language Reference")` causes Lean to consult an internal table of code generators for type class instances. If the code generator is found, then it is invoked on the provided type to create the instance. This message, however, means that no code generator was found for `ToString`.
