#  5.3. The Applicative Contract[🔗](find/?domain=Verso.Genre.Manual.section&name=applicative-laws "Permalink")
Just like `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")`, `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")`, and types that implement `[BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference")` and `[Hashable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Hashable___mk "Documentation for Hashable in Lean Language Reference")`, `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` has a set of rules that all instances should adhere to.
There are four rules that an applicative functor should follow:
  1. It should respect identity, so `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") id <*> v = v`
  2. It should respect function composition, so `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (· ∘ ·) <*> u <*> v <*> w = u <*> (v <*> w)`
  3. Sequencing pure operations should be a no-op, so `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") f <*> [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x``=``[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (f x)`
  4. The ordering of pure operations doesn't matter, so `u <*> [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x = [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (fun f => f x) <*> u`


To check these for the `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference") [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")` instance, start by expanding `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` into `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference")`.
The first rule states that `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") id <*> v = v`. The definition of `seq` for `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")` states that this is the same as `id <$> v = v`, which is one of the `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` rules that have already been checked.
The second rule states that `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (· ∘ ·) <*> u <*> v <*> w = u <*> (v <*> w)`. If any of `u`, `v`, or `w` is `[none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`, then both sides are `[none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`, so the property holds. Assuming that `u` is `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") f`, that `v` is `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") g`, and that `w` is `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x`, then this is equivalent to saying that `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (· ∘ ·) <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") f <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") g <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x = [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") f <*> ([some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") g <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x)`. Evaluating the two sides yields the same result:
`[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (· ∘ ·) <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") f <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") g <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x``[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (f ∘ ·) <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") g <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x``[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (f ∘ g) <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x``[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") ((f ∘ g) x)``[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (f (g x))`
`[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") f <*> ([some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") g <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x)``[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") f <*> ([some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (g x))``[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (f (g x))`
The third rule follows directly from the definition of `seq`:
`[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") f <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x``f <$> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x``[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (f x)`
In the fourth case, assume that `u` is `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") f`, because if it's `[none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`, both sides of the equation are `[none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`. `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") f <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x` evaluates directly to `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (f x)`, as does `[some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") (fun g => g x) <*> [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") f`.
##  5.3.1. All Applicatives are Functors[🔗](find/?domain=Verso.Genre.Manual.section&name=applicatives-are-functors "Permalink")
The two operators for `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` are enough to define `map`:
`def map [[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference") f] (g : α → β) (x : f α) : f β :=   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") g <*> x`
This can only be used to implement `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` if the contract for `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` guarantees the contract for `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")`, however. The first rule of `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` is that `id <$> x = x`, which follows directly from the first rule for `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")`. The second rule of `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")` is that `map (f ∘ g) x = map f (map g x)`. Unfolding the definition of `map` here results in `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (f ∘ g) <*> x = [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") f <*> ([pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") g <*> x)`. Using the rule that sequencing pure operations is a no-op, the left side can be rewritten to `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (· ∘ ·) <*> [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") f <*> [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") g <*> x`. This is an instance of the rule that states that applicative functors respect function composition.
This justifies a definition of `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` that extends `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")`, with a default definition of `[map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference")` given in terms of `pure` and `seq`:
`class Applicative (f : Type → Type) extends [Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference") f where   pure : α → f α   seq : f (α → β) → ([Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") → f α) → f β   [map](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor.map in Lean Language Reference") g x := seq (pure g) (fun () => x)`
##  5.3.2. All Monads are Applicative Functors[🔗](find/?domain=Verso.Genre.Manual.section&name=monads-are-applicative "Permalink")
An instance of `Monad` already requires an implementation of `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")`. Together with `bind`, this is enough to define `seq`:
`def seq [[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") m] (f : m (α → β)) (x : [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") → m α) : m β := [do](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Syntax/#Lean___Parser___Term___do "Documentation for syntax in Lean Language Reference")   let g ← f   let y ← x ()   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (g y)`
Once again, checking that the `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` contract implies the `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` contract will allow this to be used as a default definition for `[seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference")` if `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` extends `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")`.
The rest of this section consists of an argument that this implementation of `[seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference")` based on `bind` in fact satisfies the `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` contract. One of the beautiful things about functional programming is that this kind of argument can be worked out on a piece of paper with a pencil, using the kinds of evaluation rules from [the initial section on evaluating expressions](Getting-to-Know-Lean/Evaluating-Expressions/#evaluating). Thinking about the meanings of the operations while reading these arguments can sometimes help with understanding.
Replacing `do`-notation with explicit uses of `>>=` makes it easier to apply the `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` rules:
`def seq [[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") m] (f : m (α → β)) (x : [Unit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#Unit "Documentation for Unit in Lean Language Reference") → m α) : m β := [do](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Syntax/#Lean___Parser___Term___do "Documentation for syntax in Lean Language Reference")   f >>= fun g =>   x () >>= fun y =>   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (g y)`
To check that this definition respects identity, check that `seq ([pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") id) (fun () => v) = v`. The left hand side is equivalent to `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") id >>= fun g => (fun () => v) () >>= fun y => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (g y)`. The unit function in the middle can be eliminated immediately, yielding `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") id >>= fun g => v >>= fun y => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (g y)`. Using the fact that `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` is a left identity of `>>=`, this is the same as `v >>= fun y => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (id y)`, which is `v >>= fun y => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") y`. Because `fun x => f x` is the same as `f`, this is the same as `v >>= [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")`, and the fact that `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` is a right identity of `>>=` can be used to get `v`.
This kind of informal reasoning can be made easier to read with a bit of reformatting. In the following chart, read “`EXPR1 ={ REASON }= EXPR2`” as “`EXPR1` is the same as `EXPR2` because `REASON`”:
`[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") id >>= fun g => v >>= fun y => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (g y)`
`[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` is a left identity of `>>=`
`v >>= fun y => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (id y)`
Reduce the call to `id`
`v >>= fun y => [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") y`
`fun x => f x` is the same as `f`
`v >>= [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")`
`[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference")` is a right identity of `>>=`
`v`
To check that it respects function composition, check that `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (· ∘ ·) <*> u <*> v <*> w = u <*> (v <*> w)`. The first step is to replace `<*>` with this definition of `seq`. After that, a (somewhat long) series of steps that use the identity and associativity rules from the `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` contract is enough to get from one to the other:
`seq (seq (seq ([pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") (· ∘ ·)) (fun _ => u))       (fun _ => v))   (fun _ => w)`

##  5.3.3. Additional Stipulations[🔗](find/?domain=Verso.Genre.Manual.section&name=additional-stipulations "Permalink")
In addition to adhering to the individual contracts associated with each type class, combined implementations `[Functor](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Functor___mk "Documentation for Functor in Lean Language Reference")`, `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` and `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` should work equivalently to these default implementations. In other words, a type that provides both `[Applicative](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Applicative___mk "Documentation for Applicative in Lean Language Reference")` and `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instances should not have an implementation of `[seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference")` that works differently from the version that the `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance generates as a default implementation. This is important because polymorphic functions may be refactored to replace a use of `>>=` with an equivalent use of `<*>`, or a use of `<*>` with an equivalent use of `>>=`. This refactoring should not change the meaning of programs that use this code.
This rule explains why `Validate.andThen` should not be used to implement `bind` in a `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance. On its own, it obeys the monad contract. However, when it is used to implement `[seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference")`, the behavior is not equivalent to `[seq](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Seq___mk "Documentation for Seq.seq in Lean Language Reference")` itself. To see where they differ, take the example of two computations, both of which return errors.
