#  4.6. Additional Conveniences[🔗](find/?domain=Verso.Genre.Manual.section&name=monads-conveniences "Permalink")
##  4.6.1. Shared Argument Types[🔗](find/?domain=Verso.Genre.Manual.section&name=shared-argument-types "Permalink")
When defining a function that takes multiple arguments that have the same type, both can be written before the same colon. For example,
`def equal? [[BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference") α] (x : α) (y : α) : [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") α :=   [if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") x == y [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference")     [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x   [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference")     [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`
can be written
`def equal? [[BEq](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#BEq___mk "Documentation for BEq in Lean Language Reference") α] (x y : α) : [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") α :=   [if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") x == y [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference")     [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x   [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference")     [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`
This is especially useful when the type signature is large.
##  4.6.2. Leading Dot Notation[🔗](find/?domain=Verso.Genre.Manual.section&name=leading-dot-notation "Permalink")
The constructors of an inductive type are in a namespace. This allows multiple related inductive types to use the same constructor names, but it can lead to programs becoming verbose. In contexts where the inductive type in question is known, the namespace can be omitted by preceding the constructor's name with a dot, and Lean uses the expected type to resolve the constructor names. For example, a function that mirrors a binary tree can be written:
`def BinTree.mirror : [BinTree](Overloading-and-Type-Classes/Standard-Classes/#BinTree "Definition of example") α → [BinTree](Overloading-and-Type-Classes/Standard-Classes/#BinTree "Definition of example") α   | [BinTree.leaf](Overloading-and-Type-Classes/Standard-Classes/#BinTree___leaf "Definition of example") => [BinTree.leaf](Overloading-and-Type-Classes/Standard-Classes/#BinTree___leaf "Definition of example")   | [BinTree.branch](Overloading-and-Type-Classes/Standard-Classes/#BinTree___branch "Definition of example") l x r => [BinTree.branch](Overloading-and-Type-Classes/Standard-Classes/#BinTree___branch "Definition of example") ([mirror](Monads/Additional-Conveniences/#Old___BinTree___mirror "Definition of example") r) x ([mirror](Monads/Additional-Conveniences/#Old___BinTree___mirror "Definition of example") l)`
Omitting the namespaces makes it significantly shorter, at the cost of making the program harder to read in contexts like code review tools that don't include the Lean compiler:
`def BinTree.mirror : [BinTree](Overloading-and-Type-Classes/Standard-Classes/#BinTree "Definition of example") α → [BinTree](Overloading-and-Type-Classes/Standard-Classes/#BinTree "Definition of example") α   | [.leaf](Overloading-and-Type-Classes/Standard-Classes/#BinTree___leaf "Definition of example") => .leaf   | [.branch](Overloading-and-Type-Classes/Standard-Classes/#BinTree___branch "Definition of example") l x r => [.branch](Overloading-and-Type-Classes/Standard-Classes/#BinTree___branch "Definition of example") (mirror r) x (mirror l)`
Using the expected type of an expression to disambiguate a namespace is also applicable to names other than constructors. If `BinTree.empty` is defined as an alternative way of creating `[BinTree](Overloading-and-Type-Classes/Standard-Classes/#BinTree "Definition of example")`s, then it can also be used with dot notation:
`def BinTree.empty : [BinTree](Overloading-and-Type-Classes/Standard-Classes/#BinTree "Definition of example") α := [.leaf](Overloading-and-Type-Classes/Standard-Classes/#BinTree___leaf "Definition of example")```BinTree.empty : [BinTree](Overloading-and-Type-Classes/Standard-Classes/#BinTree "Definition of example") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") (.empty : [BinTree](Overloading-and-Type-Classes/Standard-Classes/#BinTree "Definition of example") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference"))`
```
BinTree.empty : [BinTree](Overloading-and-Type-Classes/Standard-Classes/#BinTree "Definition of example") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")
```

##  4.6.3. Or-Patterns[🔗](find/?domain=Verso.Genre.Manual.section&name=or-patterns "Permalink")
In contexts that allow multiple patterns, such as `match`-expressions, multiple patterns may share their result expressions. The datatype `Weekday` that represents days of the week:
`inductive Weekday where   | monday   | tuesday   | wednesday   | thursday   | friday   | saturday   | sunday deriving [Repr](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Repr___mk "Documentation for Repr in Lean Language Reference")`
Pattern matching can be used to check whether a day is a weekend:
`def Weekday.isWeekend (day : Weekday) : [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") day [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [Weekday.saturday](Monads/Additional-Conveniences/#Weekday___saturday "Definition of example") => [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")   | [Weekday.sunday](Monads/Additional-Conveniences/#Weekday___sunday "Definition of example") => [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")   | _ => [false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")`
This can already be simplified by using constructor dot notation:
`def Weekday.isWeekend (day : Weekday) : [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") day [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [.saturday](Monads/Additional-Conveniences/#Weekday___saturday "Definition of example") => [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")   | [.sunday](Monads/Additional-Conveniences/#Weekday___sunday "Definition of example") => [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")   | _ => [false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")`
Because both weekend patterns have the same result expression (`[true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")`), they can be condensed into one:
`def Weekday.isWeekend (day : Weekday) : [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") day [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [.saturday](Monads/Additional-Conveniences/#Weekday___saturday "Definition of example") | [.sunday](Monads/Additional-Conveniences/#Weekday___sunday "Definition of example") => [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")   | _ => [false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")`
This can be further simplified into a version in which the argument is not named:
`def Weekday.isWeekend : Weekday → [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")   | [.saturday](Monads/Additional-Conveniences/#Weekday___saturday "Definition of example") | [.sunday](Monads/Additional-Conveniences/#Weekday___sunday "Definition of example") => [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")   | _ => [false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")`
Behind the scenes, the result expression is simply duplicated across each pattern. This means that patterns can bind variables, as in this example that removes the `[inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference")` and `[inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference")` constructors from a sum type in which both contain the same type of value:
`def condense : α ⊕ α → α   | [.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") x | [.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference") x => x`
Because the result expression is duplicated, the variables bound by the patterns are not required to have the same types. Overloaded functions that work for multiple types may be used to write a single result expression that works for patterns that bind variables of different types:
`def stringy : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") ⊕ Weekday → [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")   | [.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") x | [.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference") x => s!"It is {[repr](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#repr-next "Documentation for repr in Lean Language Reference") x}"`
In practice, only variables shared in all patterns can be referred to in the result expression, because the result must make sense for each pattern. In `getTheNat`, only `n` can be accessed, and attempts to use either `x` or `y` lead to errors.
`def getTheNat : ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") × α) ⊕ ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") × β) → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")   | [.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") (n, x) | [.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference") (n, y) => n`
Attempting to access `x` in a similar definition causes an error because there is no `x` available in the second pattern:
`def getTheAlpha : ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") × α) ⊕ ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") × α) → α   | [.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") (n, x) | [.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference") (n, y) => `Unknown identifier `x``x`
```
Unknown identifier `x`
```

The fact that the result expression is essentially copy-pasted to each branch of the pattern match can lead to some surprising behavior. For example, the following definitions are acceptable because the `[inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference")` version of the result expression refers to the global definition of `str`:
`def str := "Some string"  def getTheString : ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") × [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) ⊕ ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") × β) → [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")   | [.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") (n, str) | [.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference") (n, y) => str`
Calling this function on both constructors reveals the confusing behavior. In the first case, a type annotation is needed to tell Lean which type `β` should be:
``"twenty"`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") [getTheString](Monads/Additional-Conveniences/#getTheString "Definition of example") ([.inl](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inl in Lean Language Reference") (20, "twenty") : ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") × [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")) ⊕ ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") × [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")))`
```
"twenty"
```

In the second case, the global definition is used:
``"Some string"`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") [getTheString](Monads/Additional-Conveniences/#getTheString "Definition of example") ([.inr](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Sum-Types/#Sum___inl "Documentation for Sum.inr in Lean Language Reference") (20, "twenty"))`
```
"Some string"
```

Using or-patterns can vastly simplify some definitions and increase their clarity, as in `Weekday.isWeekend`. Because there is a potential for confusing behavior, it's a good idea to be careful when using them, especially when variables of multiple types or disjoint sets of variables are involved.
