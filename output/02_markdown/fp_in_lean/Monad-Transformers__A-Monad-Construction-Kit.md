#  6.2. A Monad Construction Kit
`[ReaderT](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#ReaderT "Documentation for ReaderT in Lean Language Reference")` is far from the only useful monad transformer. This section describes a number of additional transformers. Each monad transformer consists of the following:
  1. A definition or datatype `T` that takes a monad as an argument. It should have a type like `(Type u ŌåÆ Type v) ŌåÆ Type u ŌåÆ Type v`, though it may accept additional arguments prior to the monad.
  2. A `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance for `T m` that relies on an instance of `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference") m`. This enables the transformed monad to be used as a monad.
  3. A `[MonadLift](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Lifting-Monads/#MonadLift___mk "Documentation for MonadLift in Lean Language Reference")` instance that translates actions of type `m ╬▒` into actions of type `T m ╬▒`, for arbitrary monads `m`. This enables actions from the underlying monad to be used in the transformed monad.


Furthermore, the `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance for the transformer should obey the contract for `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")`, at least if the underlying `[Monad](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Monad___mk "Documentation for Monad in Lean Language Reference")` instance does. In addition, `[monadLift](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Lifting-Monads/#MonadLiftT___mk "Documentation for MonadLiftT.monadLift in Lean Language Reference") ([pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x : m ╬▒)` should be equivalent to `[pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") x` in the transformed monad, and `[monadLift](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Lifting-Monads/#MonadLiftT___mk "Documentation for MonadLiftT.monadLift in Lean Language Reference")` should distribute over `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference")` so that `[monadLift](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Lifting-Monads/#MonadLiftT___mk "Documentation for MonadLiftT.monadLift in Lean Language Reference") (x >>= f : m ╬▒)` is the same as `([monadLift](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Lifting-Monads/#MonadLiftT___mk "Documentation for MonadLiftT.monadLift in Lean Language Reference") x : m ╬▒) >>= fun y => [monadLift](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Lifting-Monads/#MonadLiftT___mk "Documentation for MonadLiftT.monadLift in Lean Language Reference") (f y)`.
Many monad transformers additionally define type classes in the style of `[MonadReader](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#MonadReader___mk "Documentation for MonadReader in Lean Language Reference")` that describe the actual effects available in the monad. This can provide more flexibility: it allows programs to be written that rely only on an interface, and don't constrain the underlying monad to be implemented by a given transformer. The type classes are a way for programs to express their requirements, and monad transformers are a convenient way to meet these requirements.
##  6.2.1. Failure with `OptionT`[­¤öŚ](find/?domain=Verso.Genre.Manual.section&name=OptionT "Permalink")
Failure, represented by the `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")` monad, and exceptions, represented by the `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")` monad, both have corresponding transformers. In the case of `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference")`, failure can be added to a monad by having it contain values of type `[Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ╬▒` where it would otherwise contain values of type `╬▒`. For example, `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference") ([Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ╬▒)` represents `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` actions that don't always return a value of type `╬▒`. This suggests the definition of the monad transformer `OptionT`:
`def OptionT (m : Type u ŌåÆ Type v) (╬▒ : Type u) : Type v :=   m ([Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ╬▒)`
As an example of `OptionT` in action, consider a program that asks the user questions. The function `getSomeInput` asks for a line of input and removes whitespace from both ends. If the resulting trimmed input is non-empty, then it is returned, but the function fails if there are no non-whitespace characters:
`def getSomeInput : OptionT [IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") := [do](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Syntax/#Lean___Parser___Term___do "Documentation for syntax in Lean Language Reference")   let input ŌåÉ (ŌåÉ [IO.getStdin](https://lean-lang.org/doc/reference/4.26.0/IO/Files___-File-Handles___-and-Streams/#IO___getStdin "Documentation for IO.getStdin in Lean Language Reference")).[getLine](https://lean-lang.org/doc/reference/4.26.0/IO/Files___-File-Handles___-and-Streams/#IO___FS___Stream___mk "Documentation for IO.FS.Stream.getLine in Lean Language Reference")   let trimmed := input.[trim](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___trim "Documentation for String.trim in Lean Language Reference")   if trimmed == "" then     failure   else [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") trimmed`
This particular application tracks users with their name and their favorite species of beetle:
`structure UserInfo where   name : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")   favoriteBeetle : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")`
Asking the user for input is no more verbose than a function that uses only `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` would be:
`def getUserInfo : OptionT [IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference") [UserInfo](Monad-Transformers/A-Monad-Construction-Kit/#Opt___UserInfo "Definition of example") := [do](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Syntax/#Lean___Parser___Term___do "Documentation for syntax in Lean Language Reference")   [IO.println](https://lean-lang.org/doc/reference/4.26.0/IO/Console-Output/#IO___println "Documentation for IO.println in Lean Language Reference") "What is your name?"   let name ŌåÉ getSomeInput   [IO.println](https://lean-lang.org/doc/reference/4.26.0/IO/Console-Output/#IO___println "Documentation for IO.println in Lean Language Reference") "What is your favorite species of beetle?"   let beetle ŌåÉ getSomeInput   [pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") Ō¤©name, beetleŌ¤®`
However, because the function runs in an `OptionT [IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")` context rather than just in `[IO](https://lean-lang.org/doc/reference/4.26.0/IO/Logical-Model/#IO "Documentation for IO in Lean Language Reference")`, failure in the first call to `getSomeInput` causes the whole `getUserInfo` to fail, with control never reaching the question about beetles. An alternative solution is to define functions whose type signatures guide Lean to the correct instances. In fact, `OptionT` could have been defined as a structure:
`structure OptionT (m : Type u ŌåÆ Type v) (╬▒ : Type u) : Type v where   run : m ([Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ╬▒)`

The downside to doing this is that the resulting code is more complicated, and these structures can make it more difficult to read proofs. The best of both worlds can be achieved by defining functions that serve the same role as the constructor `OptionT.mk` and the field `[OptionT.run](Monad-Transformers/A-Monad-Construction-Kit/#Opt___Structed___OptionT___run "Definition of example")`, but that work with the direct definition:
`def OptionT.mk (x : m ([Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ╬▒)) : OptionT m ╬▒ := x  def OptionT.run (x : OptionT m ╬▒) : m ([Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") ╬▒) := x`

After defining the monad instance, it's a good idea to check that the monad contract is satisfied. The first step is to show that `[bind](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Bind___mk "Documentation for Bind.bind in Lean Language Reference") ([pure](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/#Pure___mk "Documentation for Pure.pure in Lean Language Reference") v) f` is the same as `f v`. ###  6.2.2.1. Type Classes for Exceptions[­¤öŚ](find/?domain=Verso.Genre.Manual.section&name=exceptions-type-classes "Permalink")
Exception handling fundamentally consists of two operations: the ability to throw exceptions, and the ability to recover from them. Thus far, this has been accomplished using the constructors of `[Except](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Except___error "Documentation for Except in Lean Language Reference")` and pattern matching, respectively. However, this ties a program that uses exceptions to one specific encoding of the exception handling effect. Using a type class to capture these operations allows a program that uses exceptions to be used in _any_ monad that supports throwing and catching.
Throwing an exception should take an exception as an argument, and it should be allowed in any context where a monadic action is requested. The ŌĆ£any contextŌĆØ part of the specification can be written as a type by writing `m ╬▒`ŌĆöbecause there's no way to produce a value of any arbitrary type, the `throw` operation must be doing something that causes control to leave that part of the program. Catching an exception should accept any monadic action together with a handler, and the handler should explain how to get back to the action's type from an exception:
`class MonadExcept (╬Ą : [outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference") (Type u)) (m : Type v ŌåÆ Type w) where   throw : ╬Ą ŌåÆ m ╬▒   tryCatch : m ╬▒ ŌåÆ (╬Ą ŌåÆ m ╬▒) ŌåÆ m ╬▒`
The universe levels on `MonadExcept` differ from those of `ExceptT`. In `ExceptT`, both `╬Ą` and `╬▒` have the same level, while `MonadExcept` imposes no such limitation. This is because `MonadExcept` never places an exception value inside of `m`. The most general universe signature recognizes the fact that `╬Ą` and `╬▒` are completely independent in this definition. Being more general means that the type class can be instantiated for a wider variety of types.

##  6.2.3. State[­¤öŚ](find/?domain=Verso.Genre.Manual.section&name=state-monad "Permalink")

##  6.2.4. `Of` Classes and `The` Functions[­¤öŚ](find/?domain=Verso.Genre.Manual.section&name=of-and-the "Permalink")
Thus far, each monad type class that takes extra information, like the type of exceptions for `MonadExcept` or the type of the state for `MonadState`, has this type of extra information as an output parameter. For simple programs, this is generally convenient, because a monad that combines one use each of `[StateT](Monad-Transformers/A-Monad-Construction-Kit/#St___StateT "Definition of example")`, `[ReaderT](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#ReaderT "Documentation for ReaderT in Lean Language Reference")`, and `ExceptT` has only a single state type, environment type, and exception type. As monads grow in complexity, however, they may involve multiple states or errors types. In this case, the use of an output parameter makes it impossible to target both states in the same `do`-block.
For these cases, there are additional type classes in which the extra information is not an output parameter. These versions of the type classes use the word `Of` in the name. For example, `[MonadStateOf](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#MonadStateOf___mk "Documentation for MonadStateOf in Lean Language Reference")` is like `MonadState`, but without an `[outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference")` modifier.
Instead of an `[outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference")`, these classes use a `[semiOutParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#semiOutParam "Documentation for semiOutParam in Lean Language Reference")` for their respective state, environment, or exception types. Like an `[outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference")`, a `[semiOutParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#semiOutParam "Documentation for semiOutParam in Lean Language Reference")` is not required be known before Lean begins the process of searching for an instance. However, there is an important difference: `[outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference")`s are ignored during the search for an instance, and as a result they are truly outputs. If an `[outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference")` is known prior to the search, then Lean merely checks that the result of the search is the same as what was known. On the other hand, a `[semiOutParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#semiOutParam "Documentation for semiOutParam in Lean Language Reference")` that is known prior to the start of the search can be used to narrow down candidates, just like an input parameter.
When a state monad's state type is an `[outParam](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#outParam "Documentation for outParam in Lean Language Reference")`, then each monad can have at most one type of state. This is convenient, because it improves type inference: the state type can be inferred in more circumstances. This is also inconvenient, because a monad built from multiple uses of `[StateT](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#StateT "Documentation for StateT in Lean Language Reference")` cannot provide a useful `[MonadState](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#MonadState___mk "Documentation for MonadState in Lean Language Reference")` instance. Using `[MonadStateOf](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#MonadStateOf___mk "Documentation for MonadStateOf in Lean Language Reference")`, however, causes Lean to take the state type into account when it is available to select which instance to use, so one monad may provide multiple types of state. The downside of this is that the resulting instance may not be the one that was intended when the state type has not been specified explicitly enough, which can lead to confusing error messages.
Similarly, there are versions of the type class methods that accept the type of the extra information as an _explicit_ , rather than implicit, argument. For `[MonadStateOf](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#MonadStateOf___mk "Documentation for MonadStateOf in Lean Language Reference")`, there are `[getThe](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#getThe "Documentation for getThe in Lean Language Reference")` with type
`(Žā : Type u) ŌåÆ {m : Type u ŌåÆ Type v} ŌåÆ [[MonadStateOf](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#MonadStateOf___mk "Documentation for MonadStateOf in Lean Language Reference") Žā m] ŌåÆ m Žā`
and `[modifyThe](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#modifyThe "Documentation for modifyThe in Lean Language Reference")` with type
`(Žā : Type u) ŌåÆ {m : Type u ŌåÆ Type v} ŌåÆ [[MonadStateOf](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#MonadStateOf___mk "Documentation for MonadStateOf in Lean Language Reference") Žā m] ŌåÆ (Žā ŌåÆ Žā) ŌåÆ m [PUnit](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/The-Unit-Type/#PUnit___unit "Documentation for PUnit in Lean Language Reference")`
There is no `setThe` because the type of the new state is enough to decide which surrounding state monad transformer to use.
In the Lean standard library, there are instances of the non-`Of` versions of the classes defined in terms of the instances of the versions with `Of`. In other words, implementing the `Of` version yields implementations of both. It's generally a good idea to implement the `Of` version, and then start writing programs using the non-`Of` versions of the class, transitioning to the `Of` version if the output parameter becomes inconvenient.
##  6.2.5. Transformers and `Id`[­¤öŚ](find/?domain=Verso.Genre.Manual.section&name=transformers-and-Id "Permalink")
The identity monad `[Id](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Id "Documentation for Id in Lean Language Reference")` is the monad that has no effects whatsoever, to be used in contexts that expect a monad for some reason but where none is actually necessary. Another use of `[Id](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Id "Documentation for Id in Lean Language Reference")` is to serve as the bottom of a stack of monad transformers. For instance, `[StateT](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#StateT "Documentation for StateT in Lean Language Reference") Žā [Id](https://lean-lang.org/doc/reference/4.26.0/Functors___-Monads-and--do--Notation/Varieties-of-Monads/#Id "Documentation for Id in Lean Language Reference")` works just like `State Žā`.
