#  1.7. Additional Conveniences[🔗](find/?domain=Verso.Genre.Manual.section&name=getting-to-know-conveniences "Permalink")
Lean contains a number of convenience features that make programs much more concise.
##  1.7.1. Automatic Implicit Parameters[🔗](find/?domain=Verso.Genre.Manual.section&name=automatic-implicit-parameters "Permalink")
When writing polymorphic functions in Lean, it is typically not necessary to list all the implicit parameters. Instead, they can simply be mentioned. If Lean can determine their type, then they are automatically inserted as implicit parameters. In other words, the previous definition of `length`:
`def length {α : Type} (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") xs [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [] => 0   | y :: ys => [Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") (length ys)`
can be written without `{α : Type}`:
`def length (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") xs [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [] => 0   | y :: ys => [Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") (length ys)`
This can greatly simplify highly polymorphic definitions that take many implicit parameters.
##  1.7.2. Pattern-Matching Definitions[🔗](find/?domain=Verso.Genre.Manual.section&name=pattern-matching-definitions "Permalink")
When defining functions with `def`, it is quite common to name an argument and then immediately use it with pattern matching. For instance, in `length`, the argument `xs` is used only in `match`. In these situations, the cases of the `match` expression can be written directly, without naming the argument at all.
The first step is to move the arguments' types to the right of the colon, so the return type is a function type. For instance, the type of `length` is `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`. Then, replace the `:=` with each case of the pattern match:
`def length : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")   | [] => 0   | y :: ys => [Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") (length ys)`
This syntax can also be used to define functions that take more than one argument. In this case, their patterns are separated by commas. For instance, `drop` takes a number `nnn` and a list, and returns the list after removing the first `nnn` entries.
`def drop : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α   | [Nat.zero](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.zero in Lean Language Reference"), xs => xs   | _, [] => []   | [Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") n, x :: xs => drop n xs`
Named arguments and patterns can also be used in the same definition. For instance, a function that takes a default value and an optional value, and returns the default when the optional value is `[none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`, can be written:
`def fromOption (default : α) : [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") α → α   | [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference") => default   | [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") x => x`
This function is called `[Option.getD](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___getD "Documentation for Option.getD in Lean Language Reference")` in the standard library, and can be called with dot notation:
``"salmonberry"`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") ([some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") "salmonberry").[getD](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___getD "Documentation for Option.getD in Lean Language Reference") ""`
```
"salmonberry"
```
``""`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference").[getD](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___getD "Documentation for Option.getD in Lean Language Reference") ""`

##  1.7.3. Local Definitions[🔗](find/?domain=Verso.Genre.Manual.section&name=local-definitions "Permalink")
It is often useful to name intermediate steps in a computation. In many cases, intermediate values represent useful concepts all on their own, and naming them explicitly can make the program easier to read. In other cases, the intermediate value is used more than once. As in most other languages, writing down the same code twice in Lean causes it to be computed twice, while saving the result in a variable leads to the result of the computation being saved and re-used.
For instance, `unzip` is a function that transforms a list of pairs into a pair of lists. When the list of pairs is empty, then the result of `unzip` is a pair of empty lists. When the list of pairs has a pair at its head, then the two fields of the pair are added to the result of unzipping the rest of the list. This definition of `unzip` follows that description exactly:
`def unzip : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") (α × β) → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α × [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") β   | [] => ([], [])   | (x, y) :: xys =>     (x :: (unzip xys).[fst](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.fst in Lean Language Reference"), y :: (unzip xys).[snd](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.snd in Lean Language Reference"))`
Unfortunately, there is a problem: this code is slower than it needs to be. Each entry in the list of pairs leads to two recursive calls, which makes this function take exponential time. However, both recursive calls will have the same result, so there is no reason to make the recursive call twice.
In Lean, the result of the recursive call can be named, and thus saved, using `let`. Local definitions with `let` resemble top-level definitions with `def`: it takes a name to be locally defined, arguments if desired, a type signature, and then a body following `:=`. After the local definition, the expression in which the local definition is available (called the _body_ of the `let`-expression) must be on a new line, starting at a column in the file that is less than or equal to that of the `let` keyword. A local definition with `let` in `unzip` looks like this:
`def unzip : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") (α × β) → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α × [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") β   | [] => ([], [])   | (x, y) :: xys =>     let unzipped : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α × [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") β := unzip xys     (x :: unzipped.[fst](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.fst in Lean Language Reference"), y :: unzipped.[snd](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.snd in Lean Language Reference"))`
To use `let` on a single line, separate the local definition from the body with a semicolon.
Local definitions with `let` may also use pattern matching when one pattern is enough to match all cases of a datatype. In the case of `unzip`, the result of the recursive call is a pair. Because pairs have only a single constructor, the name `unzipped` can be replaced with a pair pattern:
`def unzip : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") (α × β) → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α × [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") β   | [] => ([], [])   | (x, y) :: xys =>     let (xs, ys) : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α × [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") β := [unzip](Getting-to-Know-Lean/Additional-Conveniences/#WithPattern___unzip "Definition of example") xys     (x :: xs, y :: ys)`
Judicious use of patterns with `let` can make code easier to read, compared to writing the accessor calls by hand.
The biggest difference between `let` and `def` is that recursive `let` definitions must be explicitly indicated by writing `let rec`. For instance, one way to reverse a list involves a recursive helper function, as in this definition:
`def reverse (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α :=   let rec helper : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α     | [], soFar => soFar     | y :: ys, soFar => [helper](Getting-to-Know-Lean/Additional-Conveniences/#reverse___helper "Definition of example") ys (y :: soFar)   [helper](Getting-to-Know-Lean/Additional-Conveniences/#reverse___helper "Definition of example") xs []`
The helper function walks down the input list, moving one entry at a time over to `soFar`. When it reaches the end of the input list, `soFar` contains a reversed version of the input.
##  1.7.4. Type Inference[🔗](find/?domain=Verso.Genre.Manual.section&name=type-inference "Permalink")
In many situations, Lean can automatically determine an expression's type. In these cases, explicit types may be omitted from both top-level definitions (with `def`) and local definitions (with `let`). For example, the recursive call to `unzip` does not need an annotation:
`def unzip : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") (α × β) → [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α × [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") β   | [] => ([], [])   | (x, y) :: xys =>     let unzipped := unzip xys     (x :: unzipped.[fst](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.fst in Lean Language Reference"), y :: unzipped.[snd](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.snd in Lean Language Reference"))`
As a rule of thumb, omitting the types of literal values (like strings and numbers) usually works, although Lean may pick a type for literal numbers that is more specific than the intended type. Lean can usually determine a type for a function application, because it already knows the argument types and the return type. Omitting return types for function definitions will often work, but function parameters typically require annotations. Definitions that are not functions, like `unzipped` in the example, do not need type annotations if their bodies do not need type annotations, and the body of this definition is a function application.
Omitting the return type for `unzip` is possible when using an explicit `match` expression:
`def unzip (pairs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") (α × β)) :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") pairs [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [] => ([], [])   | (x, y) :: xys =>     let unzipped := unzip xys     (x :: unzipped.[fst](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.fst in Lean Language Reference"), y :: unzipped.[snd](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.snd in Lean Language Reference"))`
Generally speaking, it is a good idea to err on the side of too many, rather than too few, type annotations. First off, explicit types communicate assumptions about the code to readers. Even if Lean can determine the type on its own, it can still be easier to read code without having to repeatedly query Lean for type information. Secondly, explicit types help localize errors. The more explicit a program is about its types, the more informative the error messages can be. This is especially important in a language like Lean that has a very expressive type system. Thirdly, explicit types make it easier to write the program in the first place. The type is a specification, and the compiler's feedback can be a helpful tool in writing a program that meets the specification. Finally, Lean's type inference is a best-effort system. Because Lean's type system is so expressive, there is no “best” or most general type to find for all expressions. This means that even if you get a type, there's no guarantee that it's the _right_ type for a given application. For instance, `14` can be a `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` or an `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")`:
``14 : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") 14`
```
14 : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")
```
``14 : [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") (14 : [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference"))`
```
14 : [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")
```

Missing type annotations can give confusing error messages. Omitting all types from the definition of `unzip`:
`def unzip pairs :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") pairs [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | `Invalid match expression: This pattern contains metavariables:   [[](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.nil in Lean Language Reference")[]](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.nil in Lean Language Reference")`[] => ([], []) | (x, y) :: xys => let unzipped := unzip xys (x :: unzipped.[fst](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.fst in Lean Language Reference"), y :: unzipped.[snd](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Tuples/#Prod___mk "Documentation for Prod.snd in Lean Language Reference"))`
leads to a message about the `match` expression:

```
Invalid match expression: This pattern contains metavariables:
  [[](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.nil in Lean Language Reference")[]](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.nil in Lean Language Reference")
```

This is because `match` needs to know the type of the value being inspected, but that type was not available. A “metavariable” is an unknown part of a program, written `?m.XYZ` in error messages—they are described in the [section on Polymorphism](Getting-to-Know-Lean/Polymorphism/#polymorphism). In this program, the type annotation on the argument is required.
Even some very simple programs require type annotations. For instance, the identity function just returns whatever argument it is passed. With argument and type annotations, it looks like this:
`def id (x : α) : α := x`
Lean is capable of determining the return type on its own:
`def id (x : α) := x`
Omitting the argument type, however, causes an error:
`def `Failed to infer type of definition `id``id `Failed to infer type of binder `x``x := x`
```
Failed to infer type of binder `x`
```

In general, messages that say something like “failed to infer” or that mention metavariables are often a sign that more type annotations are necessary. Especially while still learning Lean, it is useful to provide most types explicitly.
##  1.7.5. Simultaneous Matching[🔗](find/?domain=Verso.Genre.Manual.section&name=simultaneous-matching "Permalink")
Pattern-matching expressions, just like pattern-matching definitions, can match on multiple values at once. Both the expressions to be inspected and the patterns that they match against are written with commas between them, similarly to the syntax used for definitions. Here is a version of `drop` that uses simultaneous matching:
`def drop (n : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") n, xs [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [Nat.zero](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.zero in Lean Language Reference"), ys => ys   | _, [] => []   | [Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") n , y :: ys => drop n ys`
Simultaneous matching resembles matching on a pair, but there is an important difference. Lean tracks the connection between the expression being matched and the patterns, and this information is used for purposes that include checking for termination and propagating static type information. Simultaneously matching both lists is accepted:
`def sameLength (xs : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") α) (ys : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") β) : [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") xs, ys [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [], [] => [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")   | x :: xs', y :: ys' => [sameLength](Getting-to-Know-Lean/Additional-Conveniences/#Both___sameLength "Definition of example") xs' ys'   | _, _ => [false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")`
##  1.7.6. Natural Number Patterns[🔗](find/?domain=Verso.Genre.Manual.section&name=natural-number-patterns "Permalink")
In the section on [datatypes and patterns](Getting-to-Know-Lean/Datatypes-and-Patterns/#datatypes-and-patterns), `even` was defined like this:
`def even (n : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") n [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [Nat.zero](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.zero in Lean Language Reference") => [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")   | [Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") k => [not](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___not "Documentation for Bool.not in Lean Language Reference") (even k)`
Just as there is special syntax to make list patterns more readable than using `[List.cons](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.cons in Lean Language Reference")` and `[List.nil](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List.nil in Lean Language Reference")` directly, natural numbers can be matched using literal numbers and `+`. For example, `even` can also be defined like this:
`def even : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Bool](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool in Lean Language Reference")   | 0 => [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")   | n + 1 => [not](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___not "Documentation for Bool.not in Lean Language Reference") (even n)`
In this notation, the arguments to the `+` pattern serve different roles. Behind the scenes, the left argument (`n` above) becomes an argument to some number of `[Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference")` patterns, and the right argument (`1` above) determines how many `[Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference")`s to wrap around the pattern. The explicit patterns in `halve`, which divides a `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` by two and drops the remainder:
`def halve : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")   | [Nat.zero](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.zero in Lean Language Reference") => 0   | [Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") [Nat.zero](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.zero in Lean Language Reference") => 0   | [Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") ([Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") n) => halve n + 1`
can be replaced by numeric literals and `+`:
`def halve : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")   | 0 => 0   | 1 => 0   | n + 2 => [halve](Getting-to-Know-Lean/Additional-Conveniences/#Match___halve "Definition of example") n + 1`
Behind the scenes, both definitions are completely equivalent. Remember: `[halve](Getting-to-Know-Lean/Additional-Conveniences/#Match___halve "Definition of example") n + 1` is equivalent to `([halve](Getting-to-Know-Lean/Additional-Conveniences/#Match___halve "Definition of example") n) + 1`, not `[halve](Getting-to-Know-Lean/Additional-Conveniences/#Match___halve "Definition of example") (n + 1)`.
When using this syntax, the second argument to `+` should always be a literal `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`. Even though addition is commutative, flipping the arguments in a pattern can result in errors like the following:
`def halve : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")   | 0 => 0   | 1 => 0   `Invalid pattern(s): `n` is an explicit pattern variable, but it only occurs in positions that are inaccessible to pattern matching:   .([Nat.add](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___add "Documentation for Nat.add in Lean Language Reference") 2 n)`| 2 + n => halve n + 1`
```
Invalid pattern(s): `n` is an explicit pattern variable, but it only occurs in positions that are inaccessible to pattern matching:
  .([Nat.add](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___add "Documentation for Nat.add in Lean Language Reference") 2 n)
```

This restriction enables Lean to transform all uses of the `+` notation in a pattern into uses of the underlying `[Nat.succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference")`, keeping the language simpler behind the scenes.
##  1.7.7. Anonymous Functions[🔗](find/?domain=Verso.Genre.Manual.section&name=anonymous-functions "Permalink")
Functions in Lean need not be defined at the top level. As expressions, functions are produced with the `fun` syntax. Function expressions begin with the keyword `fun`, followed by one or more parameters, which are separated from the return expression using `=>`. For instance, a function that adds one to a number can be written:
``fun x => x [+](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAdd___mk "Documentation for HAdd.hAdd in Lean Language Reference") 1 : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") fun x => x + 1`
```
fun x => x [+](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAdd___mk "Documentation for HAdd.hAdd in Lean Language Reference") 1 : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")
```

Type annotations are written the same way as on `def`, using parentheses and colons:
``fun x => x [+](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAdd___mk "Documentation for HAdd.hAdd in Lean Language Reference") 1 : [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") → [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") fun (x : [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")) => x + 1`
```
fun x => x [+](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAdd___mk "Documentation for HAdd.hAdd in Lean Language Reference") 1 : [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference") → [Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")
```

Similarly, implicit parameters may be written with curly braces:
``fun {α} x => x : {α : Type} → α → α`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") fun {α : Type} (x : α) => x`
```
fun {α} x => x : {α : Type} → α → α
```

This style of anonymous function expression is often referred to as a _lambda expression_ , because the typical notation used in mathematical descriptions of programming languages uses the Greek letter λ (lambda) where Lean has the keyword `fun`. Even though Lean does permit `λ` to be used instead of `fun`, it is most common to write `fun`.
Anonymous functions also support the multiple-pattern style used in `def`. For instance, a function that returns the predecessor of a natural number if it exists can be written:
``fun x =>   match x with   | 0 => [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")   | n.[succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") => [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") n : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") fun | 0 => [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference") | n + 1 => [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") n`
```
fun x =>
  match x with
  | 0 => [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")
  | n.[succ](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat.succ in Lean Language Reference") => [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") n : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")
```

Note that Lean's own description of the function has a named argument and a `match` expression. Many of Lean's convenient syntactic shorthands are expanded to simpler syntax behind the scenes, and the abstraction sometimes leaks.
Definitions using `def` that take arguments may be rewritten as function expressions. For instance, a function that doubles its argument can be written as follows:
`def double : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") := fun   | 0 => 0   | k + 1 => [double](Getting-to-Know-Lean/Additional-Conveniences/#Double___double "Definition of example") k + 2`
When an anonymous function is very simple, like `fun x => x + 1`, the syntax for creating the function can be fairly verbose. In that particular example, six non-whitespace characters are used to introduce the function, and its body consists of only three non-whitespace characters. For these simple cases, Lean provides a shorthand. In an expression surrounded by parentheses, a centered dot character `·` can stand for a parameter, and the expression inside the parentheses becomes the function's body. That particular function can also be written `(· + 1)`.
The centered dot always creates a function out of the _closest_ surrounding set of parentheses. For instance, `(· + 5, 3)` is a function that returns a pair of numbers, while `((· + 5), 3)` is a pair of a function and a number. If multiple dots are used, then they become parameters from left to right:
`(· , ·) 1 2``(1, ·) 2``(1, 2)`
Anonymous functions can be applied in precisely the same way as functions defined using `def` or `let`. The command ``10`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") (fun x => x + x) 5` results in:

```
10
```

while ``10`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") (· * 2) 5` results in:

```
10
```

##  1.7.8. Namespaces[🔗](find/?domain=Verso.Genre.Manual.section&name=namespaces "Permalink")
Each name in Lean occurs in a _namespace_ , which is a collection of names. Names are placed in namespaces using `.`, so `[List.map](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___map "Documentation for List.map in Lean Language Reference")` is the name `[map](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___map "Documentation for List.map in Lean Language Reference")` in the `List` namespace. Names in different namespaces do not conflict with each other, even if they are otherwise identical. This means that `[List.map](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___map "Documentation for List.map in Lean Language Reference")` and `[Array.map](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Arrays/#Array___map "Documentation for Array.map in Lean Language Reference")` are different names. Namespaces may be nested, so `Project.Frontend.User.loginTime` is the name `loginTime` in the nested namespace `Project.Frontend.User`.
Names can be directly defined within a namespace. For instance, the name `[double](Getting-to-Know-Lean/Additional-Conveniences/#Nat___double "Definition of example")` can be defined in the `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` namespace:
`def Nat.double (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") := x + x`
Because `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` is also the name of a type, dot notation is available to call `[Nat.double](Getting-to-Know-Lean/Additional-Conveniences/#Nat___double "Definition of example")` on expressions with type `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`:
``8`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") (4 : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")).[double](Getting-to-Know-Lean/Additional-Conveniences/#Nat___double "Definition of example")`
```
8
```

In addition to defining names directly in a namespace, a sequence of declarations can be placed in a namespace using the `namespace` and `end` commands. For instance, this defines `triple` and `quadruple` in the namespace `NewNamespace`:
`[namespace](https://lean-lang.org/doc/reference/4.26.0/Namespaces-and-Sections/#Lean___Parser___Command___namespace "Documentation for syntax in Lean Language Reference") NewNamespace def triple (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") := 3 * x def quadruple (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") := 2 * x + 2 * x end NewNamespace`
To refer to them, prefix their names with `NewNamespace.`:
``NewNamespace.triple (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") NewNamespace.triple`
```
NewNamespace.triple (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")
```
``NewNamespace.quadruple (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") NewNamespace.quadruple`
```
NewNamespace.quadruple (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")
```

Namespaces may be _opened_ , which allows the names in them to be used without explicit qualification. Writing `open` `MyNamespace ``in` before an expression causes the contents of `MyNamespace` to be available in the expression. For example, `timesTwelve` uses both `quadruple` and `triple` after opening `NewNamespace`:
`def timesTwelve (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) :=   open NewNamespace in   quadruple (triple x)`
Namespaces can also be opened prior to a command. This allows all parts of the command to refer to the contents of the namespace, rather than just a single expression. To do this, place the `open`﻿` ... ``in` prior to the command.
`[open](https://lean-lang.org/doc/reference/4.26.0/Namespaces-and-Sections/#Lean___Parser___Command___open "Documentation for syntax in Lean Language Reference") NewNamespace [in](https://lean-lang.org/doc/reference/4.26.0/Namespaces-and-Sections/#Lean___Parser___Command___in "Documentation for syntax in Lean Language Reference") `NewNamespace.quadruple (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") quadruple`
```
NewNamespace.quadruple (x : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")) : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")
```

Function signatures show the name's full namespace. Namespaces may additionally be opened for _all_ following commands for the rest of the file. To do this, simply omit the `in` from a top-level usage of `open`.
##  1.7.9. `if let`[🔗](find/?domain=Verso.Genre.Manual.section&name=if-let "Permalink")
When consuming values that have a sum type, it is often the case that only a single constructor is of interest. For example, given this type that represents a subset of Markdown inline elements:
`inductive Inline : Type where   | lineBreak   | string : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") → [Inline](Getting-to-Know-Lean/Additional-Conveniences/#Inline "Definition of example")   | emph : [Inline](Getting-to-Know-Lean/Additional-Conveniences/#Inline "Definition of example") → [Inline](Getting-to-Know-Lean/Additional-Conveniences/#Inline "Definition of example")   | strong : [Inline](Getting-to-Know-Lean/Additional-Conveniences/#Inline "Definition of example") → [Inline](Getting-to-Know-Lean/Additional-Conveniences/#Inline "Definition of example")`
a function that recognizes string elements and extracts their contents can be written:
`def Inline.string? (inline : [Inline](Getting-to-Know-Lean/Additional-Conveniences/#Inline "Definition of example")) : [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") :=   [match](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference") inline [with](https://lean-lang.org/doc/reference/4.26.0/Terms/Pattern-Matching/#Lean___Parser___Term___match "Documentation for syntax in Lean Language Reference")   | [Inline.string](Getting-to-Know-Lean/Additional-Conveniences/#Inline___string "Definition of example") s => [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") s   | _ => [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`
An alternative way of writing this function's body uses `if` together with `let`:
`def Inline.string? (inline : [Inline](Getting-to-Know-Lean/Additional-Conveniences/#Inline "Definition of example")) : [Option](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option in Lean Language Reference") [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference") :=   [if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfLet "Documentation for syntax in Lean Language Reference") [let](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfLet "Documentation for syntax in Lean Language Reference") [Inline.string](Getting-to-Know-Lean/Additional-Conveniences/#Inline___string "Definition of example") s := inline [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfLet "Documentation for syntax in Lean Language Reference")     [some](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.some in Lean Language Reference") s   [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfLet "Documentation for syntax in Lean Language Reference") [none](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Optional-Values/#Option___none "Documentation for Option.none in Lean Language Reference")`
This is very much like the pattern-matching `let` syntax. The difference is that it can be used with sum types, because a fallback is provided in the `else` case. In some contexts, using `if let` instead of `match` can make code easier to read.
##  1.7.10. Positional Structure Arguments[🔗](find/?domain=Verso.Genre.Manual.section&name=positional-structure-arguments "Permalink")
The [section on structures](Getting-to-Know-Lean/Structures/#structures) presents two ways of constructing structures:
  1. The constructor can be called directly, as in `Point.mk 1 2`.
  2. Brace notation can be used, as in `{ x := 1, y := 2 }`.


In some contexts, it can be convenient to pass arguments positionally, rather than by name, but without naming the constructor directly. For instance, defining a variety of similar structure types can help keep domain concepts separate, but the natural way to read the code may treat each of them as being essentially a tuple. In these contexts, the arguments can be enclosed in angle brackets `⟨` and `⟩`. A `Point` can be written `⟨1, 2⟩`. Be careful! Even though they look like the less-than sign `<` and greater-than sign `>`, these brackets are different. They can be input using `\<` and `\>`, respectively.
Just as with the brace notation for named constructor arguments, this positional syntax can only be used in a context where Lean can determine the structure's type, either from a type annotation or from other type information in the program. For instance, `[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") ⟨1, 2⟩` yields the following error:

```
Invalid `⟨...⟩` notation: The expected type of this term could not be determined
```

This error occurs because there is no type information available. Adding an annotation, such as in `[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") (⟨1, 2⟩ : Point)`, solves the problem:

```
{ x := 1.000000, y := 2.000000 }
```

##  1.7.11. String Interpolation[🔗](find/?domain=Verso.Genre.Manual.section&name=string-interpolation "Permalink")
In Lean, prefixing a string with `s!` triggers _interpolation_ , where expressions contained in curly braces inside the string are replaced with their values. This is similar to `f`-strings in Python and `$`-prefixed strings in C#. For instance,
``"three fives is 15"`[#eval](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___eval "Documentation for syntax in Lean Language Reference") s!"three fives is {NewNamespace.triple 5}"`
yields the output

```
"three fives is 15"
```

Not all expressions can be interpolated into a string. For instance, attempting to interpolate a function results in an error.
``toString "three fives is " [++](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#HAppend___mk "Documentation for HAppend.hAppend in Lean Language Reference") sorry : [String](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Strings/#String___mk "Documentation for String in Lean Language Reference")`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") s!"three fives is {`failed to synthesize   ToString ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference"))  Hint: Additional diagnostic information may be available using the `set_option diagnostics true` command.`NewNamespace.triple}"`
yields the error

```
failed to synthesize
  ToString ([Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference") → [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference"))

Hint: Additional diagnostic information may be available using the `set_option diagnostics true` command.
```

This is because there is no standard way to convert functions into strings. Just as the compiler maintains a table that describes how to display the result of evaluating expressions of various types, it maintains a table that describes how to convert values of various types into strings. The message `failed to synthesize instance` means that the Lean compiler didn't find an entry in this table for the given type. The chapter on [type classes](Overloading-and-Type-Classes/#type-classes) describes this mechanism in more detail, including the means of adding new entries to the table.
