#  10. Type Classes[рҹ”—](find/?domain=Verso.Genre.Manual.section&name=type-classes "Permalink")
Type classes were introduced as a principled way of enabling ad-hoc polymorphism in functional programming languages. We first observe that it would be easy to implement an ad-hoc polymorphic function (such as addition) if the function simply took the type-specific implementation of addition as an argument and then called that implementation on the remaining arguments. For example, suppose we declare a structure in Lean to hold implementations of addition.
`[namespace](https://lean-lang.org/doc/reference/4.26.0/Namespaces-and-Sections/#Lean___Parser___Command___namespace "Documentation for syntax in Lean Language Reference") Ex `
`structure Add (Оұ : Type) where   add : Оұ вҶ’ Оұ вҶ’ Оұ  ```@Add.add : {Оұ : Type} вҶ’ Add Оұ вҶ’ Оұ вҶ’ Оұ вҶ’ Оұ`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") @Add.add`
```
@Add.add : {Оұ : Type} вҶ’ Add Оұ вҶ’ Оұ вҶ’ Оұ вҶ’ Оұ

`end Ex `

Of course, it would be highly cumbersome for users to manually pass the implementations around in this way. Indeed, it would defeat most of the potential benefits of ad-hoc polymorphism.
The main idea behind type classes is to make arguments such as `Add Оұ` implicit, and to use a database of user-defined instances to synthesize the desired instances automatically through a process known as typeclass resolution. In Lean, by changing `structure` to `class` in the example above, the type of `Add.add` becomes:
`[namespace](https://lean-lang.org/doc/reference/4.26.0/Namespaces-and-Sections/#Lean___Parser___Command___namespace "Documentation for syntax in Lean Language Reference") Ex `
`class Add (Оұ : Type) where   add : Оұ вҶ’ Оұ вҶ’ Оұ  ```@Add.add : {Оұ : Type} вҶ’ [self : Add Оұ] вҶ’ Оұ вҶ’ Оұ вҶ’ Оұ`[#check](https://lean-lang.org/doc/reference/4.26.0/Interacting-with-Lean/#Lean___Parser___Command___check "Documentation for syntax in Lean Language Reference") @Add.add`
```
@Add.add : {Оұ : Type} вҶ’ [self : Add Оұ] вҶ’ Оұ вҶ’ Оұ вҶ’ Оұ

`end Ex `

The example above demonstrates how type classes are used to overload notation. Now, we explore another application. We often need an arbitrary element of a given type. Recall that types may not have any elements in Lean. It often happens that we would like a definition to return an arbitrary element in a вҖңcorner case.вҖқ For example, we may like the expression `head xs` to be of type `Оұ` when `xs` is of type `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") Оұ`. Similarly, many theorems hold under the additional assumption that a type is not empty. For example, if `Оұ` is a type, `вҲғ x : Оұ, x = x` is true only if `Оұ` is not empty. The standard library defines a type class `[Inhabited](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Inhabited___mk "Documentation for Inhabited in Lean Language Reference")` to enable type class inference to infer a вҖңdefaultвҖқ element of an inhabited type. ##  10.1. Chaining Instances[рҹ”—](find/?domain=Verso.Genre.Manual.section&name=chaining-instances "Permalink")
If that were the extent of type class inference, it would not be all that impressive; it would be simply a mechanism of storing a list of instances for the elaborator to find in a lookup table. What makes type class inference powerful is that one can _chain_ instances. That is, an instance declaration can in turn depend on an implicit instance of a type class. This causes class inference to chain through instances recursively, backtracking when necessary, in a Prolog-like search.
For example, the following definition shows that if two types `Оұ` and `ОІ` are inhabited, then so is their product:
`instance [[Inhabited](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Inhabited___mk "Documentation for Inhabited in Lean Language Reference") Оұ] [[Inhabited](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Inhabited___mk "Documentation for Inhabited in Lean Language Reference") ОІ] : [Inhabited](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Inhabited___mk "Documentation for Inhabited in Lean Language Reference") (Оұ Г— ОІ) where   [default](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Inhabited___mk "Documentation for Inhabited.default in Lean Language Reference") := ([default](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Inhabited___mk "Documentation for Inhabited.default in Lean Language Reference"), [default](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Inhabited___mk "Documentation for Inhabited.default in Lean Language Reference")) ```

Lean also features _semi-output parameters_ , which have some features of input parameters and some features of output parameters. Instance synthesis with semi-output parameters can be more difficult to predict, because the order in which instances are considered can determine which is selected, but it is also more flexible.

##  10.8. Decidable Propositions[рҹ”—](find/?domain=Verso.Genre.Manual.section&name=decidable-propositions "Permalink")
Let us consider another example of a type class defined in the standard library, namely the type class of `[Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference")` propositions. Roughly speaking, an element of `Prop` is said to be decidable if we can decide whether it is true or false. The distinction is only useful in constructive mathematics; classically, every proposition is decidable. But if we use the classical principle, say, to define a function by cases, that function will not be computable. Algorithmically speaking, the `[Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference")` type class can be used to infer a procedure that effectively determines whether or not the proposition is true. As a result, the type class supports such computational definitions when they are possible while at the same time allowing a smooth transition to the use of classical definitions and classical reasoning.
In the standard library, `[Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference")` is defined formally as follows:
`[namespace](https://lean-lang.org/doc/reference/4.26.0/Namespaces-and-Sections/#Lean___Parser___Command___namespace "Documentation for syntax in Lean Language Reference") Hidden `
`class inductive Decidable (p : Prop) where   | isFalse (h : В¬p) : Decidable p   | isTrue  (h : p)  : Decidable p `
`end Hidden `
Logically speaking, having an element `t : [Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference") p` is stronger than having an element `t' : p вҲЁ В¬p`; it enables us to define values of an arbitrary type depending on the truth value of `p`. For example, for the expression `[if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") p [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termIfThenElse "Documentation for syntax in Lean Language Reference") a else b` to make sense, we need to know that `p` is decidable. That expression is syntactic sugar for `ite p a b`, where `ite` is defined as follows:
`[namespace](https://lean-lang.org/doc/reference/4.26.0/Namespaces-and-Sections/#Lean___Parser___Command___namespace "Documentation for syntax in Lean Language Reference") Hidden `
`def ite {Оұ : Sort u}     (c : Prop) [h : [Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference") c]     (t e : Оұ) : Оұ :=   h.casesOn (motive := fun _ => Оұ) (fun _ => e) (fun _ => t) `
`end Hidden `
The standard library also contains a variant of `ite` called `dite`, the dependent if-then-else expression. It is defined as follows:
`[namespace](https://lean-lang.org/doc/reference/4.26.0/Namespaces-and-Sections/#Lean___Parser___Command___namespace "Documentation for syntax in Lean Language Reference") Hidden `
`def dite {Оұ : Sort u}     (c : Prop) [h : [Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference") c]     (t : c вҶ’ Оұ) (e : [Not](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Logical-Connectives/#Not "Documentation for Not in Lean Language Reference") c вҶ’ Оұ) : Оұ :=   Decidable.casesOn (motive := fun _ => Оұ) h e t `
`end Hidden `
That is, in `dite c t e`, we can assume `hc : c` in the вҖңthenвҖқ branch, and `hnc : В¬c` in the вҖңelseвҖқ branch. To make `dite` more convenient to use, Lean allows us to write `[if](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference") h : c [then](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference") t [else](https://lean-lang.org/doc/reference/4.26.0/Terms/Conditionals/#termDepIfThenElse "Documentation for syntax in Lean Language Reference") e` instead of `dite c (fun h : c => t h) (fun h : В¬c => e h)`.
Without classical logic, we cannot prove that every proposition is decidable. But we can prove that _certain_ propositions are decidable. For example, we can prove the decidability of basic operations like equality and comparisons on the natural numbers and the integers. They work as follows. The expression `[decide](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___decide "Documentation for Decidable.decide in Lean Language Reference") p` tries to infer a decision procedure for `p`, and, if it is successful, evaluates to either `[true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")` or `[false](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.false in Lean Language Reference")`. In particular, if `p` is a true closed expression, `[decide](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___decide "Documentation for Decidable.decide in Lean Language Reference") p` will reduce definitionally to the Boolean `[true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")`. On the assumption that `[decide](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___decide "Documentation for Decidable.decide in Lean Language Reference") p = [true](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Booleans/#Bool___false "Documentation for Bool.true in Lean Language Reference")` holds, `of_decide_eq_true` produces a proof of `p`. The tactic `decide` puts it all together to prove a target `p`. By the previous observations, `decide` will succeed any time the inferred decision procedure for `p` has enough information to evaluate, definitionally, to the `[isTrue](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable.isTrue in Lean Language Reference")` case.

At times, you may find that the type class inference fails to find an expected instance, or, worse, falls into an infinite loop and times out. To help debug in these situations, Lean enables you to request a trace of the search:
`set_option trace.Meta.synthInstance true ```
If you are using VS Code, you can read the results by hovering over the relevant theorem or definition, or opening the messages window with ``Ctrl``Shift``Enter``.
You can also limit the search using the following options:
`set_option [synthInstance.maxHeartbeats](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Synthesis/#synthInstance___maxHeartbeats "Documentation for option synthInstance.maxHeartbeats in Lean Language Reference") 10000 ``set_option synthInstance.maxSize 400 ```
Option `synthInstance.maxHeartbeats` specifies the maximum amount of heartbeats per typeclass resolution problem. A heartbeat is the number of (small) memory allocations (in thousands), 0 means there is no limit. Option `synthInstance.maxSize` is the maximum number of instances used to construct a solution in the type class instance synthesis procedure.
Remember also that in both the VS Code and Emacs editor modes, tab completion works in `set_option`, to help you find suitable options.
As noted above, the type class instances in a given context represent a Prolog-like program, which gives rise to a backtracking search. Both the efficiency of the program and the solutions that are found can depend on the order in which the system tries the instance. Instances which are declared last are tried first. Moreover, if instances are declared in other modules, the order in which they are tried depends on the order in which namespaces are opened. Instances declared in namespaces which are opened later are tried earlier.
You can change the order that type class instances are tried by assigning them a _priority_. When an instance is declared, it is assigned a default priority value. You can assign other priorities when defining an instance. The following example illustrates how this is done:
`class Foo where   a : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")   b : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")  ``instance (priority := [default](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Declarations/#prioDefault "Documentation for syntax in Lean Language Reference") + 1) i1 : Foo where   a := 1   [b](Type-Classes/#Foo-next "Definition of example") := 1  ``instance i2 : Foo where   a := 2   [b](Type-Classes/#Foo-next "Definition of example") := 2  ``example : Foo.a = 1 :=   [rfl](https://lean-lang.org/doc/reference/4.26.0/Basic-Propositions/Propositional-Equality/#rfl-next "Documentation for rfl in Lean Language Reference")  ``instance (priority := [default](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Instance-Declarations/#prioDefault "Documentation for syntax in Lean Language Reference") + 2) i3 : Foo where   a := 3   [b](Type-Classes/#Foo-next "Definition of example") := 3  ``example : Foo.a = 3 :=   rfl ```
##  10.10. Coercions using Type Classes[рҹ”—](find/?domain=Verso.Genre.Manual.section&name=coercions-using-type-classes "Permalink")
The most basic type of coercion maps elements of one type to another. For example, a coercion from `[Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` to `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")` allows us to view any element `n : [Nat](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Natural-Numbers/#Nat___zero "Documentation for Nat in Lean Language Reference")` as an element of `[Int](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Integers/#Int___ofNat "Documentation for Int in Lean Language Reference")`. But some coercions depend on parameters; for example, for any type `Оұ`, we can view any element `as : [List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") Оұ` as an element of `Set Оұ`, namely, the set of elements occurring in the list. The corresponding coercion is defined on the вҖңfamilyвҖқ of types `[List](https://lean-lang.org/doc/reference/4.26.0/Basic-Types/Linked-Lists/#List___nil "Documentation for List in Lean Language Reference") Оұ`, parameterized by `Оұ`.
Lean allows us to declare three kinds of coercions:
  * from a family of types to another family of types
  * from a family of types to the class of sorts
  * from a family of types to the class of function types


The first kind of coercion allows us to view any element of a member of the source family as an element of a corresponding member of the target family. The second kind of coercion allows us to view any element of a member of the source family as a type. The third kind of coercion allows us to view any element of the source family as a function. Let us consider each of these in turn.
In Lean, coercions are implemented on top of the type class resolution framework. We define a coercion from `Оұ` to `ОІ` by declaring an instance of `[Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference") Оұ ОІ`. `instance (p : Prop) [[Decidable](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___isFalse "Documentation for Decidable in Lean Language Reference") p] : [CoeDep](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-Between-Types/#CoeDep___mk "Documentation for CoeDep in Lean Language Reference") Prop p Bool where   [coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-Between-Types/#CoeDep___mk "Documentation for CoeDep.coe in Lean Language Reference") := [decide](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Decidable___decide "Documentation for Decidable.decide in Lean Language Reference") p ```
Lean will also chain (non-dependent) coercions as necessary. Actually, the type class `[CoeT](https://lean-lang.org/doc/reference/4.26.0/Coercions/Coercing-Between-Types/#CoeT___mk "Documentation for CoeT in Lean Language Reference")` is the transitive closure of `[Coe](https://lean-lang.org/doc/reference/4.26.0/Coercions/#Coe___mk "Documentation for Coe in Lean Language Reference")`.
Let us now consider the second kind of coercion. By the _class of sorts_ , we mean the collection of universes `Type u`. A coercion of the second kind is of the form:

```
    c : (x1 : A1) вҶ’ ... вҶ’ (xn : An) вҶ’ F x1 ... xn вҶ’ Type u

```

where `F` is a family of types as above. This allows us to write `s : t` whenever `t` is of type `F aвӮҒ ... aвӮҷ`. In other words, the coercion allows us to view the elements of `F aвӮҒ ... aвӮҷ` as types. This is very useful when defining algebraic structures in which one component, the carrier of the structure, is a `Type`. For example, we can define a semigroup as follows:
`structure Semigroup where   carrier : Type u   mul : carrier вҶ’ carrier вҶ’ carrier   mul_assoc (a b c : carrier) : mul (mul a b) c = mul a (mul b c)  ``instance (S : Semigroup) : [Mul](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Mul___mk "Documentation for Mul in Lean Language Reference") S.carrier where   [mul](https://lean-lang.org/doc/reference/4.26.0/Type-Classes/Basic-Classes/#Mul___mk "Documentation for Mul.mul in Lean Language Reference") a b := S.mul a b ```
In other words, a semigroup consists of a type, `carrier`, and a multiplication, `mul`, with the property that the multiplication is associative. The `instance` command allows us to write `a * b` instead of `Semigroup.mul S a b` whenever we have `a b : S.carrier`; notice that Lean can infer the argument `S` from the types of `a` and `b`. The function `Semigroup.carrier` maps the class `Semigroup` to the sort `Type u`:

By the _class of function types_ , we mean the collection of Pi types `(z : B) вҶ’ C`. The third kind of coercion has the form:

```
    c : (xвӮҒ : AвӮҒ) вҶ’ ... вҶ’ (xвӮҷ : AвӮҷ) вҶ’ (y : F xвӮҒ ... xвӮҷ) вҶ’ (z : B) вҶ’ C

```

In other words, the coercion enables us to view elements of `F aвӮҒ ... aвӮҷ` as functions. Continuing the example above, we can define the notion of a morphism between semigroups `S1` and `S2`.
